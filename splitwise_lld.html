Splitwise System

LLD Python Design Patterns Interview Ready

Splitwise is an expense-sharing application. Friends go on a trip, someone pays for dinner, and the app figures out who owes whom — and how much. Designing it tests your ability to model real-world relationships, apply the right design patterns, and write a clean settlement algorithm.

What This System Does

Core Operations
  • Create users and groups
  • Add expenses with flexible splits
  • Track who owes whom in a group
  • Generate a settlement plan (minimize transactions)
  • Settle debts (mark payments)
Split Strategies
  • Equal — divide by number of members
  • Exact — you specify each person's share
  • Percentage — each person pays X% of total
  • Adjustment — unequal base adjustments

Why This Problem is Great for Interviews

It naturally exercises Strategy Pattern (multiple split algorithms), Observer Pattern (notify users of new expenses), Factory Pattern (creating split objects), and a greedy settlement algorithm — all in one coherent system. Interviewers can probe any layer.

💡 Interview tip: Start by listing functional requirements, then immediately sketch the core entities: User, Expense, Group, Split. Interviewers love when you drive the conversation.

Requirements

Functional Requirements

Must Have
  • Register users with name and email
  • Create groups, add/remove members
  • Add expense: who paid, how much, split type
  • View balance of a user in a group
  • Get settlement plan for a group
  • Record that a payment was made
Split Types to Support
  • Equal Split — everyone pays 1/N
  • Exact Split — amounts must sum to total
  • Percentage Split — percentages must sum to 100

Non-Functional Requirements

  • Code must be extensible — adding a new split type shouldn't touch existing code
  • Settlement algorithm should minimize number of transactions
  • System should enforce validation (percentages add to 100, amounts add to total)
  • Balances should always be consistent — if A owes B $10, B is owed $10 by A

Out of Scope (for LLD)

  • Persistent storage / database
  • REST API / HTTP layer
  • Authentication / authorization
  • Currency conversion

Core Entities

These are the nouns of the system. Identifying them clearly is the first step in any LLD interview.

User

Represents a person. Has a unique user_id, name, and email. Users can belong to multiple groups.

Group

A collection of users sharing expenses (e.g., "Thailand Trip 2024"). Holds a list of members and a list of expenses. Also tracks the balance sheet — a dict[user_id, dict[user_id, float]] representing net amounts owed.

Expense

Represents one payment. Who paid (paid_by), the total amount, the split type, and the resulting Split objects for each participant.

Split (abstract)

Holds how much one user owes for a single expense. EqualSplit, ExactSplit, PercentSplit are concrete implementations. This is where the Strategy Pattern lives.

Transaction

Output of the settlement algorithm. A directional "A pays B $X" instruction.

⚠️ Common mistake: Don't confuse Split (a share per-person per-expense) with Transaction (a settlement payment). They are very different concepts.

Relationships

  • A User can belong to many Groups
  • A Group has many Expenses
  • An Expense has one paid_by User and many Splits
  • Each Split belongs to one User and one Expense

Settlement Algorithm

This is the crown jewel of the system. The goal: given a group's net balances, produce the fewest possible transactions to settle all debts.

The Core Idea

Each person has a net balance: positive means they're owed money, negative means they owe money. The algorithm uses two heaps — one for creditors, one for debtors — and greedily matches the largest debtor with the largest creditor.

Compute net balances
→
Split into +creditors / −debtors
→
Max-heap match
→
Emit transactions

Step-by-step Example

Three people: Alice paid $90, Bob paid $30, Carol paid $0. Each should pay $40.

  • Alice net: +$50 (owed $50)
  • Bob net: −$10 (owes $10)
  • Carol net: −$40 (owes $40)

Algorithm: Carol pays Alice $40 → Alice now +$10. Bob pays Alice $10 → settled. Only 2 transactions instead of a naive 3+.

Complexity

  • Time: O(n log n) — each pop/push on a heap
  • Space: O(n) — for the two heaps
  • Transactions: At most n−1 for n people (optimal)
settlement_engine.py Strategy Pattern 
import heapq
from dataclasses import dataclass

@dataclass
class Transaction:
    payer: str      # user_id of who pays
    payee: str      # user_id of who receives
    amount: float

class SettlementEngine:
    """
    Greedy settlement algorithm using max-heaps.
    Minimizes the total number of transactions.
    """

    def minimize_transactions(
        self,
        net_balances: dict[str, float]
    ) -> list[Transaction]:
        """
        net_balances: {user_id: net_amount}
          positive = this person is owed money (creditor)
          negative = this person owes money (debtor)
        """
        # Max-heap for creditors (negate for Python's min-heap)
        creditors = []  # (-amount, user_id)
        # Max-heap for debtors (negate to get largest debt first)
        debtors = []    # (-amount_owed, user_id)

        for user_id, balance in net_balances.items():
            if balance > 0.001:      # creditor: owed money
                heapq.heappush(creditors, (-balance, user_id))
            elif balance < -0.001:   # debtor: owes money
                heapq.heappush(debtors, (balance, user_id))
            # near-zero balances are settled already

        transactions = []

        while creditors and debtors:
            # Largest creditor
            neg_credit, creditor_id = heapq.heappop(creditors)
            credit = -neg_credit

            # Largest debtor
            neg_debt, debtor_id = heapq.heappop(debtors)
            debt = -neg_debt  # debt is a positive value now

            # Match them
            settled = min(credit, debt)
            transactions.append(Transaction(
                payer=debtor_id,
                payee=creditor_id,
                amount=round(settled, 2)
            ))

            remaining_credit = credit - settled
            remaining_debt = debt - settled

            # Push back whoever still has a remainder
            if remaining_credit > 0.001:
                heapq.heappush(creditors, (-remaining_credit, creditor_id))
            if remaining_debt > 0.001:
                heapq.heappush(debtors, (-remaining_debt, debtor_id))

        return transactions
💡 Interview tip: When asked "how do you minimize transactions?", say: "I compute net balance per user, then use a greedy two-heap approach — always match the biggest creditor with the biggest debtor. This gives at most n−1 transactions and runs in O(n log n)."

Design Patterns

Four patterns work together in this system. Each is chosen for a specific reason — not applied formulaically.

Strategy Pattern Creational-ish
Multiple interchangeable split algorithms
✓ Why here⚠ Trade-off

The problem has three split types (Equal, Exact, Percent) that all do the same job — compute a user's share — but differently. Strategy lets us define a Split interface and swap implementations without touching the Expense class.

Trade-off: A new split type requires a new class, but never modifies existing code (Open/Closed Principle). The slight increase in class count is worth it.

Factory Pattern Creational
Centralized Split object creation
✓ Why here⚠ Trade-off

Callers shouldn't need to know which Split subclass to instantiate — they just say "give me a Percent split". SplitFactory.create() handles the if/else and validation. This is especially clean because each split type has different constructor params.

Trade-off: Adding a new type requires updating the factory's mapping — one place, not scattered. Worth it.

Observer Pattern Behavioral
Decouple notifications from core logic
✓ Why here⚠ Trade-off

When an expense is added, multiple systems may care: email notification, push notification, audit logger, analytics. Observer decouples this. Group fires an event; any number of subscribers listen. Core business logic never imports NotificationService.

Trade-off: Harder to trace what runs when — debugging requires knowing all observers. For an interview, mention this explicitly.

Facade Pattern Structural
Single entry point for all operations
✓ Why here⚠ Trade-off

ExpenseManager is the Facade. It hides the complexity of coordinating Group, SplitFactory, SettlementEngine, and observers. Callers use one clean API: manager.add_expense(...) — they don't know or care about the internals.

Trade-off: The Facade can become a "god class" if it accumulates too much logic. Keep it thin — delegate everything to domain objects.

splits.py Strategy Pattern + Factory Pattern 
from __future__ import annotations
from abc import ABC, abstractmethod
from models import User, SplitType


# ── Abstract Split  (Strategy interface) ────────────────────
class Split(ABC):
    """
    Strategy Pattern: each subclass is a different algorithm
    for computing one user's share of an expense.
    """

    def __init__(self, user: User) -> None:
        self._user = user

    def get_user(self) -> User:
        return self._user

    @abstractmethod
    def get_amount(self) -> float:
        """Return the amount this user owes."""
        ...

    @abstractmethod
    def validate(self) -> bool:
        """Return True if this split is internally consistent."""
        ...


# ── Concrete Strategies ─────────────────────────────────────

class EqualSplit(Split):
    """
    Each participant pays total / num_people.
    Amount is set externally after dividing.
    """

    def __init__(self, user: User, amount: float) -> None:
        super().__init__(user)
        self._amount = amount

    def get_amount(self) -> float:
        return self._amount

    def validate(self) -> bool:
        return self._amount >= 0


class ExactSplit(Split):
    """
    Caller specifies the exact dollar amount for this user.
    Validation (that all exact splits sum to total) happens in factory.
    """

    def __init__(self, user: User, amount: float) -> None:
        super().__init__(user)
        self._amount = amount

    def get_amount(self) -> float:
        return self._amount

    def validate(self) -> bool:
        return self._amount >= 0


class PercentSplit(Split):
    """
    Caller specifies a percentage. Amount derived from total.
    """

    def __init__(self, user: User, percent: float, total: float) -> None:
        super().__init__(user)
        self._percent = percent
        self._amount = round((percent / 100) * total, 2)

    def get_amount(self) -> float:
        return self._amount

    def get_percent(self) -> float:
        return self._percent

    def validate(self) -> bool:
        return 0 < self._percent <= 100


# ── Split Factory ────────────────────────────────────────────
class SplitFactory:
    """
    Factory Pattern: centralises creation of Split objects.
    Callers don't need to know which subclass to instantiate.
    """

    @staticmethod
    def create_equal_splits(
        users: list[User],
        total: float
    ) -> list[EqualSplit]:
        share = round(total / len(users), 2)
        # Adjust rounding error on last person
        splits = [EqualSplit(u, share) for u in users]
        diff = round(total - share * len(users), 2)
        if diff:
            splits[-1]._amount = round(splits[-1]._amount + diff, 2)
        return splits

    @staticmethod
    def create_exact_splits(
        user_amounts: dict[User, float],
        total: float
    ) -> list[ExactSplit]:
        if abs(sum(user_amounts.values()) - total) > 0.01:
            raise ValueError(
                f"Exact amounts {sum(user_amounts.values()):.2f} "
                f"must sum to total {total:.2f}"
            )
        return [ExactSplit(u, amt) for u, amt in user_amounts.items()]

    @staticmethod
    def create_percent_splits(
        user_percents: dict[User, float],
        total: float
    ) -> list[PercentSplit]:
        if abs(sum(user_percents.values()) - 100) > 0.01:
            raise ValueError(
                f"Percentages must sum to 100, got {sum(user_percents.values())}"
            )
        return [
            PercentSplit(u, pct, total)
            for u, pct in user_percents.items()
        ]
group.py Group + Observer Pattern 
from __future__ import annotations
import uuid
from abc import ABC, abstractmethod
from models import User, Transaction
from splits import Split


# ── Observer Interface ────────────────────────────────────────
class ExpenseObserver(ABC):
    """
    Observer Pattern: subscribe to expense events
    without coupling to the Group/Expense internals.
    """

    @abstractmethod
    def on_expense_added(self, group_name: str, expense: "Expense") -> None: ...

    @abstractmethod
    def on_payment_made(self, txn: Transaction, users: dict[str, User]) -> None: ...


# ── Concrete Observer ─────────────────────────────────────────
class NotificationService(ExpenseObserver):
    """Prints notifications — in prod this would send push/email."""

    def on_expense_added(self, group_name: str, expense: "Expense") -> None:
        print(
            f"[NOTIFY] New expense in '{group_name}': "
            f"${expense.get_amount():.2f} by "
            f"{expense.get_paid_by().get_name()} — {expense.get_description()}"
        )

    def on_payment_made(self, txn: Transaction, users: dict[str, User]) -> None:
        print(f"[NOTIFY] Payment recorded: {txn.describe(users)}")


# ── Expense ──────────────────────────────────────────────────
class Expense:
    def __init__(
        self,
        paid_by: User,
        amount: float,
        splits: list[Split],
        description: str = ""
    ) -> None:
        self._expense_id = str(uuid.uuid4())[:8]
        self._paid_by = paid_by
        self._amount = amount
        self._splits = splits
        self._description = description

    def get_expense_id(self) -> str:     return self._expense_id
    def get_paid_by(self) -> User:       return self._paid_by
    def get_amount(self) -> float:       return self._amount
    def get_splits(self) -> list[Split]: return self._splits
    def get_description(self) -> str:    return self._description


# ── Group ────────────────────────────────────────────────────
class Group:
    """
    Core aggregate. Maintains member list, expense list,
    and the net balance sheet.

    Observer Pattern: fires events to registered observers.
    """

    def __init__(self, name: str) -> None:
        self._group_id = str(uuid.uuid4())[:8]
        self._name = name
        self._members: list[User] = []
        self._expenses: list[Expense] = []
        # balances[A][B] = amount A is owed by B  (positive = A is owed)
        self._balances: dict[str, dict[str, float]] = {}
        self._observers: list[ExpenseObserver] = []

    # ── Observer management ──────────────────────────────────
    def add_observer(self, obs: ExpenseObserver) -> None:
        self._observers.append(obs)

    def _notify_expense_added(self, expense: Expense) -> None:
        for obs in self._observers:
            obs.on_expense_added(self._name, expense)

    def _notify_payment(self, txn: Transaction, users: dict[str, User]) -> None:
        for obs in self._observers:
            obs.on_payment_made(txn, users)

    # ── Members ──────────────────────────────────────────────
    def add_member(self, user: User) -> None:
        if user not in self._members:
            self._members.append(user)
            uid = user.get_user_id()
            self._balances.setdefault(uid, {})

    def get_members(self) -> list[User]:
        return list(self._members)

    # ── Expenses ─────────────────────────────────────────────
    def add_expense(self, expense: Expense) -> None:
        self._expenses.append(expense)
        self._update_balances(expense)
        self._notify_expense_added(expense)

    def _update_balances(self, expense: Expense) -> None:
        """
        After adding expense, update the net balance sheet.
        paid_by gets +amount from each split participant.
        Each split participant gets -amount from paid_by.
        """
        payer_id = expense.get_paid_by().get_user_id()

        for split in expense.get_splits():
            participant_id = split.get_user().get_user_id()
            if participant_id == payer_id:
                continue  # payer doesn't owe themselves

            amt = split.get_amount()

            # participant owes payer `amt`
            self._balances.setdefault(participant_id, {})
            self._balances.setdefault(payer_id, {})

            # payer is owed `amt` by participant
            self._balances[payer_id][participant_id] = (
                self._balances[payer_id].get(participant_id, 0) + amt
            )
            # participant owes payer `amt`
            self._balances[participant_id][payer_id] = (
                self._balances[participant_id].get(payer_id, 0) - amt
            )

    def get_net_balances(self) -> dict[str, float]:
        """Collapse the balance matrix into net per-user balance."""
        net: dict[str, float] = {}
        for uid, owed_dict in self._balances.items():
            net[uid] = round(sum(owed_dict.values()), 2)
        return net

    def get_balance_for(self, user_id: str) -> dict[str, float]:
        """Return what user_id owes/is owed from each member."""
        return dict(self._balances.get(user_id, {}))

    def record_payment(
        self,
        txn: Transaction,
        users: dict[str, User]
    ) -> None:
        """When payer pays payee, reduce those balances."""
        payer = txn.payer_id
        payee = txn.payee_id
        amt = txn.amount

        self._balances.setdefault(payer, {})
        self._balances.setdefault(payee, {})

        self._balances[payer][payee] = (
            self._balances[payer].get(payee, 0) + amt
        )
        self._balances[payee][payer] = (
            self._balances[payee].get(payer, 0) - amt
        )
        self._notify_payment(txn, users)

    # ── Getters ──────────────────────────────────────────────
    def get_group_id(self) -> str: return self._group_id
    def get_name(self) -> str:     return self._name
expense_manager.py Facade Pattern + Settlement Engine 
from __future__ import annotations
import heapq
from models import User, Transaction, SplitType
from splits import SplitFactory
from group import Group, Expense, ExpenseObserver, NotificationService


# ── Settlement Engine (Strategy for algorithm) ────────────────
class SettlementEngine:
    """
    Greedy two-heap algorithm.
    O(n log n) time, minimises transaction count.
    """

    def minimize_transactions(
        self,
        net_balances: dict[str, float]
    ) -> list[Transaction]:
        creditors: list[tuple[float, str]] = []
        debtors: list[tuple[float, str]] = []

        for uid, balance in net_balances.items():
            if balance > 0.001:
                heapq.heappush(creditors, (-balance, uid))
            elif balance < -0.001:
                heapq.heappush(debtors, (balance, uid))

        transactions: list[Transaction] = []

        while creditors and debtors:
            neg_credit, creditor_id = heapq.heappop(creditors)
            credit = -neg_credit

            neg_debt, debtor_id = heapq.heappop(debtors)
            debt = -neg_debt

            settled = min(credit, debt)
            transactions.append(Transaction(
                payer_id=debtor_id,
                payee_id=creditor_id,
                amount=round(settled, 2)
            ))

            if credit - settled > 0.001:
                heapq.heappush(creditors, (-(credit - settled), creditor_id))
            if debt - settled > 0.001:
                heapq.heappush(debtors, (-(debt - settled), debtor_id))

        return transactions


# ── Expense Manager (Facade) ─────────────────────────────────
class ExpenseManager:
    """
    Facade Pattern: single entry point for all Splitwise operations.
    Coordinates Group, SplitFactory, SettlementEngine, and Observers.
    """

    def __init__(self) -> None:
        self._users: dict[str, User] = {}
        self._groups: dict[str, Group] = {}
        self._engine = SettlementEngine()
        self._default_observer = NotificationService()

    # ── User Management ──────────────────────────────────────
    def register_user(self, name: str, email: str) -> User:
        user = User(name, email)
        self._users[user.get_user_id()] = user
        return user

    # ── Group Management ─────────────────────────────────────
    def create_group(self, name: str) -> Group:
        group = Group(name)
        group.add_observer(self._default_observer)
        self._groups[group.get_group_id()] = group
        return group

    def add_member_to_group(self, group: Group, user: User) -> None:
        group.add_member(user)

    # ── Expense Management ───────────────────────────────────
    def add_expense(
        self,
        group: Group,
        paid_by: User,
        amount: float,
        split_type: SplitType,
        description: str = "",
        participants: list[User] | None = None,
        exact_amounts: dict[User, float] | None = None,
        percents: dict[User, float] | None = None,
    ) -> Expense:
        """
        Facade hides the complexity of creating splits.
        """
        members = participants or group.get_members()

        if split_type == SplitType.EQUAL:
            splits = SplitFactory.create_equal_splits(members, amount)

        elif split_type == SplitType.EXACT:
            if not exact_amounts:
                raise ValueError("Exact split requires exact_amounts dict")
            splits = SplitFactory.create_exact_splits(exact_amounts, amount)

        elif split_type == SplitType.PERCENT:
            if not percents:
                raise ValueError("Percent split requires percents dict")
            splits = SplitFactory.create_percent_splits(percents, amount)

        else:
            raise ValueError(f"Unknown split type: {split_type}")

        expense = Expense(paid_by, amount, splits, description)
        group.add_expense(expense)
        return expense

    # ── Balances ─────────────────────────────────────────────
    def get_balances(self, group: Group) -> dict[str, float]:
        """Net balance per user: +means owed, -means owes."""
        return group.get_net_balances()

    def print_balances(self, group: Group) -> None:
        net = self.get_balances(group)
        print(f"\n── Balances in '{group.get_name()}' ──")
        for uid, balance in net.items():
            name = self._users[uid].get_name()
            if balance > 0:
                print(f"  {name} is owed  ${balance:.2f}")
            elif balance < 0:
                print(f"  {name} owes     ${abs(balance):.2f}")
            else:
                print(f"  {name} is settled")

    # ── Settlement ───────────────────────────────────────────
    def settle_up(self, group: Group) -> list[Transaction]:
        """Generate minimum-transaction settlement plan."""
        net = self.get_balances(group)
        return self._engine.minimize_transactions(net)

    def print_settlement(self, group: Group) -> None:
        txns = self.settle_up(group)
        print(f"\n── Settlement plan for '{group.get_name()}' ──")
        if not txns:
            print("  Already settled!")
            return
        for txn in txns:
            print(f"  {txn.describe(self._users)}")

    # ── Record Payment ───────────────────────────────────────
    def record_payment(self, group: Group, txn: Transaction) -> None:
        group.record_payment(txn, self._users)
demo.py Running Example 
from models import SplitType
from expense_manager import ExpenseManager

def main():
    mgr = ExpenseManager()

    # ── Step 1: Create users ────────────────────────────────
    alice = mgr.register_user("Alice", "alice@example.com")
    bob   = mgr.register_user("Bob",   "bob@example.com")
    carol = mgr.register_user("Carol", "carol@example.com")

    # ── Step 2: Create a group ──────────────────────────────
    trip = mgr.create_group("Thailand Trip 2024")
    for u in [alice, bob, carol]:
        mgr.add_member_to_group(trip, u)

    # ── Step 3: Add expenses ────────────────────────────────
    # Alice pays $90 for dinner, split equally
    mgr.add_expense(
        group=trip,
        paid_by=alice,
        amount=90.0,
        split_type=SplitType.EQUAL,
        description="Dinner at Nahm"
    )

    # Bob pays $60 for hotel, with exact amounts
    mgr.add_expense(
        group=trip,
        paid_by=bob,
        amount=60.0,
        split_type=SplitType.EXACT,
        description="Hotel deposit",
        exact_amounts={alice: 20.0, bob: 20.0, carol: 20.0}
    )

    # Carol pays $100 for tours, split by percentage
    mgr.add_expense(
        group=trip,
        paid_by=carol,
        amount=100.0,
        split_type=SplitType.PERCENT,
        description="Island tours",
        percents={alice: 50, bob: 30, carol: 20}
    )

    # ── Step 4: View balances ───────────────────────────────
    mgr.print_balances(trip)

    # ── Step 5: Get settlement plan ─────────────────────────
    mgr.print_settlement(trip)


if __name__ == "__main__":
    main()

SOLID Principles

S
Single Responsibility
Group manages members + balances. SettlementEngine only computes settlements. Never mixed.
O
Open / Closed
New split type? Add a new class. Never modify Expense or SplitFactory dispatch.
L
Liskov Substitution
EqualSplit, ExactSplit, PercentSplit are all valid Split anywhere a Split is expected.
I
Interface Segregation
ExpenseObserver is a thin interface. Implementors only define on_expense_added and on_payment_made.
D
Dependency Inversion
Group depends on ExpenseObserver (abstract), not on NotificationService (concrete). Easily swappable.

SOLID in Practice — Interview Q

Q: How does your design handle a new "Adjustment Split" type?

  • Create class AdjustmentSplit(Split) — implement get_amount()
  • Add create_adjustment_splits() to SplitFactory
  • Add SplitType.ADJUSTMENT to the enum
  • Add one elif branch in ExpenseManager.add_expense()
  • Zero changes to Group, Expense, SettlementEngine, or any existing split class
This is the Open/Closed Principle in action. The answer demonstrates you've thought about extensibility, not just the happy path.

Balance Data Structure

The balance sheet is a nested dict: balances[A][B] = how much A is owed by B. Positive = owed, negative = owes. The get_net_balances() method collapses this to a flat dict[user_id, float].

balance_example.py Mental Model 
# After Alice pays $90 split equally (3 people, $30 each):
balances = {
    "alice_id": {"bob_id": +30.0, "carol_id": +30.0},  # Alice is owed
    "bob_id":   {"alice_id": -30.0},                    # Bob owes Alice
    "carol_id": {"alice_id": -30.0},                    # Carol owes Alice
}

# net_balances (collapsed):
net = {
    "alice_id": +60.0,   # owed $60 total
    "bob_id":   -30.0,   # owes $30
    "carol_id": -30.0,   # owes $30
}

Trade-offs & Extensions

Design Decisions & Alternatives

Balance storage: nested dict
  • Chosen: dict[uid][uid] → float
  • Why: O(1) lookup, easy to update incrementally
  • Alternative: Recompute from expense list each time — simpler but O(expenses × members)
  • Trade-off: Consistency burden — must update both directions on each expense
Settlement algorithm
  • Chosen: Greedy two-heap
  • Why: O(n log n), provably minimal transactions
  • Alternative: Simple pairwise — O(n²), easy to understand
  • Alternative: Graph-based flow network — can be optimal but harder to explain in 30 min
Observer vs direct call
  • Chosen: Observer (interface + register)
  • Why: Decouples Group from notification details
  • Alternative: Inject NotificationService directly into Group
  • Trade-off: Observer is slightly harder to trace at runtime
Split creation: Factory vs Enum dispatch
  • Chosen: Separate create_* methods per type
  • Why: Each type has different params — a single create(type, **kwargs) would hide required args
  • Alternative: Registry dict {"EQUAL": EqualSplit, ...} + dynamic dispatch

Possible Extensions

  • Currency support: Add a Currency field to Expense; inject a CurrencyConverter into SettlementEngine
  • Recurring expenses: Add a RecurringExpense subclass with a Scheduler that fires ExpenseManager.add_expense() on a schedule
  • Expense categories: Add Category enum to Expense; queries become easy
  • Audit log: Add an AuditObserver that persists all events — zero changes to core classes
  • Multiple settlement strategies: Wrap SettlementEngine in an interface, inject via constructor — different algorithms at runtime

What design patterns did you use and why?▶
Four patterns, each chosen for a specific reason:

Strategy — for split types. Each split is a different algorithm for computing a share. Swapping algorithms without touching Expense.
Factory — for creating split objects. Each type has different constructor params; factory hides that complexity.
Observer — for notifications. Group fires events; notification/audit systems subscribe. Core logic stays clean.
Facade — ExpenseManager is the single entry point. Hides coordination between Group, SplitFactory, SettlementEngine.
How does the settlement algorithm work?▶
I compute each person's net balance (positive = owed, negative = owes). Then I use two max-heaps — one for creditors, one for debtors. Each iteration: pop the largest creditor and largest debtor, have the debtor pay the creditor the minimum of (what creditor is owed, what debtor owes). Push back any remainder. This produces at most n−1 transactions and runs in O(n log n).
How would you add a new split type, say "Adjustment Split"?▶
1. Create AdjustmentSplit(Split) — implement get_amount().
2. Add SplitType.ADJUSTMENT to the enum.
3. Add SplitFactory.create_adjustment_splits().
4. Add one elif branch in ExpenseManager.add_expense().

No changes to Group, Expense, SettlementEngine, or existing split classes. This is the Open/Closed Principle.
How do you keep balances consistent?▶
The balance sheet is a nested dict updated in Group._update_balances() every time an expense is added. For each split, I update both directions atomically in the same method: balances[payer][participant] += amount AND balances[participant][payer] -= amount. This invariant means if A is owed $10 by B, then B owes $10 to A — always. Record payment does the same in reverse.
What's the difference between Split and Transaction?▶
A Split represents one person's share of a single expense. It's created when an expense is added. A Transaction is the output of the settlement algorithm — it says "Person A should pay Person B $X" to settle accumulated debts across all expenses. One expense creates many Splits; many expenses together produce a few Transactions.
Why is ExpenseManager a Facade and not just a utility class?▶
A utility class is a grab-bag of static helpers. A Facade is a deliberate design: it provides a simplified interface to a complex subsystem and hides the internal coordination. ExpenseManager does exactly that — callers don't know about SplitFactory, SettlementEngine, or how observers are wired. The Facade is thin: it delegates, it doesn't implement domain logic itself.
How would you support currency conversion?▶
Add a Currency field to Expense. Define a CurrencyConverter interface with a convert(amount, from_currency, to_currency) method. Inject it into SettlementEngine. When computing net balances, convert all amounts to a base currency before summing. In tests, inject a mock converter. This follows Dependency Inversion — SettlementEngine depends on the abstract converter, not a real FX feed.
How does the Observer pattern help here vs just calling a method?▶
If Group directly called NotificationService.notify(), it would depend on that concrete class. Tomorrow you'd also need an AuditLogger, then an AnalyticsService — Group's constructor would grow. With Observer, Group only knows about the ExpenseObserver interface. New subscribers are added by registering them, zero changes to Group. The core domain object stays stable; the surrounding infrastructure is hot-swappable.
