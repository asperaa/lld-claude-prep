Parking Lot System

This guide takes you from zero to confident on the Parking Lot LLD problem — one of the most frequently asked system design questions in interviews at companies like Amazon, Google, Uber, and Flipkart. We'll build the full system step by step, understand why each design pattern is used, and prepare you for the tricky follow-up questions interviewers love to ask.

💡 How to use this guide
Read Foundation → understand the entities → then read each pattern section including running the code mentally. At the end, close the guide and try to redraw the UML from scratch. That's how you know you've got it.

1. Requirements

Before writing a single line of code in an interview, always clarify requirements. Interviewers reward candidates who ask smart questions upfront.

Functional (what it must DO)

  • Park vehicles: cars, motorcycles, trucks
  • Find & assign an available spot on entry
  • Issue a ticket when a vehicle enters
  • Calculate fee on exit based on time parked
  • Process payment and free the spot
  • Show availability: spots per floor/type
  • Support multiple floors in the lot

Non-Functional (how well it does it)

  • Thread-safe: multiple entries at once
  • Extensible: add new vehicle/spot types easily
  • Scalable: 100s of spots without changing logic
  • Readable: clear separation of concerns
  • Testable: mockable dependencies
⚡ Clarifying Questions to ask the interviewer
"Does a motorcycle need a small spot or can it park in a large spot?" · "Is pricing hourly, per-minute, or flat rate?" · "Do we need payment gateway integration, or just fee calculation?" · "Should we support reservations / pre-booking?" · "Single building or multiple locations?"

2. Core Entities & Data Classes

These are the nouns of our system — the real-world things we model in code. Identify these first before thinking about patterns.

Python · enums.py — Core Types
from enum import Enum, auto

class VehicleType(Enum):
    """What kind of vehicle is parking."""
    MOTORCYCLE = auto()
    CAR        = auto()
    TRUCK      = auto()

class SpotSize(Enum):
    """Physical size of a parking spot."""
    SMALL  = auto()   # fits motorcycles only
    MEDIUM = auto()   # fits cars (and motorcycles)
    LARGE  = auto()   # fits trucks (and cars, motorcycles)

class TicketStatus(Enum):
    """Where is the ticket in its lifecycle?"""
    ACTIVE = auto()   # vehicle is currently parked
    PAID   = auto()   # payment done, vehicle can leave
    CLOSED = auto()   # vehicle has exited

# Why Enum over plain strings?
# - Typos like "motrrcycle" become an error, not a silent bug
# - IDE autocomplete works
# - Easy to iterate over all types: for t in VehicleType: ...
# - Why not IntEnum? We don't need numeric comparison, Enum is safer
Python · models.py — Vehicle & Spot
from dataclasses import dataclass, field
from datetime import datetime
from typing import Optional
from enums import VehicleType, SpotSize, TicketStatus


@dataclass
class Vehicle:
    """
    Represents a vehicle entering the lot.
    We store only what we need — license plate identifies it uniquely.
    """
    license_plate: str
    vehicle_type: VehicleType

    def can_fit_in(self, spot_size: SpotSize) -> bool:
        """Can this vehicle physically fit in that spot?"""
        # A smaller vehicle CAN use a bigger spot (if rules allow)
        # This table encodes the compatibility:
        compatibility = {
            VehicleType.MOTORCYCLE: [SpotSize.SMALL, SpotSize.MEDIUM, SpotSize.LARGE],
            VehicleType.CAR:        [SpotSize.MEDIUM, SpotSize.LARGE],
            VehicleType.TRUCK:      [SpotSize.LARGE],
        }
        return spot_size in compatibility[self.vehicle_type]


@dataclass
class ParkingSpot:
    """
    A single parking space on a floor.
    Knows its own size and whether it's occupied.
    """
    spot_id: str          # e.g. "F1-S001" (floor 1, spot 001)
    floor_number: int
    spot_size: SpotSize
    is_occupied: bool = False
    current_vehicle: Optional[Vehicle] = None

    def is_available_for(self, vehicle: Vehicle) -> bool:
        return not self.is_occupied and vehicle.can_fit_in(self.spot_size)

    def park(self, vehicle: Vehicle) -> None:
        self.is_occupied = True
        self.current_vehicle = vehicle

    def vacate(self) -> None:
        self.is_occupied = False
        self.current_vehicle = None


@dataclass
class ParkingFloor:
    """
    One floor of the parking lot.
    Contains multiple spots of various sizes.
    """
    floor_number: int
    spots: list = field(default_factory=list)

    def add_spot(self, spot: ParkingSpot) -> None:
        self.spots.append(spot)

    def find_available_spot(self, vehicle: Vehicle) -> Optional[ParkingSpot]:
        """Return the first available spot for this vehicle, or None."""
        for spot in self.spots:
            if spot.is_available_for(vehicle):
                return spot
        return None

    def get_availability_count(self) -> dict:
        """How many free spots of each size on this floor?"""
        counts = {size: 0 for size in SpotSize}
        for spot in self.spots:
            if not spot.is_occupied:
                counts[spot.spot_size] += 1
        return counts


@dataclass
class Ticket:
    """
    Issued when a vehicle enters. Holds all info needed to compute the fee.
    Think of it like a real paper ticket from a parking machine.
    """
    ticket_id: str
    vehicle: Vehicle
    spot: ParkingSpot
    entry_time: datetime
    exit_time: Optional[datetime] = None
    amount_paid: float = 0.0
    status: TicketStatus = TicketStatus.ACTIVE

    def duration_hours(self) -> float:
        """How long has this vehicle been parked (in hours)?"""
        end = self.exit_time or datetime.now()
        delta = end - self.entry_time
        return delta.total_seconds() / 3600.0
💡 Interview tip — Spot compatibility
The can_fit_in() method on Vehicle is a clean way to encode spot compatibility rules. Interviewers like seeing this centralized — if rules change (e.g. "motorcycles can no longer use medium spots"), you only change one place.

4. Pattern: Singleton — ParkingLot Manager

🏛️
Singleton
Guarantees exactly one ParkingLot instance exists
🧠 Why Singleton here?
There is physically ONE parking lot. If two parts of the system accidentally created two separate ParkingLot objects, they'd have inconsistent views of which spots are available. Singleton enforces that everyone uses the same object. It's the "single source of truth" for the entire system.
Python · parking_lot.py — Singleton ParkingLot
import threading
import uuid
from datetime import datetime
from typing import Optional
from models import Vehicle, ParkingFloor, ParkingSpot, Ticket
from enums import TicketStatus


class ParkingLot:
    """
    The central manager of the entire parking lot.
    Implemented as a Singleton: only ONE instance can exist.

    Pattern: Singleton
    Why: There is one physical parking lot. All code must share the
         same state (which spots are free, what tickets are active).
    """
    _instance = None
    _lock = threading.Lock()     # Lock for thread-safe Singleton creation

    def __new__(cls, name: str = "Main Parking Lot"):
        # Double-checked locking: check once before acquiring lock,
        # check again after. Avoids slow lock acquisition on every call.
        if cls._instance is None:
            with cls._lock:
                if cls._instance is None:         # second check
                    cls._instance = super().__new__(cls)
                    cls._instance._initialized = False
        return cls._instance

    def __init__(self, name: str = "Main Parking Lot"):
        # __init__ runs every time ParkingLot() is called,
        # so guard against re-initialization.
        if self._initialized:
            return
        self.name = name
        self.floors: list[ParkingFloor] = []
        self.active_tickets: dict[str, Ticket] = {}  # ticket_id → Ticket
        self._operation_lock = threading.Lock()       # for parking operations
        self._fee_strategy = None    # set via set_fee_strategy()
        self._observers: list = []   # availability observers
        self._initialized = True

    def set_fee_strategy(self, strategy) -> None:
        """Swap out the pricing strategy at runtime (Strategy pattern)."""
        self._fee_strategy = strategy

    def add_observer(self, observer) -> None:
        """Register an observer to be notified of spot changes."""
        self._observers.append(observer)

    def _notify_observers(self, floor: ParkingFloor) -> None:
        counts = floor.get_availability_count()
        for obs in self._observers:
            obs.on_spot_change(floor.floor_number, counts)

    def add_floor(self, floor: ParkingFloor) -> None:
        self.floors.append(floor)

    def park_vehicle(self, vehicle: Vehicle) -> Optional[Ticket]:
        """
        Main entry point: find a spot, issue a ticket.
        Thread-safe — only one vehicle can be parked at a time.
        Returns the Ticket, or None if lot is full.
        """
        with self._operation_lock:
            spot = self._find_spot(vehicle)
            if spot is None:
                print(f"❌ No available spot for {vehicle.license_plate}")
                return None

            spot.park(vehicle)
            ticket = Ticket(
                ticket_id=str(uuid.uuid4())[:8].upper(),
                vehicle=vehicle,
                spot=spot,
                entry_time=datetime.now(),
            )
            self.active_tickets[ticket.ticket_id] = ticket
            self._notify_observers(
                next(f for f in self.floors if f.floor_number == spot.floor_number)
            )
            print(f"✅ Parked {vehicle.license_plate} at spot {spot.spot_id} | Ticket: {ticket.ticket_id}")
            return ticket

    def process_exit(self, ticket_id: str) -> float:
        """
        On exit: calculate fee, free the spot, close the ticket.
        Returns the fee charged.
        """
        with self._operation_lock:
            ticket = self.active_tickets.get(ticket_id)
            if ticket is None:
                raise ValueError(f"Ticket {ticket_id} not found or already closed")

            ticket.exit_time = datetime.now()
            fee = self._fee_strategy.calculate_fee(ticket)
            ticket.amount_paid = fee
            ticket.status = TicketStatus.PAID

            ticket.spot.vacate()
            self._notify_observers(
                next(f for f in self.floors if f.floor_number == ticket.spot.floor_number)
            )
            del self.active_tickets[ticket_id]
            print(f"💰 Vehicle {ticket.vehicle.license_plate} exited. Fee: ₹{fee:.2f}")
            return fee

    def _find_spot(self, vehicle: Vehicle) -> Optional[ParkingSpot]:
        """Search floors in order, return first available spot."""
        for floor in self.floors:
            spot = floor.find_available_spot(vehicle)
            if spot:
                return spot
        return None

    def get_availability(self) -> dict:
        """Show free spots per floor per size."""
        result = {}
        for floor in self.floors:
            result[floor.floor_number] = floor.get_availability_count()
        return result

5. Pattern: Strategy — Fee Calculation

💰
Strategy
Swap pricing algorithms without changing ParkingLot
🧠 Why Strategy here?
Fee calculation rules can vary widely: hourly, flat-rate, peak-hour pricing, discounts for frequent visitors. Without Strategy, you'd have a giant if/elif in ParkingLot. With Strategy, you define a common interface (calculate_fee) and plug in any implementation. Adding a new pricing model = adding one new class, zero changes to ParkingLot.
Python · fee_strategy.py — Pluggable Pricing
from abc import ABC, abstractmethod
from models import Ticket
from enums import VehicleType, SpotSize


class FeeStrategy(ABC):
    """
    Abstract base class for fee calculation.

    Pattern: Strategy
    This is the "interface" that all pricing algorithms must implement.
    ParkingLot only knows about FeeStrategy — never the concrete class.
    """
    @abstractmethod
    def calculate_fee(self, ticket: Ticket) -> float:
        """Given a ticket (with entry/exit time), return the fee in rupees."""
        pass


class HourlyFeeStrategy(FeeStrategy):
    """
    Simple hourly pricing. Each vehicle type has a different rate.
    Minimum charge: 1 hour.

    Example: CAR parked 2.5 hours → 2.5 × ₹50 = ₹125
    """
    RATES = {
        VehicleType.MOTORCYCLE: 20,
        VehicleType.CAR:        50,
        VehicleType.TRUCK:      100,
    }

    def calculate_fee(self, ticket: Ticket) -> float:
        hours = max(1.0, ticket.duration_hours())  # minimum 1 hour charge
        rate = self.RATES[ticket.vehicle.vehicle_type]
        return round(hours * rate, 2)


class FlatRateFeeStrategy(FeeStrategy):
    """
    Fixed price per visit, regardless of time.
    Great for airports or events where parking is charged at entry.

    Example: CAR → always ₹100, no matter how long.
    """
    RATES = {
        VehicleType.MOTORCYCLE: 30,
        VehicleType.CAR:        100,
        VehicleType.TRUCK:      200,
    }

    def calculate_fee(self, ticket: Ticket) -> float:
        return self.RATES[ticket.vehicle.vehicle_type]


class PeakHourFeeStrategy(FeeStrategy):
    """
    Hourly pricing with a 1.5× surcharge during peak hours (8–10am, 5–8pm).
    Demonstrates how Strategy can wrap another strategy (Decorator-like).
    """
    BASE_RATES = {
        VehicleType.MOTORCYCLE: 20,
        VehicleType.CAR:        50,
        VehicleType.TRUCK:      100,
    }
    PEAK_HOURS = [(8, 10), (17, 20)]   # (start_hour, end_hour)
    PEAK_MULTIPLIER = 1.5

    def _is_peak(self, hour: int) -> bool:
        return any(start <= hour < end for start, end in self.PEAK_HOURS)

    def calculate_fee(self, ticket: Ticket) -> float:
        hours = max(1.0, ticket.duration_hours())
        rate = self.BASE_RATES[ticket.vehicle.vehicle_type]
        multiplier = self.PEAK_MULTIPLIER if self._is_peak(ticket.entry_time.hour) else 1.0
        return round(hours * rate * multiplier, 2)


class TieredFeeStrategy(FeeStrategy):
    """
    Different rates for different time brackets:
      First 1 hour: flat fee
      Hours 1–3: hourly rate A
      After 3 hours: hourly rate B (usually cheaper per hour for long stays)
    """
    TIERS = {
        VehicleType.CAR: [
            (1,  30),   # first hour: ₹30 flat
            (3,  25),   # hours 1-3: ₹25/hr
            (float('inf'), 15),  # after 3hrs: ₹15/hr
        ],
        VehicleType.MOTORCYCLE: [
            (1, 10), (3, 8), (float('inf'), 5)
        ],
        VehicleType.TRUCK: [
            (1, 50), (3, 40), (float('inf'), 30)
        ],
    }

    def calculate_fee(self, ticket: Ticket) -> float:
        hours = ticket.duration_hours()
        tiers = self.TIERS[ticket.vehicle.vehicle_type]
        fee = 0.0
        remaining = hours
        prev_limit = 0.0

        for limit, rate in tiers:
            if remaining <= 0:
                break
            bracket = min(remaining, limit - prev_limit)
            fee += bracket * rate
            remaining -= bracket
            prev_limit = limit

        return round(fee, 2)

⚖️ Trade-off
Strategy adds a class for every pricing rule. For 2 strategies, a simple if/else might be fine. The pattern earns its complexity when you have 3+ interchangeable algorithms and especially when you want to allow runtime switching (e.g. changing pricing on weekends).

6. Pattern: Factory — Spot & Vehicle Creation

🏭
Factory Method
Centralizes and simplifies object creation
🧠 Why Factory here?
Creating a ParkingSpot involves: generating a unique ID, validating the floor number, setting defaults. Scattering this logic everywhere leads to bugs. Factory centralizes it: one place where spots are born, one place to fix if spot creation logic changes.
Python · factories.py — Vehicle & Spot Factories
from models import Vehicle, ParkingSpot, ParkingFloor
from enums import VehicleType, SpotSize


class VehicleFactory:
    """
    Creates Vehicle objects from raw data (e.g. from API input or a form).

    Pattern: Factory
    Why: Validates input and maps string types to enums in one place.
         Caller doesn't need to know about VehicleType enum directly.
    """
    _type_map = {
        "motorcycle": VehicleType.MOTORCYCLE,
        "car":        VehicleType.CAR,
        "truck":      VehicleType.TRUCK,
    }

    @classmethod
    def create(cls, license_plate: str, vehicle_type_str: str) -> Vehicle:
        """
        Create a Vehicle from a license plate string and type string.
        Raises ValueError for unknown types.
        """
        vtype_str = vehicle_type_str.lower().strip()
        if vtype_str not in cls._type_map:
            raise ValueError(
                f"Unknown vehicle type: '{vehicle_type_str}'. "
                f"Valid types: {list(cls._type_map.keys())}"
            )
        return Vehicle(
            license_plate=license_plate.upper().strip(),
            vehicle_type=cls._type_map[vtype_str],
        )


class SpotFactory:
    """
    Creates ParkingSpot objects with auto-generated IDs.

    The ID format: F{floor}-{size_initial}{counter:03d}
    e.g. F1-S001 = Floor 1, Small spot #1
         F2-M015 = Floor 2, Medium spot #15
    """
    _counters: dict[str, int] = {}   # tracks count per (floor, size)

    @classmethod
    def create(cls, floor_number: int, spot_size: SpotSize) -> ParkingSpot:
        key = f"{floor_number}-{spot_size.name}"
        cls._counters[key] = cls._counters.get(key, 0) + 1
        count = cls._counters[key]
        size_initial = spot_size.name[0]  # S, M, or L
        spot_id = f"F{floor_number}-{size_initial}{count:03d}"
        return ParkingSpot(
            spot_id=spot_id,
            floor_number=floor_number,
            spot_size=spot_size,
        )

    @classmethod
    def reset(cls):
        """Reset counters (useful in tests)."""
        cls._counters.clear()


class ParkingLotBuilder:
    """
    A Builder helper to assemble the full parking lot from a configuration.
    Not strictly a GoF pattern, but shows how to use factories together.
    """
    def __init__(self, lot):
        self.lot = lot

    def add_floor(self, floor_number: int, config: dict) -> 'ParkingLotBuilder':
        """
        config = {SpotSize.SMALL: 10, SpotSize.MEDIUM: 20, SpotSize.LARGE: 5}
        """
        floor = ParkingFloor(floor_number=floor_number)
        for size, count in config.items():
            for _ in range(count):
                spot = SpotFactory.create(floor_number, size)
                floor.add_spot(spot)
        self.lot.add_floor(floor)
        return self     # fluent interface — builder.add_floor(...).add_floor(...)

    def build(self):
        return self.lot

7. Pattern: Template Method — Entry/Exit Flow

📋
Template Method
Defines the skeleton of entry/exit; subclasses customize steps
🧠 Why Template Method here?
Entry and exit follow a fixed sequence of steps (validate → find spot → issue ticket → notify). But certain steps differ — a VIP lane might skip payment, an automated lot might use a different notification. Template Method defines the skeleton once and lets subclasses override individual steps.
Python · entry_handler.py — Template Method
from abc import ABC, abstractmethod
from models import Vehicle, Ticket
from parking_lot import ParkingLot


class ParkingEntryHandler(ABC):
    """
    Template Method: defines the fixed algorithm for a vehicle entering.
    Subclasses can override individual steps without changing the overall flow.

        handle_entry()     ← the "template method" — DO NOT override this
            ↓ validate_vehicle()     — CAN override
            ↓ find_and_assign_spot() — CAN override
            ↓ issue_ticket()         — CAN override
            ↓ post_entry_hook()      — CAN override (default: do nothing)
    """
    def __init__(self, lot: ParkingLot):
        self.lot = lot

    def handle_entry(self, vehicle: Vehicle) -> Ticket | None:
        """
        THE template method. Orchestrates the full entry process.
        Final: subclasses must NOT override this (the skeleton is fixed).
        """
        if not self.validate_vehicle(vehicle):
            print(f"⛔ Vehicle {vehicle.license_plate} failed validation.")
            return None

        ticket = self.find_and_assign_spot(vehicle)
        if ticket is None:
            return None

        ticket = self.issue_ticket(ticket)
        self.post_entry_hook(ticket)
        return ticket

    def validate_vehicle(self, vehicle: Vehicle) -> bool:
        """Default validation: always allow. Override to add checks."""
        return True

    def find_and_assign_spot(self, vehicle: Vehicle) -> Ticket | None:
        """Default: use the lot's standard assignment."""
        return self.lot.park_vehicle(vehicle)

    def issue_ticket(self, ticket: Ticket) -> Ticket:
        """Default: ticket is already created. Override to add printing, etc."""
        return ticket

    @abstractmethod
    def post_entry_hook(self, ticket: Ticket) -> None:
        """Each subclass MUST implement what happens after entry."""
        pass


class StandardEntryHandler(ParkingEntryHandler):
    """Regular entry: print a ticket and log it."""
    def post_entry_hook(self, ticket: Ticket) -> None:
        print(f"🎟️  Ticket #{ticket.ticket_id} issued. Please keep it safe.")


class VIPEntryHandler(ParkingEntryHandler):
    """
    VIP vehicles: validated against a whitelist,
    always get a large spot, no fee on exit.
    """
    VIP_PLATES = {"DL01VIP0001", "KA01VIP9999"}

    def validate_vehicle(self, vehicle: Vehicle) -> bool:
        if vehicle.license_plate not in self.VIP_PLATES:
            print(f"🚫 {vehicle.license_plate} is not on the VIP list.")
            return False
        return True

    def post_entry_hook(self, ticket: Ticket) -> None:
        print(f"⭐ VIP vehicle {ticket.vehicle.license_plate} parked. Complimentary parking.")


class SubscriberEntryHandler(ParkingEntryHandler):
    """Monthly subscribers: skip payment, log entry for billing records."""
    def __init__(self, lot, subscriber_db: dict):
        super().__init__(lot)
        self.subscriber_db = subscriber_db   # plate → plan_info

    def validate_vehicle(self, vehicle: Vehicle) -> bool:
        return vehicle.license_plate in self.subscriber_db

    def post_entry_hook(self, ticket: Ticket) -> None:
        plan = self.subscriber_db[ticket.vehicle.license_plate]
        print(f"📋 Subscriber entry logged. Plan: {plan['plan_name']}. No payment needed.")


# ─── Why not just put all this in ParkingLot? ──────────────
# ParkingLot should manage spots and tickets — not VIP lists,
# subscriber databases, or printing logic. Template Method lets
# us extend behavior without touching the core parking logic.

8. Pattern: State — Ticket Lifecycle

🔄
State
Models the ticket's lifecycle as explicit states with valid transitions
🧠 Why State here?
A Ticket can be ACTIVE → PAID → CLOSED. Certain actions are only valid in certain states (you can't exit without paying; you can't pay a closed ticket). Without State pattern, you end up checking if ticket.status == ACTIVE everywhere. With State, each state knows its own valid transitions.
Python · ticket_state.py — State Machine for Ticket
from abc import ABC, abstractmethod
from datetime import datetime


class TicketStateBase(ABC):
    """
    Abstract state. Each concrete state implements only the
    transitions that are valid from that state.
    """
    def pay(self, ticket, fee: float) -> None:
        raise InvalidTransitionError(f"Cannot pay from state: {self.__class__.__name__}")

    def process_exit(self, ticket) -> None:
        raise InvalidTransitionError(f"Cannot exit from state: {self.__class__.__name__}")

    def cancel(self, ticket) -> None:
        raise InvalidTransitionError(f"Cannot cancel from state: {self.__class__.__name__}")

    @property
    @abstractmethod
    def name(self) -> str:
        pass


class InvalidTransitionError(Exception):
    pass


class ActiveState(TicketStateBase):
    """Vehicle is currently parked. Can pay (and then exit)."""

    @property
    def name(self): return "ACTIVE"

    def pay(self, ticket, fee: float) -> None:
        ticket.amount_paid = fee
        ticket.exit_time = datetime.now()
        ticket.state = PaidState()
        print(f"✅ Payment of ₹{fee:.2f} accepted for ticket {ticket.ticket_id}")

    def cancel(self, ticket) -> None:
        # e.g. vehicle leaves in emergency, no payment
        ticket.state = ClosedState()
        print(f"⚠️ Ticket {ticket.ticket_id} cancelled without payment.")


class PaidState(TicketStateBase):
    """Payment done. Vehicle can now exit (gate opens)."""

    @property
    def name(self): return "PAID"

    def process_exit(self, ticket) -> None:
        ticket.spot.vacate()
        ticket.state = ClosedState()
        print(f"🚗 Vehicle {ticket.vehicle.license_plate} exited. Spot {ticket.spot.spot_id} is free.")


class ClosedState(TicketStateBase):
    """Ticket is done. No further transitions allowed."""

    @property
    def name(self): return "CLOSED"
    # All methods raise InvalidTransitionError (inherited defaults)


class StatefulTicket:
    """
    A Ticket that uses the State pattern to manage its lifecycle.
    Compare to the simple Ticket dataclass — this version
    enforces valid transitions at every step.
    """
    def __init__(self, ticket_id, vehicle, spot, entry_time):
        self.ticket_id = ticket_id
        self.vehicle = vehicle
        self.spot = spot
        self.entry_time = entry_time
        self.exit_time = None
        self.amount_paid = 0.0
        self.state: TicketStateBase = ActiveState()   # starts ACTIVE

    def pay(self, fee: float):
        self.state.pay(self, fee)

    def process_exit(self):
        self.state.process_exit(self)

    def cancel(self):
        self.state.cancel(self)

    @property
    def status(self) -> str:
        return self.state.name

💡 Beginner note — State vs. if/else
The simple Ticket uses if ticket.status == "ACTIVE" checks scattered in the code. The State pattern moves those checks INTO the state objects themselves. The benefit grows as the number of states and transitions grows.

9. Pattern: Observer — Availability Notifications

📢
Observer
Notify displays & alerts when spot availability changes
🧠 Why Observer here?
Multiple things care when a spot becomes free or occupied: the display board at the entrance, a mobile app, an SMS alert system. If ParkingLot directly called each of these, it'd be tightly coupled to them all. Observer lets ParkingLot just "announce" a change, and anyone who cares can listen — without ParkingLot knowing about them.
Python · observers.py — Display Board & Alerts
from abc import ABC, abstractmethod
from enums import SpotSize


class AvailabilityObserver(ABC):
    """
    Anyone who wants to know about spot changes implements this.
    Pattern: Observer (this is the "Subscriber" / "Listener")
    """
    @abstractmethod
    def on_spot_change(self, floor_number: int, availability: dict) -> None:
        """
        Called whenever a spot is occupied or vacated on a floor.
        availability = {SpotSize.SMALL: 3, SpotSize.MEDIUM: 7, ...}
        """
        pass


class DisplayBoardObserver(AvailabilityObserver):
    """
    Updates the physical display board at the parking lot entrance.
    In real life this might send data to a screen controller API.
    """
    def on_spot_change(self, floor_number: int, availability: dict) -> None:
        print(f"\n📊 [Display Board] Floor {floor_number} availability updated:")
        for size, count in availability.items():
            status = "✅" if count > 0 else "❌"
            print(f"   {status} {size.name:<8}: {count} spots free")


class CapacityAlertObserver(AvailabilityObserver):
    """
    Sends an alert when a floor is running low on spots.
    Useful for operations team to redirect incoming traffic.
    """
    def __init__(self, threshold: int = 2):
        self.threshold = threshold   # alert when fewer than this many spots free

    def on_spot_change(self, floor_number: int, availability: dict) -> None:
        total_free = sum(availability.values())
        if total_free <= self.threshold:
            print(f"🚨 [ALERT] Floor {floor_number} is nearly full! "
                  f"Only {total_free} spot(s) remaining.")


class LoggingObserver(AvailabilityObserver):
    """
    Logs all spot changes for audit purposes.
    Could write to a file, database, or monitoring system.
    """
    def __init__(self):
        self.log: list[str] = []

    def on_spot_change(self, floor_number: int, availability: dict) -> None:
        from datetime import datetime
        entry = f"{datetime.now().isoformat()} | Floor {floor_number} | {availability}"
        self.log.append(entry)
        # In production: write to a log file or send to monitoring

    def get_log(self) -> list[str]:
        return self.log.copy()


class MobileAppObserver(AvailabilityObserver):
    """
    Pushes availability updates to a mobile app backend.
    (Simulated here — real version would call an API.)
    """
    def on_spot_change(self, floor_number: int, availability: dict) -> None:
        payload = {
            "floor": floor_number,
            "availability": {size.name: count for size, count in availability.items()}
        }
        # In real code: requests.post("https://api.parkinglot.com/update", json=payload)
        print(f"📱 [Mobile App] Pushed update: {payload}")

10. Thread Safety

Real parking lots have many entry/exit gates running simultaneously. In Python, that means multiple threads. Without locks, two threads could assign the same spot to two cars — a classic race condition.

Python · thread_safety_demo.py
"""
Demonstrating the race condition and how our locks prevent it.
"""
import threading
from parking_lot import ParkingLot
from factories import VehicleFactory, SpotFactory, ParkingLotBuilder
from fee_strategy import HourlyFeeStrategy
from enums import SpotSize

# Set up a small lot with only 2 medium spots
lot = ParkingLot("Demo Lot")
lot.set_fee_strategy(HourlyFeeStrategy())
SpotFactory.reset()
ParkingLotBuilder(lot).add_floor(1, {SpotSize.MEDIUM: 2}).build()

tickets = []
lock = threading.Lock()    # for safely appending to tickets list

def park_car(plate: str):
    vehicle = VehicleFactory.create(plate, "car")
    ticket = lot.park_vehicle(vehicle)
    if ticket:
        with lock:
            tickets.append(ticket)

# Simulate 5 cars arriving simultaneously
plates = ["KA01AB0001", "KA01AB0002", "KA01AB0003", "KA01AB0004", "KA01AB0005"]
threads = [threading.Thread(target=park_car, args=(p,)) for p in plates]

for t in threads: t.start()
for t in threads: t.join()

print(f"\n✅ Successfully parked: {len(tickets)} car(s)")
print(f"🚫 Rejected (lot full): {5 - len(tickets)} car(s)")
# Expected: 2 parked, 3 rejected — even with 5 simultaneous requests

# ─── How the lock works (from parking_lot.py) ──────────────
#
# def park_vehicle(self, vehicle):
#     with self._operation_lock:       # ← only ONE thread can be here at a time
#         spot = self._find_spot(vehicle)
#         if spot is None:
#             return None
#         spot.park(vehicle)           # ← atomic: find + park is one locked block
#         ...
#
# Without the lock:
# Thread A reads spot F1-M001 as available ─┐  both find the same spot!
# Thread B reads spot F1-M001 as available ─┘
# Thread A parks Car1 in F1-M001
# Thread B parks Car2 in F1-M001  ← OVERWRITE! Car1 effectively disappears
#
# With the lock:
# Thread A acquires lock, finds F1-M001, parks Car1, releases lock
# Thread B acquires lock, finds F1-M002 (M001 is now taken), parks Car2
# Thread C acquires lock, finds no spot, returns None ← correctly rejected

Double-Checked Locking (for Singleton)

Python · singleton_locking.py — Why two checks?
# In ParkingLot.__new__:
#
# if cls._instance is None:          ← CHECK 1 (fast path, no lock)
#     with cls._lock:
#         if cls._instance is None:  ← CHECK 2 (slow path, inside lock)
#             cls._instance = super().__new__(cls)
#
# Why check TWICE?
#
# Without double-check:
# Thread A sees _instance is None, starts acquiring lock...
# Thread B sees _instance is None, starts acquiring lock...
# Thread A acquires lock, creates instance, releases lock
# Thread B acquires lock (was waiting!), creates ANOTHER instance ← BUG
#
# With double-check:
# Thread A acquires lock, creates instance, releases lock
# Thread B acquires lock, sees instance is NOT None → does nothing ← Correct!
#
# The first check is a fast path: once the singleton is created,
# no thread ever acquires the lock again — that's the performance win.

11. Testability

Clean design makes testing easy. Here's how our patterns make the system testable without needing a real parking lot.

Python · test_parking_lot.py
"""
Unit tests for the parking lot system.
No real infrastructure needed — we use test doubles and
reset the Singleton between tests.
"""
import pytest
from datetime import datetime, timedelta
from unittest.mock import MagicMock

from parking_lot import ParkingLot
from factories import VehicleFactory, SpotFactory, ParkingLotBuilder
from fee_strategy import HourlyFeeStrategy, FlatRateFeeStrategy
from models import Ticket, Vehicle, ParkingSpot
from enums import VehicleType, SpotSize, TicketStatus


# ── Fixture: fresh lot for every test ──────────────────────
@pytest.fixture(autouse=True)
def reset_singleton():
    """
    The Singleton pattern means the same object persists across tests.
    We MUST reset it, or tests interfere with each other.
    This is a known pain point of Singleton — document it!
    """
    ParkingLot._instance = None   # force fresh instance per test
    SpotFactory.reset()
    yield


def make_lot(small=0, medium=5, large=0):
    lot = ParkingLot("Test Lot")
    lot.set_fee_strategy(HourlyFeeStrategy())
    ParkingLotBuilder(lot).add_floor(1, {
        SpotSize.SMALL: small,
        SpotSize.MEDIUM: medium,
        SpotSize.LARGE: large,
    }).build()
    return lot


# ── Test: basic parking ─────────────────────────────────────
def test_car_gets_ticket():
    lot = make_lot(medium=3)
    car = VehicleFactory.create("KA01AB1234", "car")
    ticket = lot.park_vehicle(car)
    assert ticket is not None
    assert ticket.vehicle.license_plate == "KA01AB1234"
    assert ticket.status == TicketStatus.ACTIVE


def test_lot_full_returns_none():
    lot = make_lot(medium=1)
    car1 = VehicleFactory.create("KA01AA0001", "car")
    car2 = VehicleFactory.create("KA01AA0002", "car")
    t1 = lot.park_vehicle(car1)
    t2 = lot.park_vehicle(car2)
    assert t1 is not None
    assert t2 is None     # lot is full after first car


def test_spot_freed_after_exit():
    lot = make_lot(medium=1)
    car = VehicleFactory.create("KA01AB9999", "car")
    ticket = lot.park_vehicle(car)
    assert ticket is not None
    lot.process_exit(ticket.ticket_id)

    # Now the same spot should be available again
    car2 = VehicleFactory.create("KA01AB8888", "car")
    ticket2 = lot.park_vehicle(car2)
    assert ticket2 is not None


# ── Test: fee calculation with mock ────────────────────────
def test_hourly_fee_charged_correctly():
    lot = make_lot(medium=2)
    strategy = HourlyFeeStrategy()
    lot.set_fee_strategy(strategy)

    car = VehicleFactory.create("MH01AB5678", "car")
    ticket = lot.park_vehicle(car)

    # Simulate 2 hours parked by backdating entry_time
    ticket.entry_time = datetime.now() - timedelta(hours=2)
    fee = lot.process_exit(ticket.ticket_id)

    # HourlyFeeStrategy: CAR = ₹50/hr × 2hr = ₹100
    assert fee == 100.0


def test_strategy_swappable():
    """Strategy Pattern test: switching strategy changes the fee."""
    lot = make_lot(medium=2)

    # Park once with hourly strategy
    lot.set_fee_strategy(HourlyFeeStrategy())
    car = VehicleFactory.create("DL01CD1111", "car")
    t1 = lot.park_vehicle(car)
    t1.entry_time = datetime.now() - timedelta(hours=3)
    hourly_fee = lot.process_exit(t1.ticket_id)   # 3hrs × ₹50 = ₹150

    # Park again with flat rate strategy
    lot.set_fee_strategy(FlatRateFeeStrategy())
    car2 = VehicleFactory.create("DL01CD2222", "car")
    t2 = lot.park_vehicle(car2)
    t2.entry_time = datetime.now() - timedelta(hours=3)
    flat_fee = lot.process_exit(t2.ticket_id)    # Always ₹100

    assert hourly_fee == 150.0
    assert flat_fee == 100.0


# ── Test: Observer ─────────────────────────────────────────
def test_observer_notified_on_park():
    lot = make_lot(medium=3)
    mock_observer = MagicMock()
    lot.add_observer(mock_observer)

    car = VehicleFactory.create("KA05EF3456", "car")
    lot.park_vehicle(car)

    # Observer should have been called once (on parking)
    mock_observer.on_spot_change.assert_called_once()
    call_args = mock_observer.on_spot_change.call_args
    floor_num, availability = call_args[0]
    assert floor_num == 1
    assert availability[SpotSize.MEDIUM] == 2   # 3 - 1 = 2 free
⚡ Key testability insight
Because we used Strategy (injectable fee logic) and Observer (injectable listeners), tests can use MagicMock() instead of real implementations. The Singleton requires explicit reset between tests — always document this caveat when using Singleton!

12. Interview Q&A Cards

Click each question to reveal a model answer. Practice answering these out loud before your interview.

How would you design a Parking Lot system? Walk me through your approach. Design ▼

I'd start by clarifying requirements: vehicle types, number of floors, pricing model, and concurrency needs. Then I'd identify the core entities — ParkingLot, ParkingFloor, ParkingSpot, Vehicle, and Ticket — as these map directly to real-world objects.

For design patterns, I'd use:

  • Singleton for ParkingLot — one physical lot means one shared object.
  • Strategy for fee calculation — pricing rules can vary (hourly, flat, peak-hour).
  • Factory for creating spots and vehicles — centralizes validation and ID generation.
  • Observer for display boards — decouples the lot from UI/notification systems.
  • Template Method for entry/exit flow — different entry lanes (VIP, standard) share the same skeleton.
Why use Singleton for ParkingLot? What are its downsides? Pattern ▼

Singleton ensures there's only one ParkingLot object in memory, which matches the physical reality — there's only one lot. Without it, two parts of the system could create separate objects and have inconsistent views of which spots are free.

Downsides:

  • Hard to test: Singleton state persists across tests; you must manually reset it.
  • Global state: Makes the system harder to reason about — any code can access and mutate it.
  • Tight coupling: Code that calls ParkingLot() is tightly coupled to the concrete class.
  • Alternative: Dependency injection — pass the lot object as a parameter rather than having everything call ParkingLot() directly.
How does your design handle a motorcycle parking in a car spot? Logic ▼

This is encoded in the Vehicle.can_fit_in(spot_size) method. A motorcycle can fit in SMALL, MEDIUM, or LARGE spots; a car can fit in MEDIUM or LARGE; a truck only fits in LARGE.

The system searches for the "best fit" first — a motorcycle gets a SMALL spot if available, a MEDIUM if not, etc. This is controlled by the order we check spots. The compatibility table makes business rule changes easy: to say "motorcycles can no longer use car spots," we just update the compatibility dictionary in one place.

How would you make the parking lot thread-safe? Concurrency ▼

The critical section is finding and assigning a spot — if two threads run this simultaneously, they could assign the same spot to two vehicles.

I wrap the find + assign operation in a threading.Lock(). The lock ensures only one thread executes that critical section at a time. For the Singleton creation, I use double-checked locking: check if the instance exists before acquiring the lock (fast path), then check again after acquiring it (to handle the race between the first check and the lock acquisition).

How would you add a new vehicle type, say an Electric Vehicle (EV)? Extensibility ▼

Good question — this tests Open/Closed Principle. I'd need to:

  • Add ELECTRIC to the VehicleType enum.
  • Add a new spot size or a boolean flag like has_charger on ParkingSpot.
  • Update the compatibility table in can_fit_in().
  • Update VehicleFactory to accept "electric" as a string type.
  • Optionally create an EVFeeStrategy with different pricing.

The key is that ParkingLot itself doesn't need to change — that's the benefit of our Factory + Strategy design.

How is your spot assignment strategy? Could it be improved? Optimization ▼

Currently we do a linear scan through all spots on each floor — O(n) per floor. For small lots this is fine. For larger systems:

  • Priority queue: Keep available spots in a min-heap sorted by spot ID. O(log n) to find next spot.
  • Separate lists by type: Maintain available_small, available_medium, available_large sets per floor. O(1) assignment, O(1) removal.
  • Nearest-entry optimization: Assign the spot closest to the entry ramp first, reducing walking distance.

I'd use the separate-lists approach — it's simple, O(1), and easy to understand.

How would you support reservations / pre-booking? Feature Ext. ▼

I'd add a Reservation class with: reservation_id, vehicle, spot (nullable — or assigned at booking), start_time, end_time, and status.

ParkingSpot would gain a reservations list. The spot assignment logic would need to check both current occupancy AND upcoming reservations — if a spot is reserved in the next 30 minutes, don't assign it. A background job (scheduler) would release spots whose reservation window has passed. The fee would be calculated from the reservation time, not just entry time.

How would this design scale to a chain of parking lots? Scale ▼

The Singleton would become a problem at scale — it's process-local. I'd move to:

  • Remove Singleton: ParkingLot becomes a regular class with a lot_id.
  • Database: Spot availability stored in a database (PostgreSQL), not in-memory. Thread safety becomes database-level locking (SELECT FOR UPDATE or optimistic concurrency).
  • Service layer: A ParkingLotService that manages multiple lots and routes vehicles to the nearest one with availability.
  • Cache: Redis to cache spot counts per lot — fast reads for display boards.
  • Event bus: Observer becomes a message queue (Kafka/RabbitMQ) — observers are separate services.

13. SOLID Principles in This Design

S

Single Responsibility

Each class has one job. ParkingLot manages the lot (not fee calculation). FeeStrategy calculates fees (not spot assignment). Ticket stores data (not business logic). Observer handles notifications (not parking).

O

Open/Closed

Add a new pricing model? Create a new FeeStrategy subclass — don't touch ParkingLot. Add a new observer (e.g. SMS)? Create a new AvailabilityObserver subclass. The core is closed to modification, open to extension.

L

Liskov Substitution

Any FeeStrategy subclass can be passed to lot.set_fee_strategy() without ParkingLot knowing or caring which one it is. Any AvailabilityObserver can be added via add_observer(). Substitution works without breaking behavior.

I

Interface Segregation

AvailabilityObserver has exactly one method. FeeStrategy has exactly one method. Classes aren't forced to implement methods they don't need. If we added a separate PaymentObserver interface, observers wouldn't be forced to handle both events.

D

Dependency Inversion

ParkingLot depends on the abstract FeeStrategy, not on HourlyFeeStrategy or FlatRateFeeStrategy. High-level policy (parking logic) doesn't depend on low-level details (pricing math). Both depend on the abstraction.

15. Trade-offs & Design Decisions

Interviewers love it when you acknowledge alternatives. Use these "I chose X over Y because Z" formulas:

⚖️ Singleton vs. Dependency Injection

I chose Singleton for simplicity in this design. In production, I'd prefer DI — pass the ParkingLot instance to whatever needs it. DI makes testing dramatically easier (no global state to reset) and supports multiple lots naturally.

⚖️ Strategy for fees vs. a pricing table in the database

Strategy works well when pricing logic is code-driven (tiered, peak-hour). If pricing rules need to be configurable by non-developers at runtime, a database-driven pricing table is better — then the Strategy reads from the DB rather than having hardcoded rates.

⚖️ Linear spot search vs. priority queue

I used linear search for clarity. For a lot with <500 spots, this is fast enough. For thousands of spots or high-traffic lots, I'd maintain separate sets of available spots per size per floor — giving O(1) spot assignment at the cost of slightly more complex bookkeeping on vacate.

⚖️ Template Method vs. Strategy for entry handling

I used Template Method because the order of steps is fixed (validate → find spot → issue ticket) but individual steps vary. If the entire algorithm for entry could be interchanged wholesale, Strategy would be better. Template Method is appropriate when steps are customizable but the skeleton is invariant.

⚖️ In-memory state vs. database

This design stores everything in memory. That's fine for a single-process system or for an interview. In production, spot availability and tickets must be in a database — so the system survives restarts, supports multiple server instances, and provides audit history.

🎯 Final Advice
In an interview, you rarely need ALL these patterns. Start simple: Singleton + Strategy + basic models. Then layer in Factory, Observer, and State only when the interviewer asks follow-ups. Showing you know when NOT to use a pattern is just as impressive as knowing when to use one.
