Step 1 · ~5 minutes
Requirements & Clarifications
You'll be handed a vague sentence. Your first job is not to design — it's to interrogate. Every clarifying question you ask is a bug you're preventing before writing a single class. Notice how much the prompt below leaves unsaid.
The prompt you'll receive
"Design an inventory management system that tracks products across multiple warehouses. The system needs to handle adding and removing inventory, transferring stock between locations, and alerting when inventory runs low."

This sentence leaves almost everything ambiguous. How many warehouses? Fixed or dynamic? What does "alert" mean — email, webhook, log? Can stock go negative? What if two threads try to modify the same warehouse simultaneously? These aren't nitpicks — each answer changes a class you must write or eliminates one you shouldn't.

Clarifying Questions — with deep reasoning

You
"When you say 'multiple warehouses' — is the set of warehouses fixed at startup, or can we add/remove warehouses dynamically at runtime?"
Why ask this first: "Multiple warehouses" is an architectural fork. If warehouses are dynamic, you need warehouse lifecycle management: create, delete, maybe even merge. If they're fixed, you skip all of that and just configure them in the constructor. This one question can remove an entire class (WarehouseRegistry) from your design. Asking it takes 5 seconds. Missing it means you might design a warehouse-creation flow the interviewer never wanted, using up precious time.
Interviewer
"Keep it simple. Fixed set of warehouses. They're configured when the system initializes."
You
"For the low-stock alerts — is the threshold global per product, or is it warehouse-specific? E.g., Warehouse A might need 50 units of Product X as a buffer, while Warehouse B only needs 10."
Why this matters enormously: If thresholds are global, you store one number per product. Simple. If thresholds are per product per warehouse, you need a 2D mapping: (product_id, warehouse_id) → threshold. The storage structure is completely different. Additionally, "per warehouse" thresholds mean a product can be "low" in Warehouse A but "fine" in Warehouse B — so your alert logic must check per location, not globally. This question defines the shape of your AlertConfig class.
Interviewer
"Per product per warehouse. Different locations have different needs for the same product."
You
"How should alerts be delivered? Should the system send emails, call a webhook, or just invoke some callback interface that the caller provides?"
Why "pluggable" is the right answer: If you hardcode email or logging into your alert logic, you've violated the Open/Closed Principle — adding a new notification channel requires modifying core inventory code. The right design is an Observer pattern: your system fires an event, and whoever subscribed (email sender, Slack bot, dashboard) handles it. This also makes your system unit-testable — you can register a mock listener in tests instead of sending real emails. By asking this question, you signal you know about OCP and the Observer pattern without having to say so explicitly.
Interviewer
"Keep it pluggable. The system calls a callback interface when stock is low. What happens after — email, webhook, logging — is someone else's problem."
You
"What happens if someone tries to remove more units than exist? Should we allow negative inventory, or reject the operation with an error?"
Why this defines your entire validation story: If negative inventory is allowed, you never need to validate before a remove — just decrement and move on. If it's disallowed, you must validate before every remove AND every transfer (both the subtract and add sides). The answer also tells you what exception/error type to define. A surprisingly large portion of your implementation logic hinges on this one answer — yet it takes one sentence to ask.
Interviewer
"Reject them. If someone tries to remove 100 units but we only have 50, that should fail. Same with transfers — validate before moving anything."
You
"What about concurrency? If two processes try to modify the same warehouse's inventory simultaneously — say, one shipment arriving while an order is being fulfilled — do we need to handle that?"
Why you must ask this explicitly: Concurrency changes nearly every method signature. Without it, you write simple Python dicts and plain assignments. With it, you must think about locks, deadlocks (especially for transfers which touch two warehouses), and granularity — do you lock the whole system or per-warehouse? Asking this question before coding shows the interviewer you think about production realities, not just happy-path logic. It also sets up your Thread Safety deep-dive section.
Interviewer
"Yes, concurrency is important. Multiple operations could be happening simultaneously. Make sure operations are thread-safe."
You
"Last one — what's out of scope? Are we managing the product catalog, handling orders, payments, or shipment logistics?"
Why scoping out matters: Calling out what you're NOT building does three things. First, it proves you thought about those things and deliberately excluded them. Second, it protects you from scope creep — if the interviewer drifts toward order management, you can redirect cleanly. Third, it makes your design look complete rather than incomplete. A design with no stated boundaries looks like you forgot things. A design with explicit exclusions looks like engineering judgement.
Interviewer
"Correct. Products are external — they arrive with an ID and name. Orders and payments are handled upstream. Keep focus on inventory tracking."

Final Requirements

In scope

  • Track inventory (product → quantity) across a fixed set of warehouses configured at startup.
  • Add stock to a specific warehouse (receiving shipments).
  • Remove stock from a specific warehouse (fulfilling orders). Reject if insufficient stock.
  • Check availability: given a product and quantity, return which warehouses can fulfill it.
  • Transfer stock between warehouses atomically. Validate before moving anything.
  • Low-stock alerts: per product per warehouse threshold. Fire a pluggable callback when stock drops below threshold.
  • Thread-safe: concurrent operations on different or same warehouses must not corrupt state.

Out of scope

  • Product catalog management (products arrive with an external ID).
  • Order processing, payment, or purchase flow.
  • Persistence / database layer.
  • Warehouse lifecycle (create, delete, merge).
  • GPS, routing, or logistics optimization.
Step 2 · ~5 minutes
Core Entities & Relationships
Scan the requirements for nouns. Circle every one. Then apply a ruthless filter: does this noun have state that changes over time AND behavior that enforces rules on that state? If yes → class. If only data → dataclass. If neither → it's a field on something else. The most common beginner mistake is either making everything a class, or collapsing things that should be separate.

Noun-by-noun evaluation

NounDecisionDeep reasoning — including what breaks if you get it wrong
KeepWarehouse Full class Has mutable state (a dict mapping product → quantity that changes constantly) and rich behavior: add stock, remove stock with validation, transfer, fire alerts. If Warehouse were just a nested dict inside InventoryManager, the validation and alert logic would have to live elsewhere — probably scattered across the manager. That means two places can modify warehouse stock, which is where concurrency bugs are born. Encapsulating stock inside Warehouse means there is exactly one place to enforce "stock can't go negative" — the Warehouse class itself.
KeepInventoryManager Full class This is the façade — the single public entry point for all operations. It owns the collection of warehouses and orchestrates cross-warehouse operations like transfers. Without it, the caller would have to know about individual warehouses, find the right one, lock it, etc. The manager hides that complexity. It's also where the "check availability across all warehouses" feature lives naturally, since it needs to query multiple warehouses in one pass.
KeepAlertConfig Dataclass Pure configuration data: a threshold value for a specific (product, warehouse) combination. It has no behavior of its own — it just holds the number. A dataclass or named tuple is the right choice here. The behavior that uses the config (checking if current stock is below threshold) lives in Warehouse, not in AlertConfig itself.
KeepAlertListener Abstract class / Interface This is the Observer interface. It defines the contract: "any class that wants to receive low-stock notifications must implement on_low_stock()." You never instantiate AlertListener directly — you subclass it (EmailAlerter, SlackAlerter, LogAlerter). If you skip this abstraction and call a specific alerter directly (e.g., send_email() hardcoded in Warehouse), you've broken OCP — adding a new alert channel requires modifying inventory code. The interface decouples inventory logic from notification logic completely.
SkipProduct Just an ID string Products are external per our requirements. The system doesn't manage a product catalog — it just stores quantities indexed by a product ID. A product_id: str field is sufficient everywhere. Creating a full Product class would give us something with no behavior that just wraps a string — that's over-engineering. If requirements changed to include product metadata (weight, category, reorder formula), we'd add it then.
SkipShipment / Order Out of scope These represent the events that trigger add_stock() or remove_stock(). The calling system handles them. Our system only cares about the resulting quantity change, not the cause. Modeling Shipment or Order here would pull in order IDs, customer data, payment state — none of which we need for inventory tracking.

Relationships at a glance

How the four entities connect

InventoryManager owns a dict of warehouse_id → Warehouse. It also holds a list of registered AlertListener instances.

Warehouse owns a dict of product_id → quantity and a dict of (product_id) → AlertConfig. It has a threading.Lock for thread safety.

AlertConfig is a small dataclass with just product_id and threshold.

AlertListener is an abstract base class. Concrete implementations (EmailAlerter, LogAlerter, etc.) register themselves with the InventoryManager and are called when stock drops below threshold.

Step 3 · ~10 minutes
Class Design
For each class, think about: what state does it own, what methods does it expose, and what invariants must it always maintain? The key question for each method is — who is the right owner? The class that owns the state is the right owner of the behavior that changes it.

Warehouse — the heart of the system

warehouse.py (design sketch)
class Warehouse: def __init__(self, warehouse_id): self.id = warehouse_id self._stock: dict[str, int] = {} self._alert_configs: dict[str, AlertConfig] = {} self._lock = threading.Lock()
Why _stock is private (underscore prefix). The only way to change stock is through add_stock() and remove_stock() — never by direct assignment from outside. This is encapsulation: the rules (no negatives, fire alerts) are always enforced because no one can bypass the method. Key insight: If stock were public (self.stock), any caller could write wh.stock["apples"] -= 200 and bypass every validation. Private state + public methods is what makes invariants hold.
def add_stock(self, product_id, quantity, listeners): with self._lock: if quantity <= 0: raise ValueError(...) self._stock[product_id] = \ self._stock.get(product_id, 0) \ + quantity
Why the lock is inside the method The with self._lock block ensures that the read-then-write sequence (get current stock, add to it) is atomic. Without this, two threads could both read "stock = 10", both add 5, and both write "stock = 15" — resulting in 15 instead of the correct 20. This is called a race condition.
def remove_stock(self, product_id, quantity, listeners): with self._lock: current = self._stock.get(product_id, 0) if current < quantity: raise InsufficientStockError(...) self._stock[product_id] = current - quantity self._check_alert(product_id, listeners)
Why validate-then-modify inside the lock The check (current < quantity) and the modification (-= quantity) must happen together atomically. If you check outside the lock and modify inside, another thread could remove stock between your check and your write — making stock go negative. Checking and modifying inside the same lock block eliminates this window entirely. If you forget the lock here: Two orders for the last 10 units could both pass the validation check, both subtract 10, and you end up with -10 inventory.
def _check_alert(self, product_id, listeners): # called inside lock already cfg = self._alert_configs.get(product_id) if cfg and \ self._stock.get(product_id, 0) < cfg.threshold: for listener in listeners: listener.on_low_stock( self.id, product_id, self._stock[product_id])
Why private helper, not public method Alert checking is an internal consequence of stock removal — it's not something external callers invoke directly. Making it a private helper (_check_alert) keeps the public interface clean. The caller just calls remove_stock() and alerts fire automatically as a side-effect.

InventoryManager — the façade

inventory_manager.py (design sketch)
class InventoryManager: def __init__(self, warehouse_ids: list[str]): self._warehouses = { wid: Warehouse(wid) for wid in warehouse_ids } self._listeners: list[AlertListener] = []
Why a dict, not a list Callers identify warehouses by ID (a string like "warehouse_nyc"). A dict gives O(1) lookup by ID. A list would require iterating all warehouses to find the right one — O(n) for every operation. For 100 warehouses, this is a 100× slowdown on every stock operation.
def transfer(self, product_id, quantity, from_wh, to_wh): # Must lock BOTH warehouses to avoid # partial transfers and deadlocks first, second = sorted([from_wh, to_wh]) with first._lock: with second._lock: from_wh._transfer_out(product_id, qty) to_wh._transfer_in(product_id, qty)
Why sorted lock ordering prevents deadlocks Imagine Thread A transfers A→B (locks A, then B) while Thread B simultaneously transfers B→A (locks B, then A). Both threads hold one lock and wait for the other — a deadlock. Sorted ordering ensures both threads always acquire locks in the same order (alphabetically), so deadlock is impossible. This is a classic concurrency interview topic. Without sorted ordering: Your transfer() can deadlock under concurrent load, hanging indefinitely — a production-killing bug.
def check_availability(self, product_id, quantity) -> list[str]: return [ wid for wid, wh in self._warehouses.items() if wh.get_stock(product_id) >= quantity ]
Why this lives in the Manager, not Warehouse check_availability queries multiple warehouses and returns a list of IDs. A single Warehouse instance has no visibility into its siblings — it only knows its own stock. Cross-entity queries must live in the object that owns the collection of entities: the Manager.

AlertListener — the Observer interface

alert_listener.py
from abc import ABC, abstractmethod class AlertListener(ABC): @abstractmethod def on_low_stock(self, warehouse_id: str, product_id: str, current_stock: int) -> None: ...
Why ABC, not a plain class Using Python's ABC (Abstract Base Class) makes AlertListener un-instantiable directly and forces every subclass to implement on_low_stock(). If you forget to implement it, Python raises a TypeError at instantiation time — a fast, clear error instead of a silent runtime failure where no alerts fire because the method is missing. Concrete example of a subclass:
class LogAlerter(AlertListener):
def on_low_stock(self, wh_id, prod_id, qty):
print(f"LOW STOCK: {prod_id} in {wh_id}: {qty}")

Key Flows — Step by Step
Walk through each major operation as a sequence of steps. In an interview, tracing a flow on the whiteboard shows you understand runtime behavior, not just class structure.

Flow 1: remove_stock() with alert trigger

1
Caller: manager.remove_stock("wh_delhi", "apples", 85)
The caller doesn't know anything about warehouses directly. They call the manager's public API with a warehouse ID string.
↓
2
InventoryManager: Look up warehouse object
self._get_warehouse("wh_delhi") does a dict lookup → returns the Warehouse object. Raises WarehouseNotFoundError if the ID doesn't exist.
↓
3
Warehouse: Acquire lock
with self._lock: — the thread blocks here until no other thread holds the lock. This ensures the read-validate-modify sequence below is atomic.
↓
4
Warehouse: Validate
Read current stock: current = self._stock.get("apples", 0) → say it's 100. Check: 100 < 85 is False → validation passes. If validation failed, raise InsufficientStockError immediately (lock is released automatically by the with block).
↓
5
Warehouse: Modify & check alert
Set self._stock["apples"] = 100 - 85 = 15. Then call self._check_alert("apples", listeners). Alert config says threshold = 20. Current stock (15) < threshold (20) → alert fires. All registered listeners have on_low_stock("wh_delhi", "apples", 15) called.
↓
6
Lock released, call returns
The with self._lock block exits, releasing the lock. Other threads waiting on this warehouse can now proceed.

Flow 2: transfer() with deadlock prevention

1
Caller: manager.transfer("apples", 30, "wh_mumbai", "wh_delhi")
A transfer touches two warehouses. The challenge: we need both warehouse locks held simultaneously so the operation is atomic — no other thread sees a state where stock has been removed from one but not yet added to the other.
↓
2
Manager: Sort lock order
sorted(["wh_mumbai", "wh_delhi"]) → ["wh_delhi", "wh_mumbai"]. Regardless of which direction the transfer goes, we always acquire "wh_delhi"'s lock first. This prevents deadlocks. See Thread Safety section for the full explanation.
↓
3
Acquire first lock, then second
with first._lock: with second._lock: — both locks now held. The "inside" body runs exclusively — no other thread can touch either warehouse.
↓
4
_transfer_out then _transfer_in
Validate and subtract from source. Then add to destination. If the validation in step one fails (insufficient stock), an exception is raised and neither warehouse is modified — the operation is all-or-nothing.
↓
5
Both locks released
Nested with blocks exit, releasing both locks in reverse acquisition order. State is consistent: stock moved cleanly from source to destination with no windows of inconsistency.
Step 6 · Deep Dive
Thread Safety
Concurrency is the most interview-differentiating topic in this problem. Understanding why specific choices were made — not just that locks exist — is what separates junior from senior candidates.

Why per-warehouse locks, not one global lock?

❌ One global lock (bad)
class InventoryManager:
    def __init__(self):
        self._lock = threading.Lock()

    def add_stock(self, wh_id, prod, qty):
        with self._lock:
            # ALL warehouses blocked
            # even if they're unrelated
A global lock means operations on completely unrelated warehouses — say Delhi and Mumbai — block each other. Only one operation can run system-wide at a time. This is a huge bottleneck in a real system with hundreds of warehouses.
✅ Per-warehouse locks (correct)
class Warehouse:
    def __init__(self):
        self._lock = threading.Lock()

# Result:
# Thread A modifies wh_delhi
# Thread B modifies wh_mumbai
# → Both proceed simultaneously ✓
Per-warehouse locks allow operations on different warehouses to run in parallel. Only threads touching the same warehouse serialize. This is much more scalable — parallelism is O(warehouses) instead of 1.

The deadlock problem in transfer()

Transfers are dangerous because they require holding two locks simultaneously. Consider this scenario:

❌ No lock ordering — DEADLOCK
Thread A: transfer(Delhi → Mumbai)
  → acquires Delhi lock ✓
  → waiting for Mumbai lock...

Thread B: transfer(Mumbai → Delhi)
  → acquires Mumbai lock ✓
  → waiting for Delhi lock...

→ Both wait forever. DEADLOCK.
✅ Sorted lock ordering — SAFE
Both threads sort: [Delhi, Mumbai]
Both acquire Delhi first, then Mumbai

Thread A: acquires Delhi, then Mumbai ✓
Thread B: tries Delhi — blocked, waits

When A finishes:
Thread B: acquires Delhi, then Mumbai ✓

→ No deadlock. Correct serialization.
The intuition behind sorted lock ordering

Deadlocks happen when two threads each hold one lock and wait for the other. The fix is to ensure all threads always acquire locks in the same global order. Since dictionary keys are strings, sorting alphabetically gives a consistent, deterministic order that every thread follows — no matter which direction the transfer goes.

This technique is called Lock Ordering (also known as the "resource hierarchy" solution to deadlock). It's a classic concurrency pattern and an extremely common interview question at senior levels.

The race condition in remove_stock()

❌ Check outside lock — RACE CONDITION
def remove_stock(self, product_id, qty):
    current = self._stock.get(product_id, 0)
    # THREAD SWITCH HAPPENS HERE
    if current < qty:
        raise InsufficientStockError(...)
    with self._lock:  # lock acquired too late!
        self._stock[product_id] -= qty

# Thread A and B both check current=10,
# both pass validation, both subtract 10
# Result: stock = -10 ← IMPOSSIBLE STATE
✅ Check and modify inside lock — SAFE
def remove_stock(self, product_id, qty):
    with self._lock:
        current = self._stock.get(product_id, 0)
        if current < qty:
            raise InsufficientStockError(...)
        self._stock[product_id] = current - qty

# Thread A acquires lock, checks, subtracts.
# Thread B waits. When B runs, current
# reflects A's change → validation correct.
# Stock never goes negative. ✓
Step 7 · Deep Dive
Design Patterns
Good LLD doesn't just mean working code — it means recognizing the recurring problems in your design and applying proven solutions. Here are three patterns baked into this system, each solving a real problem.
🏭 Façade Pattern InventoryManager

The problem it solves: The caller shouldn't have to know about Warehouse objects, their lock management, or which warehouse to look up for a given ID. That's internal complexity.

How it's applied: InventoryManager is the single entry point for all operations. Callers call manager.add_stock("wh_delhi", "apples", 100) — they don't interact with Warehouse directly at all. The manager handles warehouse lookup, lock orchestration for transfers, and listener notification.

The concrete benefit: If you later change the internal representation (say, switching from dict to a database-backed structure), no caller code changes — only the manager changes.

👁 Observer Pattern AlertListener

The problem it solves: The inventory system shouldn't need to know how alerts are delivered. If it calls send_email() directly, you can never add Slack notifications without modifying inventory code. That's an OCP violation.

How it's applied: AlertListener defines a contract. Any class implementing on_low_stock() can register with the manager. When stock drops below threshold, all registered listeners are notified. The inventory code calls a method on an interface — it has zero knowledge of email, Slack, or anything else.

The concrete benefit: Adding a new notification channel is zero-change to inventory code — just write a new subclass and register it. You can also add multiple listeners simultaneously (email + Slack + log, all at once).

🔒 Template Method (lock-then-act) Warehouse operations

The problem it solves: Every warehouse operation follows the same skeleton: acquire lock → validate → modify → release lock → fire side effects. Without a consistent structure, developers might forget the lock, or check before locking, or fire alerts before modifying.

How it's applied: Each warehouse method follows the same rigid structure. The lock acquisition is always the first thing in the method body. Validation always precedes modification. Alert checking always follows modification. This isn't formalized as a classic Template Method class hierarchy, but the pattern — a fixed skeleton with hook points — is identical.

The concrete benefit: Consistency. Every operation in the codebase is safe by construction because they all follow the same template.

🏗 Strategy Pattern (extensibility) AlertListener subclasses

The problem it solves: Different production environments need different alert strategies. Unit tests need a mock that records calls. Production needs email. A monitoring dashboard needs a webhook. Each is a different "strategy" for the same operation.

How it's applied: The AlertListener interface lets you swap notification strategies at runtime by choosing which subclass to register. The inventory system is parameterized on the strategy — it never hardcodes a choice.

The concrete benefit: Testability. In pytest, you register a MockAlerter that records calls. You assert it was called with the right arguments. No emails sent, no side effects, fast tests.

SOLID Principles
SOLID is a checklist of five design principles. The goal isn't to recite them — it's to show you applied them. For each one, the question is: where does this design follow the principle, and what would happen if it didn't?
S
Single Responsibility Principle
"A class should have only one reason to change."
Violation: If Warehouse also contained email-sending logic, it would have two reasons to change: inventory rules AND notification format changes.
Applied: Warehouse manages stock. AlertListener handles notifications. AlertConfig holds thresholds. InventoryManager coordinates. Each has one job.
O
Open/Closed Principle
"Open for extension, closed for modification."
Violation: Hardcoding send_email() in _check_alert(). Adding Slack would require modifying Warehouse — which is "closed" and shouldn't be touched for new features.
Applied: Adding a new alert channel = write a new AlertListener subclass + register it. Zero changes to existing classes. The system is extended, not modified.
L
Liskov Substitution Principle
"Subclasses must be substitutable for their base class."
Violation: A subclass of AlertListener that throws an exception inside on_low_stock() would break the inventory system's assumption that listeners complete without error.
Applied: All AlertListener subclasses honor the contract: accept the three arguments, return None, don't throw. The caller (Warehouse._check_alert) can treat any subclass identically.
I
Interface Segregation Principle
"Clients shouldn't be forced to implement interfaces they don't use."
Violation: A fat AlertHandler interface with on_low_stock(), on_overstock(), on_shipment_arrived() forces every implementer to stub out methods they don't care about.
Applied: AlertListener has exactly one method: on_low_stock(). It's the only thing the inventory system needs. Implementers only provide what's actually used.
D
Dependency Inversion Principle
"Depend on abstractions, not concretions."
Violation: InventoryManager.__init__ instantiates ConsoleAlerter() directly. Now the manager depends on a concrete implementation — you can't swap it without modifying the manager.
Applied: InventoryManager depends on AlertListener (the abstract interface), not ConsoleAlerter or EmailAlerter. Listeners are injected at runtime via register_listener(). The manager has zero knowledge of how alerts are delivered.
Step 9
Common Interview Q&A
These are the follow-up questions interviewers actually ask after you walk through your design. Having crisp, reasoned answers here is what pushes you from "decent" to "strong hire."
How would you prevent overselling — e.g., two orders claiming the last 10 units simultaneously?

This is exactly what the per-warehouse lock in remove_stock() prevents. The read-validate-modify sequence inside with self._lock is atomic. Thread A acquires the lock, reads stock = 10, validates 10 ≥ 10 (passes), subtracts → stock = 0, releases lock. Thread B then acquires the lock, reads stock = 0, validates 0 ≥ 10 (fails) → raises InsufficientStockError. Exactly one order succeeds.

If you're asking about reservations — e.g., "hold 10 units while the user is on the payment screen" — that requires a reservation/hold pattern: a separate reserved field alongside available. Stock is split into (available, reserved, sold). reserve() moves units from available to reserved; confirm() moves from reserved to sold; cancel() returns from reserved to available. This is how e-commerce systems like Amazon handle cart-level holds.

How would you handle inventory in-transit between warehouses?

Today's transfer() is instant — stock vanishes from source and appears at destination atomically. In the real world, a truck takes 2 days. During that time, the stock is neither here nor there — it's in-transit.

The solution is a third state in the data model. Instead of just available_stock, each warehouse tracks three quantities: available, in_transit_out (removed from source, not yet arrived), and in_transit_in (expected to arrive). The transfer becomes a two-phase operation:

Phase 1 (on dispatch): source._available -= quantity source._in_transit_out += quantity Create a Transfer object with status=IN_TRANSIT Phase 2 (on arrival, triggered externally): source._in_transit_out -= quantity dest._available += quantity Transfer.status = COMPLETED

This also naturally enables the "check availability" query to be smarter: "available now" vs "available including incoming transfers."

Your lock is per-warehouse. What if you need to check availability across all warehouses atomically?

Today's check_availability() queries each warehouse one at a time. In theory, a warehouse's stock could change between the first and last query, so the result isn't a "snapshot" of the whole system at one moment.

For most use cases this is fine — availability checking is advisory, not transactional. The actual reservation happens in a subsequent remove_stock() which is fully atomic.

If you needed a truly consistent global snapshot, you'd have to acquire all warehouse locks at once (sorted to prevent deadlock), perform all reads, then release all locks. This is expensive and should only be done if the use case demands it — for example, a regulatory audit report requiring point-in-time accuracy.

How would you add support for product expiry / FIFO lot management?

Currently, _stock is a flat dict[product_id → quantity]. For expiry tracking, you need lot-level granularity: each batch of stock has a lot ID and expiry date.

The change: _stock becomes dict[product_id → list[Lot]], where each Lot has (lot_id, quantity, expiry_date). When removing stock, you implement FIFO: always consume from the lot with the nearest expiry date first. When checking availability, you can filter out expired lots entirely.

@dataclass class Lot: lot_id: str quantity: int expiry_date: date def remove_stock(self, product_id, quantity, ...): with self._lock: lots = sorted(self._stock.get(product_id, []), key=lambda l: l.expiry_date) # consume from earliest-expiry lots first (FIFO) remaining = quantity for lot in lots: if remaining <= 0: break taken = min(lot.quantity, remaining) lot.quantity -= taken remaining -= taken if remaining > 0: raise InsufficientStockError(...)

The AlertConfig threshold logic also gains a new dimension: alert when any lot expires within N days, even if total quantity is fine.

Why did you use ABC for AlertListener rather than just duck-typing?

Python technically supports duck-typing — if a class has on_low_stock(), it "works" without inheriting from AlertListener. So why bother with ABC?

Three reasons: (1) Error timing. With ABC, if you write a class that inherits from AlertListener but forget to implement on_low_stock(), Python raises TypeError at instantiation time — before the class is ever registered. With duck-typing, you get a runtime AttributeError only when the alert fires, potentially in production at 2am. (2) Documentation. The abstract base class is a contract that's discoverable in the codebase. New developers know exactly what interface to implement. (3) Type-checking. Static analyzers (mypy, pyright) can verify that everything registered is actually an AlertListener, catching errors before runtime.

How would you make this system persistent across restarts?

Currently all state is in-memory — restart = data lost. Adding persistence means deciding where to write state and when.

Option 1 — Database-backed: Each add_stock() and remove_stock() writes to a database row. The Warehouse's _stock dict becomes a write-through cache — changes are persisted immediately. On startup, InventoryManager.__init__ reads current stock from the database to populate the in-memory cache.

Option 2 — Event sourcing: Instead of storing the current quantity, store every operation as an immutable event: (timestamp, warehouse, product, delta). Current stock = replay all events. This gives you a complete audit trail and the ability to time-travel to any past state. Kafka or a simple append-only log works as the event store.

The current design supports both — since all state changes funnel through add_stock() and remove_stock(), you only need to add persistence hooks in two places.
