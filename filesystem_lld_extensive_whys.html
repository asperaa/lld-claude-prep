Step 1 · ~5 minutes
Requirements & Clarifications
The interviewer gives you a vague sentence. Your first job is not to design — it is to interrogate. Every question you ask is a bug you're preventing before you write a single class. Interviewers expect you to shape an ambiguous prompt into a concrete scope. A file system prompt especially looks simple on the surface but hides many decisions that change your entire class diagram.
The prompt you'll receive
"Design a file system. It should support creating, reading, updating, and deleting files and directories."

Four verbs. That's it. But behind those four verbs lie: a permission model, a path resolution algorithm, hard links vs soft links, metadata tracking, concurrency semantics, and the entire Composite pattern. Each answer you get from the interviewer eliminates a class or forces you to add one. Start asking.

Clarifying Questions — with deep reasoning

You
"Does this need to support multiple users with separate ownership and permissions, or is it a single-user system?"
Why this question defines two completely different designs: A single-user system has no permission model — every operation succeeds or fails based purely on whether the path exists. A multi-user system needs a Permission class, an owner_id field on every node, a check_permission(user, node, operation) method, and a User class. That's potentially 3–4 extra classes and a permission check on every single operation. If you design for single-user and the interviewer meant multi-user, you will have to tear out your FileSystem methods and re-thread user context through every call. Ask first.
Interviewer
"Support multiple users. Each file/directory has an owner. Permissions are read, write, execute — like Unix."
You
"Should the system support hard links and symbolic (soft) links, or just regular files and directories?"
Why this changes the entire node model: In a plain design, every path maps to exactly one node object. With hard links, multiple paths can point to the same underlying file data — a single File object can be referenced from two completely different directory locations. This means you cannot store the path inside the file object (because it has two paths). It also means deletion becomes tricky: deleting one path should not delete the data if another path still references it — you need a reference count. Symbolic links are even more complex: a symlink is itself a separate file whose content is a path string, and you need to handle dangling symlinks (pointing to deleted targets). If you skip this question and start designing, you'll build a simpler model that cannot support either concept.
Interviewer
"Support symbolic links. Skip hard links to keep scope manageable."
You
"Should we support file metadata — creation time, last modified time, file size? And do we track the actual file content, or just the structure?"
Why the content question determines one of the most important design decisions: If you track actual content (strings, bytes), then files have a content field and methods like read() and write(data). If you only model structure (paths, names, metadata), files are lightweight metadata objects with no content. The former is needed for a realistic in-memory file system. The latter is enough if the interviewer wants to focus on directory traversal and search. The size field also matters: if you store content, size is derived from len(content). If you store only metadata, size is an explicit integer field. Getting this wrong means your method signatures are either too rich or too shallow.
Interviewer
"Track content as strings (in-memory). Track creation time, modified time, and size. Size should update automatically when content changes."
You
"Should we support directory size — meaning the total size of all files inside a directory recursively?"
Why this one question unlocks the Composite pattern: If yes, then both File and Directory need a get_size() method that returns an integer. A file returns its own content size. A directory returns the sum of get_size() on all its children. This is exactly the Composite pattern — a uniform interface on leaf and container objects. If no, you only need size on File. Ask before you decide whether to use Composite, because the interviewer is often specifically testing whether you'll reach for it.
Interviewer
"Yes — get_size() should work recursively for directories."
You
"Do we need to support search — finding all files matching a name pattern or extension within a directory tree?"
Why this tests your Iterator/Visitor pattern knowledge: A search over a directory tree is a tree traversal. If you embed the search logic directly in the Directory class, you have tight coupling: changing the traversal order or adding a new filter type requires modifying Directory. The clean design is a separate FileSearcher or FindVisitor that traverses the tree independently. If the interviewer says yes to search, you have a natural opening to introduce either DFS/BFS traversal or the Visitor pattern — both are impressive additions in an LLD interview.
Interviewer
"Yes — support search by name and by extension."
You
"Should we handle concurrency — multiple threads reading/writing the same file or directory simultaneously?"
Why this is the last thing to scope, not the first: Thread safety in a file system involves reader-writer locks: multiple readers can hold a file simultaneously (reading doesn't corrupt state), but writers must be exclusive (two simultaneous writes produce garbage). If in scope, every File and Directory needs a threading.RLock or threading.Lock, and every method that mutates state must acquire the lock. This is a significant implementation complexity. Ask the interviewer explicitly. For most 45-minute interviews, they will say "acknowledge it but keep the main design thread-safe via locks, don't implement the full lock protocol." Knowing to ask signals maturity.
Interviewer
"Acknowledge it in the design. Show where you'd add locks. Don't implement the full protocol."

In Scope vs Out of Scope

In Scope
  • Create, read, write, delete files and directories
  • Multi-user ownership with Unix-style permissions (rwx)
  • Symbolic links with dangling link detection
  • Recursive get_size() via Composite pattern
  • File metadata: creation time, modified time, size
  • Search by name or extension across directory tree
  • Absolute path resolution from root
  • Thread-safety acknowledgement with lock placement
Out of Scope
  • Hard links (reference-counted inodes)
  • Disk persistence (on-disk format, inodes, blocks)
  • Full reader-writer lock protocol implementation
  • File compression or encryption
  • Network file system (NFS) semantics
  • File watching / change notification events
Step 2 · ~5 minutes
Core Entities — What Exists and Why
Before writing a single class, name your entities and justify each one. The key question is: "Is this a class, or is it just a field on another class?" An entity becomes a class when it has its own rules to enforce or its own behaviour that would be wrong to put elsewhere. Pure data with no behaviour stays as a simple dataclass or attribute.
EntityClass or Dataclass?Why it exists and what it owns
FileSystemNode Abstract Class The root of the Composite hierarchy. Both File and Directory share: a name, owner, permissions, timestamps, and the get_size() interface. Making this abstract forces every node type to implement get_size() — the compiler/type-checker enforces it. Without this, you'd have duplicated fields in both File and Directory and no guarantee both classes support the same interface.
File Class A leaf node. Has behaviour: read() and write(data) enforce rules — write updates both content and modified_at and recalculates size. If you stored these as a plain struct, nothing would stop content and size drifting apart. The rule "size is always the length of content" must be enforced by the class.
Directory Class A composite node. Owns a children: dict[str, FileSystemNode]. Has rules: no two children with the same name, only the directory's owner or a superuser can add/remove children, deletion must be recursive or checked for emptiness. These rules must live on Directory, not on the caller.
SymbolicLink Class A special leaf that holds a target_path: str instead of content. Behaves like a node in the tree but redirects all real operations to its target. Deserves its own class because its resolution logic (follow the path, detect cycles, detect dangling links) is non-trivial and completely unlike File or Directory behaviour.
Permission Class / Enum flags Encodes read/write/execute for owner, group, and others — the Unix 9-bit model. Has a method can(user, operation) → bool that checks whether a given user has permission. Permission checking logic must not be scattered across every operation — it lives here and is called centrally.
User Class Has a user_id, a username, and a groups: set[str]. Permission checks need to know both the user's direct identity and their group memberships. A plain string user_id field cannot express group membership. A class is needed.
FileSystem Class (Orchestrator) The single entry point. Owns the root directory, the user registry, and all cross-entity operations like path resolution and search. External code should only call FileSystem — never Directory.add_child() directly. This is the Facade pattern.
FileMetadata Dataclass (fields, not class) Creation time, modified time, size — pure data with no behaviour. These are fields directly on FileSystemNode, not a separate class. There is no rule to enforce on metadata in isolation — size's rule ("stay in sync with content") is enforced by File.write(), not by a metadata object.
Why FileSystem is an orchestrator, not just a helper

Beginners often make Directory responsible for path resolution — put resolve_path("/a/b/c") on the root directory. But path resolution needs to traverse multiple directories, and when it hits a SymbolicLink, it needs to start the resolution from the root again. That means Directory.resolve_path() would need a reference back to the root — a circular dependency. FileSystem owns the root and therefore is the only entity that can resolve absolute paths correctly. Orchestrators exist precisely because some operations cannot be owned by any single domain object.

The Composite insight — the most important pattern in this problem

A directory can contain files and other directories. That's a tree. The genius of the Composite pattern is: treat all nodes uniformly. get_size() on a File returns its content length. get_size() on a Directory returns the sum of get_size() on all children. The caller doesn't need to know what type of node they have — they just call get_size(). Without Composite, every tree traversal becomes a mess of if isinstance(node, File): ... elif isinstance(node, Directory): ... spread across the codebase. Say this to the interviewer before you start coding.

Step 3 · ~15 minutes
Class Design — Every Line Justified
Design top-down: start with the abstract base (the Composite root), then the leaf nodes, then the orchestrator. For every field: "what breaks if this field doesn't exist?" For every method: "who else would own this logic if I removed it from here?"

FileSystemNode — The Composite Abstract Base

The abstract base class that both File and Directory inherit from. Every field here is shared by both. Every abstract method here is a promise that all node types must keep.

FileSystemNode — shared fields and abstract interface
from abc import ABC, abstractmethod from datetime import datetime from uuid import uuid4 class FileSystemNode(ABC): def __init__(self, name, owner): self.node_id = str(uuid4()) self.name = name self.owner_id = owner.user_id self.permissions = Permission.default() self.created_at = datetime.now() self.modified_at = datetime.now() self._lock = threading.RLock()
Why ABC and not a regular base class? If FileSystemNode is a regular class, nothing stops someone from instantiating it directly — node = FileSystemNode("x", user). This object has no get_size() implementation and no children. It's a broken phantom object. ABC makes instantiation of the base class a TypeError. The abstract methods are a contract: you cannot create a FileSystemNode unless you implement get_size(). The design is enforced at the language level.
Without ABC: A developer accidentally creates a raw FileSystemNode. Calling get_size() either raises AttributeError or returns a wrong value. The bug surfaces during traversal, far from where the object was created.
With ABC: The attempt to instantiate FileSystemNode directly raises TypeError: Can't instantiate abstract class immediately, at the line where the mistake was made.
self.node_id = str(uuid4())
Why a UUID node_id when we already have a name? Names are mutable — a file can be renamed. If other parts of the system reference a node by its name and the name changes, those references break. A UUID is stable for the lifetime of the node. It also makes hard links easy to model later: two directory entries (two names/paths) both pointing to the same node_id. Without a stable ID, you'd have no way to know two paths refer to the same underlying data.
self._lock = threading.RLock()
Why RLock (re-entrant lock) and not a plain Lock? A plain Lock deadlocks if the same thread tries to acquire it twice. This happens naturally in a file system: Directory.get_size() acquires the directory's lock, then calls child.get_size() which also acquires the child's lock. If the child were the same object (a directory containing itself — a cycle), a plain Lock would deadlock. RLock (re-entrant lock) allows the same thread to acquire it multiple times without deadlocking. For a tree structure, this is the safe default.
@abstractmethod def get_size(self) -> int: ...
Why is get_size() abstract here? This is the Composite pattern's key method. The abstract declaration says: "I don't know how to compute size — each subclass knows its own rule." File returns content length. Directory returns recursive sum. SymbolicLink returns size of its target (or 0 if dangling). By making it abstract on the base class, every caller in the system can simply call node.get_size() without caring what type of node they have. The polymorphism does the work.

File — the Leaf Node with Write Rules

File — content management and metadata sync
class File(FileSystemNode): def __init__(self, name, owner): super().__init__(name, owner) self._content: str = "" self.size: int = 0
Why store content as a private field _content? Content and size must always be consistent: size == len(content) is a class invariant. If content is public, any external code can do file.content = "hello" and size silently becomes stale. The underscore prefix signals "use the methods, not this field directly." Methods read() and write() are the public interface — they enforce the invariant.
def read(self, user) -> str: perm.check(user, self, READ) with self._lock: return self._content
Why check permission before acquiring the lock? Permission checking is a pure read on the permission object — it doesn't mutate anything and is very fast. Acquiring a lock is potentially a blocking wait. If the user has no permission, there is no point waiting for the lock — fail fast and fail early. The pattern is: validate first, mutate second. The same principle as the two-phase validation in place_order(): check everything before touching any shared resource.
def write(self, user, data: str): perm.check(user, self, WRITE) with self._lock: self._content = data self.size = len(data) self.modified_at = datetime.now()
Why update three things atomically inside the lock? These three fields (content, size, modified_at) are the write's observable effects. They must all update together or not at all — there must never be a moment where _content is updated but size still reflects the old length. The with self._lock block ensures no other thread can read this file between the three assignments. This is the same reasoning as database transactions: all-or-nothing atomicity.
Without the lock: Thread A starts writing. It updates _content to "hello world". Before it updates size, Thread B reads. Thread B sees _content = "hello world" (length 11) but size = 0. Inconsistent state. Any code using size for allocation or display is now wrong.
def get_size(self) -> int: return self.size
Why return self.size instead of len(self._content)? Both are equivalent because write() keeps them in sync. But size is an O(1) field read. len(self._content) on a very large file is also O(1) in Python, but in a language with different string models it could be O(n). More importantly: size is the canonical public field. Returning it here reinforces that size is the truth. _content is private. External code (including Directory's recursive sum) should never reach into _content.

Directory — the Composite Node

Directory — children management and recursive operations
class Directory(FileSystemNode): def __init__(self, name, owner): super().__init__(name, owner) self._children: dict[str, FileSystemNode] = {}
Why dict[str, FileSystemNode] and not list[FileSystemNode]? Lookup by name is the most common operation in any directory: "does a file named 'x' exist in this directory?" With a dict, that's O(1). With a list, it's O(n) — scan every child. A directory with 10,000 entries would scan all 10,000 entries on every path component resolution. Directories in real file systems use hash tables internally for exactly this reason. Use a dict unless you have a specific reason not to.
def add_child(self, user, node: FileSystemNode): perm.check(user, self, WRITE) with self._lock: if node.name in self._children: raise DuplicateNameError(node.name) self._children[node.name] = node
Why raise DuplicateNameError instead of silently overwriting? Silent overwriting destroys data. If a directory already has report.txt and you add another file named report.txt, the original file's data disappears permanently with no warning. Real file systems raise errors and require the caller to explicitly delete or rename first. The loud failure principle: never silently discard data. A lost file is far harder to debug than a clear DuplicateNameError at the point of insertion.
With silent overwrite: A script creates a file, processes data, then creates another file with the same name (a bug). The original file is gone. No error, no trace. The developer spends hours wondering why their output file is always empty.
def remove_child(self, user, name: str): perm.check(user, self, WRITE) with self._lock: if name not in self._children: raise FileNotFoundError(name) child = self._children[name] if isinstance(child, Directory) \ and child._children: raise DirectoryNotEmptyError(name) del self._children[name]
Why refuse to delete a non-empty directory? This matches Unix rmdir semantics — you must explicitly delete contents before removing the directory, or use a force-recursive delete (rm -rf). The reason: recursively deleting a directory with thousands of files is irreversible and potentially catastrophic. By requiring an explicit choice — "I know this is non-empty and I'm choosing to delete everything" — you prevent accidents. The FileSystem's delete() method can offer a recursive=True flag for intentional bulk deletion, but the Directory itself refuses by default.
def get_size(self) -> int: with self._lock: return sum( child.get_size() for child in self._children.values() )
Why does this look so simple? This is the Composite pattern paying off. Each child (whether File, Directory, or SymbolicLink) knows how to compute its own size. The directory just asks each child and sums. There is no isinstance check, no special casing, no branching. A 10-deep nested directory structure is handled by this one line through recursive polymorphism. If you had not declared get_size() abstract on the base class, this clean delegation would be impossible — you would be writing if isinstance(child, File): ... elif isinstance(child, Directory): ... here.

SymbolicLink — the Redirecting Node

SymbolicLink — target path and dangling detection
class SymbolicLink(FileSystemNode): def __init__(self, name, owner, target_path: str): super().__init__(name, owner) self.target_path = target_path
Why store target_path as a string, not a reference to the target node? A symbolic link stores a path string, not a direct object reference. This is the defining property of symlinks. The path is resolved at the time of use, not at creation time. This means: (1) you can create a symlink pointing to a path that doesn't exist yet — it becomes valid when the target is created; (2) if the target is deleted, the symlink becomes dangling, and that's detectable by resolution failure; (3) you can have symlinks that cross directory boundaries freely. A direct object reference would break all three of these behaviours.
def resolve(self, fs: 'FileSystem' ) -> FileSystemNode | None: visited = set() path = self.target_path while True: node = fs.get_node(path) if node is None: return None # dangling if not isinstance(node, SymbolicLink): return node if node.node_id in visited: raise SymlinkCycleError() visited.add(node.node_id) path = node.target_path
Why does resolution need cycle detection? Without cycle detection, a symlink loop causes infinite recursion and a stack overflow. It is trivially easy to create: link_a → /b/link_b, link_b → /a/link_a. Without the visited set, resolving link_a calls link_b's resolve, which calls link_a's resolve, forever. The visited set of node_ids detects the cycle the moment we'd visit an already-seen node. Real Linux kernels have a maximum symlink resolution depth (40) for the same reason — they bound rather than detect.
Without cycle detection: A user creates two symlinks pointing to each other. Any operation that resolves them — get_size(), read() — crashes the interpreter with RecursionError. In production, this kills the entire file system service, not just the offending call.
def get_size(self) -> int: node = self.resolve(self._fs_ref) return node.get_size() if node else 0
Why return 0 for a dangling symlink instead of raising? This is a design decision with tradeoffs. Returning 0 means Directory.get_size() can sum all children including dangling symlinks without crashing. The directory's reported size is technically "wrong" (it excludes the dangling link's target), but the operation completes. Raising would mean any directory containing a dangling symlink can never report its size — that seems too harsh. The alternative is to log the dangling link as a warning and exclude it from the sum. In an interview, mention both options and state your choice explicitly.

Permission — Unix-Style Access Control

Permission — the 9-bit Unix model in Python
from enum import Flag, auto class PermBit(Flag): READ = auto() # 4 WRITE = auto() # 2 EXECUTE = auto() # 1 class Permission: def __init__(self, owner: PermBit, group: PermBit, others: PermBit, group_name: str): self.owner = owner self.group = group self.others = others self.group_name = group_name
Why Flag enum instead of three plain booleans per role? Three booleans per role (owner_can_read, owner_can_write, owner_can_execute) would give you 9 separate fields per permission object — verbose and error-prone. Flag enums support bitwise combination: PermBit.READ | PermBit.WRITE is a single value meaning "can read and write." You can check PermBit.READ in node.permissions.owner cleanly. This also makes it trivial to translate to/from Unix octal notation (755, 644) in an interview. Interviewers who know Unix are impressed when you reach for Flag.
def check(self, user: User, node: FileSystemNode, op: PermBit) -> None: if user.is_superuser: return if user.user_id == node.owner_id: bits = self.owner elif self.group_name in user.groups: bits = self.group else: bits = self.others if op not in bits: raise PermissionDeniedError( f"{user.username} cannot " f"{op.name} {node.name}" )
Why raise PermissionDeniedError instead of returning False? Every single file operation — read(), write(), add_child() — calls permission.check(). If check returns False, every caller must remember to handle it: if not perm.check(...): raise PermissionDeniedError(). With 10 operations, that's 10 places to forget. By raising inside check(), permission denial is never silently ignored — it propagates up automatically. The calling code needs zero boilerplate. One consistent failure mode across the entire system.
@classmethod def default(cls) -> 'Permission': # 644: owner rw, group r, others r return cls( owner=READ|WRITE, group=READ, others=READ, group_name="users" )
Why a factory classmethod for the default permission? Every new file needs a sensible default permission. Without this, every caller of File("name", user) must remember to also set permissions manually — and "remember" is the enemy of correct code. The default (644 in Unix notation: owner read/write, group read, others read) is a well-known safe starting point. By encoding it once as a factory method, you make the right thing the easy thing. Any change to the default (e.g., change to 640 for security) requires updating one place.

FileSystem — the Single Entry Point (Orchestrator)

FileSystem — path resolution, CRUD, search
class FileSystem: def __init__(self): superuser = User("root", is_superuser=True) self.root = Directory("/", superuser) self._users: dict[str, User] = { superuser.user_id: superuser }
Why does FileSystem own the root Directory? Path resolution always starts at root. If the root were stored elsewhere, every call to resolve_path("/a/b") would need to locate the root first — adding another level of indirection. FileSystem is the natural owner because it is the single entity with visibility into the whole tree. It's also the only entity that can re-start symlink resolution from the root when following a symlink that points to an absolute path.
def resolve_path(self, path: str ) -> FileSystemNode | None: if not path.startswith("/"): raise InvalidPathError(path) parts = [p for p in path.split("/") if p] node: FileSystemNode = self.root for part in parts: if not isinstance(node, Directory): return None # path through a file child = node._children.get(part) if child is None: return None if isinstance(child, SymbolicLink): child = child.resolve(self) if child is None: return None # dangling node = child return node
Why does resolve_path live on FileSystem, not Directory? Absolute path resolution requires the root. When you traverse "/a/symlink/b", and symlink points to "/x/y", the resolution must restart from root ("/") to evaluate "/x/y". A Directory.resolve() method would need a reference back to the root — which means Directory holds a reference to FileSystem — a circular dependency. FileSystem owns the root, so FileSystem resolves paths. Clean separation, no cycles.
If Directory owned resolution: When a symlink is followed, Directory would need self.filesystem.root to restart. Now Directory depends on FileSystem and FileSystem depends on Directory — a circular import in Python. The program won't even start.
def create_file(self, user, parent_path: str, name: str) -> File: parent = self._get_dir(parent_path) new_file = File(name, user) parent.add_child(user, new_file) return new_file
Why resolve parent before creating the File object? Same principle as place_order()'s two-phase design: validate before mutating. If parent_path doesn't exist, raise immediately — don't create a File object that will never be inserted anywhere. Creating the file first and then failing on parent lookup leaves an orphan File object that will be garbage collected but adds confusion. Check everything first; create only if the path is valid. Guard ordering matters.
def delete(self, user, path: str, recursive: bool = False): parts = path.rsplit("/", 1) parent = self._get_dir(parts[0] or "/") name = parts[1] child = parent.get_child(name) if isinstance(child, Directory) \ and child._children \ and not recursive: raise DirectoryNotEmptyError(name) if recursive: self._delete_recursive(user, child) parent.remove_child(user, name)
Why is recursive=False the default? Recursive deletion is irreversible and potentially catastrophic. By defaulting to False, any call to delete() on a non-empty directory fails loudly. The developer must explicitly opt in to recursive=True, making the destructive intent unambiguous in the code. The Unix principle: the safe operation is the default. The dangerous operation requires an explicit flag. This is also why rm -rf requires both -r and -f flags — two explicit opt-ins for the most dangerous operation.
Step 4 · ~10 minutes
Key Flows — Traced End to End
Interviewers judge you on whether you can trace a request end-to-end. Not just "I have a method for that" — but which entity acts first, what it validates, what it passes on, and what the final state of the system is. Walk through these flows during the interview.

Flow 1 — Creating a File at a Nested Path

The request: fs.create_file(alice, "/home/alice", "report.txt").

Step-by-step: create_file() internals
Step 1: Resolve parent path parent = fs.resolve_path("/home/alice")
What happens during resolve_path? Starting from fs.root, we look up "home" in root's children — find a Directory. Then look up "alice" in that directory's children — find another Directory. Return it. If either lookup returns None, raise FileNotFoundError immediately. The path is validated before any new object is created.
Step 2: Check parent is a Directory if not isinstance(parent, Directory): raise NotADirectoryError()
Why check this explicitly? resolve_path returns a FileSystemNode. If the resolved path is a file named "alice" (not a directory), trying to add a child to it would fail confusingly at add_child(). Checking explicitly here produces a clear error: "Cannot create file inside a File." The check also prevents SymbolicLink nodes from being treated as directories — a symlink to a file is not a directory, even if it resolves to one through indirection (the resolution already handled that).
Step 3: Create File object new_file = File("report.txt", alice)
Why create the File before calling add_child? The file object needs to exist before it can be added anywhere. But critically: the File is created only after the parent path has been validated. If path resolution failed, we never reach this line. The object is created knowing it will be inserted successfully. This is the "read-only phase then mutation phase" principle — we've already confirmed the parent exists. Now we create the child.
Step 4: Add to parent directory parent.add_child(alice, new_file) # Checks: write permission on parent # Checks: no duplicate name # Inserts into parent._children
Why does add_child do its own permission check rather than trusting FileSystem? Design by contract. add_child() must enforce its own preconditions regardless of what the caller did. Today the only caller is FileSystem.create_file(). Tomorrow someone might call parent.add_child() directly from a test, a script, or a new feature — bypassing the FileSystem entirely. The permission check inside add_child() ensures the rule is always enforced, no matter who calls it. Rules about an entity's own state must live on the entity.

Flow 2 — Recursive get_size() on a Directory Tree

The Composite pattern in action. Call fs.root.get_size() on a tree: root → /home (dir) → /home/alice (dir) → report.txt (file, 1024 bytes), notes.txt (file, 512 bytes).

root.get_size() → home.get_size() → alice.get_size() → report: 1024 + notes: 512 = 1536

Each directory calls get_size() on its children and sums. Each file returns its own size. No isinstance check anywhere. No special case for "what if the child is a symlink?" — the SymbolicLink.get_size() handles that. The tree traversal is completely uniform. This is why the Composite pattern exists: it removes the type-checking branching from the traversal code and pushes it into each node's own class.

Flow 3 — Search by Extension

Request: fs.search(alice, "/home", extension=".txt"). This is a DFS tree traversal with a filter.

FileSystem.search() — DFS with filter
def search(self, user, path: str, name: str = None, extension: str = None ) -> list[FileSystemNode]: root_node = self.resolve_path(path) results = [] self._dfs_search( user, root_node, name, extension, results) return results
Why separate search() from _dfs_search()? The public search() method handles the entry contract: resolve the path, validate it, set up the results list. The private _dfs_search() handles the recursive traversal. This separation keeps the public API clean — callers don't need to provide a results list or worry about path resolution. It also makes _dfs_search() independently testable: you can call it on a constructed directory tree without going through path resolution. Single responsibility: one method validates inputs, one method does the traversal.
def _dfs_search(self, user, node, name, ext, results): if not isinstance(node, Directory): if self._matches(node, name, ext): results.append(node) return for child in node._children.values(): self._dfs_search( user, child, name, ext, results)
Why DFS and not BFS for search? Both DFS and BFS find all matching nodes. DFS has better memory characteristics for deep directory trees: its call stack grows as deep as the tree (typically 10–20 levels), while BFS's queue could hold all nodes at the widest level (potentially thousands of entries in a flat directory). For a file system where directories are often deeper than they are wide, DFS is more memory-efficient. However, if the tree could have cycles (which symlinks can create), BFS with a visited set is safer. In practice, you handle cycles at the symlink resolution level, so DFS is safe here.

Design Patterns — Justified, Not Just Named
Name a pattern only when you can complete this sentence: "Without this pattern, I would face this specific problem." Each pattern below includes the exact problem it solves, how it's applied in the file system, and a before/after showing the difference.
1. Composite Pattern applied in FileSystemNode / File / Directory

The specific problem without it: Every operation that traverses the file system tree — get_size(), search(), delete() — would need to check the type of every node: if isinstance(node, File): return node.size elif isinstance(node, Directory): sum children. As you add new node types (SymbolicLink, SpecialFile, MountPoint), every traversal method grows an extra branch. Adding a new node type means finding and updating every traversal in the codebase.

Without Composite
def get_total_size(node):
    if isinstance(node, File):
        return node.size
    elif isinstance(node, Directory):
        return sum(get_total_size(c)
                   for c in node.children)
    elif isinstance(node, SymbolicLink):
        target = node.resolve(fs)
        return get_total_size(target) if target else 0
    # Add SpecialFile → update this function
    # Add MountPoint → update this function
    # Every new type = update every traversal
The function must know about every node type. Adding a new type requires updating every function that traverses the tree. The traversal logic and the type logic are tangled together.
With Composite
class File(FileSystemNode):
    def get_size(self): return self.size

class Directory(FileSystemNode):
    def get_size(self):
        return sum(c.get_size()
                   for c in self._children.values())

class SymbolicLink(FileSystemNode):
    def get_size(self):
        node = self.resolve(self._fs)
        return node.get_size() if node else 0

# Adding SpecialFile:
# class SpecialFile(FileSystemNode):
#     def get_size(self): return 0
# Zero changes to existing code.
Each node type owns its size computation. Traversal code calls get_size() with no type checks. Adding a new node type requires only implementing the interface — zero changes to existing traversal code.
2. Facade Pattern applied in FileSystem

The specific problem without it: External code must know which entity owns which operation. "Create a file" — do I call Directory.add_child() directly? Do I construct a File object first? Do I call resolve_path() separately? Without a Facade, callers must understand the internal structure of the entire system — coupling them tightly to implementation details that may change.

How it's applied: FileSystem is the single public interface. fs.create_file(user, path, name) internally resolves the path, constructs the File, adds it to the parent directory, and returns the result — all in one call. Callers never touch Directory._children directly. If the internal implementation changes (e.g., switching to an inode model), only FileSystem changes. Callers are unaffected.

What to say in the interview: "FileSystem acts as a Facade — a single, clean entry point that hides the complexity of path resolution, object creation, and tree manipulation. External code only talks to FileSystem. Nothing inside the system is directly accessible from outside."
3. Template Method Pattern applied in FileSystemNode base operations

The specific problem without it: Every node type's delete() or move() method starts with the same steps: check permissions, acquire the lock, validate the operation is allowed. If you implement these in each subclass independently, you have three copies of the same boilerplate that must all be kept in sync. When the permission model changes, you update three places.

How it's applied: The base class FileSystemNode defines a template method _perform_write_op(user, op_fn) that handles the invariant steps — check permissions, acquire lock — and then calls the abstract op_fn() which each subclass supplies. The structure (check → lock → execute → release) is fixed in the base class. The specifics (what "execute" means) vary per subclass.

# In FileSystemNode (base class) def _perform_write_op(self, user, op_fn): perm.check(user, self, WRITE) # invariant step 1 with self._lock: # invariant step 2 op_fn() # varies per subclass self.modified_at = datetime.now() # invariant step 3 # In File (subclass) def write(self, user, data): self._perform_write_op( user, lambda: self._do_write(data) # the varying part only )
4. Strategy Pattern applied in FileSearcher

The specific problem without it: Search is currently hard-coded as DFS inside FileSystem._dfs_search(). If you need BFS (breadth-first search for files closest to the top of the tree), or parallel search across many subtrees, you modify FileSystem itself. Adding search strategies means growing a single method with if/else branches: if strategy == "dfs" ... elif strategy == "bfs" ...

How it's applied: Extract a SearchStrategy abstract class with a search(root, predicate) method. DFSSearchStrategy and BFSSearchStrategy implement it. FileSystem accepts a SearchStrategy in its constructor. Swapping strategies requires zero changes to FileSystem — inject a different strategy object.

class SearchStrategy(ABC): @abstractmethod def search(self, root: FileSystemNode, predicate) -> list[FileSystemNode]: ... class DFSSearchStrategy(SearchStrategy): def search(self, root, predicate): results = [] stack = [root] while stack: node = stack.pop() if predicate(node): results.append(node) if isinstance(node, Directory): stack.extend(node._children.values()) return results class FileSystem: def __init__(self, strategy: SearchStrategy = DFSSearchStrategy()): self._search_strategy = strategy ...
5. Singleton Pattern applied (carefully) to FileSystem

The specific problem it solves: There should be exactly one file system instance in the process. If two FileSystem objects are created, they have separate roots — no paths from one are visible in the other. Any operation that crosses systems (copying a file from one FS to another) must handle two different root trees. In practice, you want one authoritative root.

Why to use it with caution: Singleton is a controversial pattern precisely because it makes testing hard — you cannot inject a fresh, clean FileSystem for each test if the class enforces single-instance semantics at the class level. The modern alternative is Dependency Injection: make FileSystem a regular class, but create only one instance at startup and inject it everywhere. This gives singleton-like behaviour at runtime while keeping the class testable in isolation.

What to say in the interview: "I'd use a single shared FileSystem instance rather than enforcing Singleton at the class level — the same effect, but easier to test because I can create a fresh instance for each test without fighting the class design."

SOLID Principles — Applied to This Design
Don't recite SOLID definitions. Show the interviewer where each principle appears in this specific design, what problem it solves, and what would go wrong if you violated it. One concrete example per principle beats a textbook definition every time.
S
Single Responsibility Principle
"A class should have only one reason to change."

Where it appears: Permission is a separate class. It only changes if permission rules change. File only changes if file content semantics change. FileSystem only changes if the public API changes.

Violation: Embedding permission checking inside File.write() directly. Now File changes for two reasons: content semantics changes, or permission rules change. The same class is touched by the filesystem team and the security team.
Fix: Permission class owns all permission logic. File.write() calls perm.check(). The two concerns evolve independently.
O
Open/Closed Principle
"Open for extension, closed for modification."

Where it appears: Adding a new node type (SpecialFile, MountPoint) requires only implementing FileSystemNode — no changes to existing traversal code. Adding a new search strategy requires only a new SearchStrategy subclass.

Violation: Hardcoding if isinstance(node, File) ... elif isinstance(node, Directory) in traversal methods. Every new node type requires modifying existing methods.
Fix: The Composite abstract interface. Each node type handles its own behaviour. Traversal code is closed for modification when new types are added.
L
Liskov Substitution Principle
"Subtypes must be substitutable for their base type."

Where it appears: Any code that holds a FileSystemNode reference can call get_size() and get a valid integer — regardless of whether the node is a File, Directory, or SymbolicLink. No caller needs to know the concrete type.

Violation: A SymbolicLink.get_size() that raises NotImplementedError. Code that works fine on File and Directory now crashes on SymbolicLink. The subtype is not substitutable — it breaks callers that trusted the interface.
Fix: Every subclass provides a meaningful implementation of get_size(). SymbolicLink resolves its target and returns 0 for dangling links — never raises from a method the base class declares as returnable.
I
Interface Segregation Principle
"Clients should not depend on interfaces they don't use."

Where it appears: File has read() and write(). Directory has add_child() and list_children(). These are not on the base class. Code that handles generic nodes only calls get_size() — which all nodes share. Code that needs to write only holds a File reference.

Violation: Putting read(), write(), add_child(), and list_children() all on FileSystemNode. Now every Directory must implement read() and write() (which make no sense for a directory). Every File must implement add_child() (which makes no sense for a file). Forced to provide meaningless implementations everywhere.
D
Dependency Inversion Principle
"High-level modules should not depend on low-level modules. Both should depend on abstractions."

Where it appears: FileSystem.search() depends on the SearchStrategy abstract class, not on DFSSearchStrategy concretely. If you need BFS, inject BFSSearchStrategy() — FileSystem never changes. Similarly, SymbolicLink.resolve() accepts a FileSystem parameter rather than holding a direct reference — it depends on the interface, not the concrete state.

Violation: FileSystem directly instantiates DFSSearchStrategy inside its methods: strategy = DFSSearchStrategy(); strategy.search(...). You cannot swap strategies without modifying FileSystem. Unit testing with a mock strategy is impossible.
Fix: Inject SearchStrategy into FileSystem.__init__. In production, pass DFSSearchStrategy(). In tests, pass a MockSearchStrategy() that returns predetermined results without traversing any actual tree. Testable, flexible, zero changes to FileSystem when the strategy changes.
Step 7 · Deep Dive
Trade-offs — Show You Can Think Beyond the Obvious
The best LLD candidates don't just design — they acknowledge what their design sacrifices and when a different design would be the right choice. Raising trade-offs unprompted signals engineering maturity. These are the ones most likely to come up in a file system interview.
In-Memory Content vs. Metadata-Only Design
The question: Should File store actual content as a string in self._content, or should it only hold metadata (size, timestamps) with content stored elsewhere?
In-Memory Content ✓Simple to implement. read() and write() are trivial. Perfect for an LLD interview scope. get_size() is always correct.
LimitationEntire file content lives in RAM. A 4GB file object crashes the process. Real file systems never store content in objects — they use block pointers to disk sectors. This design doesn't scale beyond toy usage.

In an interview, say: "I'm using in-memory strings for simplicity. In a real implementation, File would hold an inode reference with block pointers to disk sectors. The class interface — read(), write() — stays the same; only the storage layer changes. This is the Dependency Inversion principle: File depends on a storage abstraction, not on in-memory strings directly."

Per-Node Locks vs. Global Lock
The question: Should every node have its own lock (per-node locking), or should the entire file system use one global lock?
Per-Node Locks ✓Maximum concurrency. Multiple threads can read different files simultaneously. Only writers on the same node block each other. Scales well with many concurrent users.
LimitationComplex. Moving a file (locking both source directory and destination directory simultaneously) risks deadlock if two threads move files in opposite directions. Must establish a lock-ordering protocol to prevent deadlocks.
Global Lock (simple)Zero deadlock risk. Simple to implement. Every operation just acquires one lock. Easy to reason about correctness.
LimitationAll operations serialize — one thread runs at a time. A long get_size() on a large directory tree blocks all concurrent reads. Poor throughput under load.

In an interview: "Per-node locking is the right answer for production, but requires careful lock ordering to prevent deadlocks. For a two-directory move() operation, always acquire locks in a consistent order (e.g. lower node_id first) to avoid the classic ABBA deadlock where thread 1 holds A and waits for B while thread 2 holds B and waits for A."

Recursive vs. Iterative Directory Traversal
The question: get_size() and search() use recursive calls. Is this a problem?
Recursive (elegant)Clean, readable code. Exactly mirrors the recursive structure of the tree. Easy to reason about correctness. Default Python call stack of 1000 is deep enough for almost all real directory structures.
LimitationA malicious or corrupt file system with 1500-deep nesting causes a RecursionError in Python. In production, paths like /a/b/c/d/... nested 10,000 deep are theoretically possible. Default stack limit is not guaranteed.

Defensive fix: convert recursion to an explicit stack (iterative DFS). Replace the call stack with a Python list used as a stack — identical logic, but the "stack" lives on the heap, not the call stack. No recursion limit. Say this in the interview if the interviewer pushes on "what happens with very deep trees?"

# Iterative DFS — avoids RecursionError on deep trees def get_size_iterative(root: Directory) -> int: total = 0 stack = [root] while stack: node = stack.pop() if isinstance(node, File): total += node.size elif isinstance(node, Directory): stack.extend(node._children.values()) return total
Absolute Paths Only vs. Supporting Relative Paths
The question: Should the file system only accept absolute paths (starting with "/") or also relative paths (relative to a "current working directory")?
Absolute Paths Only ✓Simple to implement. Resolution always starts at root — no ambiguity. resolve_path() is deterministic regardless of who calls it.
LimitationVerbose for users who are already deep in a directory. Real shells default to relative paths for usability. Supporting relative paths requires each user session to track a "current working directory" (cwd) — adding session state to the design.

Extension: add a Session class with a cwd: str field. resolve_path() takes an optional session parameter. If path doesn't start with "/", prepend the session's cwd. FileSystem itself stays stateless — all session state lives in Session. Clean separation.

Step 8 · Deep Dive
Interview Q&A
These are the follow-up questions interviewers ask after you've laid out the design. The best answers are specific, acknowledge trade-offs, and show that you've thought beyond the obvious solution.
Walk me through what happens when I call fs.delete(user, "/home/alice/report.txt").

1. Path split: Split "/home/alice/report.txt" into parent path "/home/alice" and name "report.txt".

2. Resolve parent: Call resolve_path("/home/alice") → traverses root → home → alice. Returns the alice Directory.

3. Validate parent: Confirm it's a Directory (not a File or SymbolicLink). If it's a File, raise NotADirectoryError.

4. Look up the target: alice._children.get("report.txt"). If None, raise FileNotFoundError.

5. Empty check: The target is a File, not a Directory, so the "non-empty directory" check doesn't apply.

6. Remove: alice.remove_child(user, "report.txt") — checks write permission on alice (not on the file), acquires alice's lock, deletes the entry from _children.

Key point for the interviewer: Deletion requires write permission on the parent directory, not on the file itself. This is how Unix works: "write" on a directory means "modify the directory's contents." You can delete a file you don't own if you own (or have write access to) the directory containing it. This is why Unix has the "sticky bit" — to prevent other users from deleting your files from a shared directory.

How would you handle a move operation (rename/move a file to a different directory)?

A move is: remove from source directory, add to destination directory — while holding locks on both simultaneously to prevent a race condition where another thread sees neither location between the two operations.

The naive approach: source_dir.remove_child(user, name) then dest_dir.add_child(user, node). Between these two calls, the file exists nowhere in the tree. Any concurrent search will miss it entirely.

The correct approach: Acquire both locks together before mutating either. But this creates a deadlock risk: Thread A wants to move file from dir1 to dir2 (locks dir1 first). Thread B wants to move file from dir2 to dir1 (locks dir2 first). Both threads are now waiting for each other forever.

Solution — lock ordering: Always acquire directory locks in a consistent order based on their node_id (alphabetically or numerically). Both Thread A and Thread B will attempt to acquire the lock with the smaller node_id first. One of them acquires it, proceeds, and releases. The other waits and then acquires both. No deadlock possible.

def move(self, user, src_path, dst_dir_path, new_name): src_parent_path, src_name = src_path.rsplit("/", 1) src_dir = self._get_dir(src_parent_path) dst_dir = self._get_dir(dst_dir_path) # Acquire locks in consistent order to prevent deadlock first, second = sorted( [src_dir, dst_dir], key=lambda d: d.node_id) with first._lock: with second._lock: node = src_dir._children.pop(src_name) node.name = new_name or src_name dst_dir._children[node.name] = node
How would you add a "file watcher" feature — notify interested parties when a file changes?

This is the Observer pattern. The interviewer is testing whether you can extend your design without modifying existing classes.

The design: Add a FileChangeObserver abstract class with an on_change(file, event_type) method. File maintains a list of registered observers. write() notifies all observers after the write completes. FileSystem.watch(user, path, observer) registers an observer on the target node.

What existing code changes: File.write() gets two extra lines: iterate observers, call on_change. That's it. The observer list on File starts empty — all existing code that never registers an observer is completely unaffected. This is the Observer pattern's core promise: adding notification capability doesn't break anything that existed before.

Extension examples: A LoggingObserver appends to an audit log. A CacheInvalidationObserver clears any cached read of this file. A SyncObserver pushes the changed content to a remote replica. All added without touching File's core logic. New observers are new classes — zero modifications to existing code. Open/Closed Principle in action.

Why is get_size() on Directory not cached? Wouldn't a large tree be slow to recompute every time?

Good question — you're right that recomputing recursively every time is O(n) in the number of nodes. Let's think through caching.

The problem with caching size: A cache is a stored derived value. It's correct only if nothing in the subtree has changed since the last computation. In our design, File.write() updates file.size, but it doesn't notify parent directories. So if you cache directory.cached_size = 1024 and then a file inside is written with new content, the cache is stale. To fix this, every write must propagate size updates up the tree — adding O(depth) work to every write operation.

The right trade-off: Depends on read/write ratio. If get_size() is called far more often than files are written to, cache with invalidation on write (propagate up). If writes are frequent and get_size() is rare, compute lazily. For an interview, say: "I'd add lazy caching with a _size_cache: int | None = None field on Directory. File.write() calls a _invalidate_parent_cache() method. Each Directory recomputes and re-caches on the next get_size() call after invalidation." This trades write overhead for read performance and is the approach real file systems take for directory size accounting.

How would you support disk persistence — surviving a process restart?

This is the domain of real file systems (ext4, NTFS, APFS) and is out of scope for LLD — but knowing the concepts shows depth.

The core concept — inodes: Real file systems separate the tree structure (directories, names, paths) from the data (file content, metadata). An inode is a fixed-size metadata record stored at a known disk offset. Every file and directory has an inode. A directory is a special file whose content is a list of (name → inode number) mappings. File content is stored in data blocks pointed to by the inode.

What changes in our design: Instead of File._content: str, a File would hold an inode_number: int. Content would be fetched via block_device.read_blocks(inode.block_pointers). The FileSystem would have a InodeTable and a BlockAllocator. On restart, the entire tree is reconstructed by reading the root inode and recursively following directory inodes.

Key insight for the interviewer: The public interface — File.read(), File.write(), Directory.get_size() — stays completely the same. Only the storage layer behind those methods changes. This is the Dependency Inversion principle: the high-level logic (what a file system does) is decoupled from the storage mechanism (how it persists). Our LLD design is already structured to support this swap without changing a single interface.

How do you test this design in isolation?

Testability follows directly from Dependency Inversion. Here's what a meaningful test suite looks like:

def test_write_updates_size(): user = User("alice") f = File("test.txt", user) f.write(user, "hello world") assert f.size == 11 assert f.read(user) == "hello world" def test_directory_size_recursive(): user = User("alice") root = Directory("root", user) subdir = Directory("sub", user) f1 = File("a.txt", user); f1.write(user, "hello") # 5 bytes f2 = File("b.txt", user); f2.write(user, "hi") # 2 bytes subdir.add_child(user, f2) root.add_child(user, f1) root.add_child(user, subdir) assert root.get_size() == 7 # 5 + 2 def test_permission_denied_on_read(): owner = User("alice") other = User("bob") f = File("secret.txt", owner) f.write(owner, "classified") f.permissions = Permission( owner=READ|WRITE, group=PermBit(0), others=PermBit(0), group_name="admin") with pytest.raises(PermissionDeniedError): f.read(other) # bob should not be able to read

Each test: runs in milliseconds, no disk I/O, no real threads. All because each entity is independently constructible with simple arguments. This is only possible because we kept concerns separated — File doesn't reach into FileSystem, Permission is injectable, and the lock in each node doesn't affect single-threaded tests.
