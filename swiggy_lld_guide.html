Step 1 · 5 min
Requirements & Clarifications
Before writing a single line of code, nail down exactly what you're building. The interviewer gives you a vague prompt — your job is to turn it into a concrete spec.
The prompt: "Design a food ordering app like Swiggy. Users should be able to browse restaurants, place orders, and track delivery."

That's too vague. Ask questions structured around four areas: core operations, error handling, scope boundaries, and extensibility.

Clarifying Questions

You
"What are the core actions in this system? Is it just order placement, or does it include restaurant management and delivery tracking too?"
Interviewer
"Yes — users browse restaurants and menus, place orders, and get real-time status updates. Restaurants manage their own menus. Delivery agents pick up and drop off orders."
You
"Does a single order span multiple restaurants, or is it one restaurant per order?"
Interviewer
"One restaurant per order. Keep it simple."
You
"What order states exist? How does the order move through the system?"
Interviewer
"Think: PLACED → ACCEPTED → PREPARING → READY → PICKED_UP → DELIVERED. Also CANCELLED if the restaurant rejects or the user cancels before PREPARING."
You
"Who can cancel an order — only the user, or also the restaurant?"
Interviewer
"Both, but only before the order is in PREPARING state."
You
"Should we model payment processing, or just track payment state?"
Interviewer
"Just track it — PENDING, PAID, REFUNDED. Don't integrate with a real payment gateway."
You
"Do restaurants have operating hours? Should we reject orders outside those hours?"
Interviewer
"Good question — yes, restaurants have an open/closed status. Reject orders if the restaurant is closed."
You
"Is delivery agent assignment automatic or manual?"
Interviewer
"Automatic — the system assigns the nearest available agent when the order is READY."
You
"Do we need to handle concurrent orders? Like, two users ordering the last item at the same time?"
Interviewer
"Great question — menu items have inventory counts. Decrement on order, raise an error if stock hits zero."
Why ask about the cancellation window?

Because it reveals a state machine constraint. Once you know "cancel only before PREPARING," you immediately understand the state machine needs guards — not every transition is always valid. This shapes your entire Order class design.

Final Requirements

In scope

  • Users browse restaurants filtered by city/cuisine. Menus show items with price and description.
  • Users place an order from one restaurant. Order can contain multiple items with quantities.
  • Order state machine: PLACED → ACCEPTED → PREPARING → READY → PICKED_UP → DELIVERED (+ CANCELLED).
  • Restaurants can accept/reject orders and update status up to READY.
  • Cancellation allowed only before PREPARING. Both user and restaurant can cancel.
  • Delivery agent auto-assigned when order reaches READY. Agent updates to PICKED_UP and DELIVERED.
  • Menu items have inventory. Orders decrement stock; reject if insufficient.
  • Payment state tracked: PENDING → PAID → REFUNDED (on cancel).
  • Restaurants have an open/closed flag. Reject orders if closed.

Out of scope

  • Real payment gateway integration (Razorpay, Stripe).
  • Multi-restaurant orders / cart merging.
  • Surge pricing, promotions, or coupons.
  • GPS and route optimization for delivery.
  • User authentication and sessions.
  • Ratings, reviews, and recommendations.
  • UI / rendering layer.
Why call out "out of scope" explicitly?

Like in the Amazon Locker breakdown, calling out what you're not building signals maturity. It shows you considered GPS routing, coupon logic, and auth — and made a deliberate choice. Interviewers reward this. It mirrors a real-world project scoping call.

Core Entities & Relationships
Look at the requirements and spot the nouns. Not every noun deserves a class — some are just fields. The key question: does this thing have behavior or just data?

Why is OrderItem a separate entity?

This is the kind of insight interviewers love. If you store only MenuItem references in an order, and the restaurant later changes the price of a burger from ₹120 to ₹150, your order history shows the wrong amount. OrderItem snapshots the price at the time of ordering. This is called temporal data integrity — it appears constantly in real e-commerce systems.

Step 3 · 15 min
Class Design
Start top-down — the orchestrator first, then supporting classes. For each class ask: what does it need to remember (state) and what does it need to do (methods)?

SwiggySystem — the Orchestrator

Every well-designed LLD has one entry point. Like the Locker class in Amazon Locker, SwiggySystem owns all registries and coordinates high-level flows. It doesn't implement business logic — it delegates to the right entity.

class SwiggySystem: # Registries (like Locker's compartments + accessTokenMapping) restaurants: dict[str, Restaurant] # restaurant_id → Restaurant users: dict[str, User] # user_id → User orders: dict[str, Order] # order_id → Order agents: dict[str, DeliveryAgent] # agent_id → DeliveryAgent # Core operations def place_order(user_id, restaurant_id, items) → Order def cancel_order(order_id, cancelled_by) → bool def update_order_status(order_id, new_status, actor_id) → bool def search_restaurants(city, cuisine) → list[Restaurant] def assign_delivery_agent(order_id) → DeliveryAgent
Why not put all logic in SwiggySystem?

That creates a "God Class" — one class that knows everything. Instead, SwiggySystem only coordinates: finds the right restaurant, calls the right method. The actual business logic (is this transition valid? is stock available?) lives on the entity itself. This is the Single Responsibility Principle.

Order — the Heart of the System

from enum import Enum, auto class OrderStatus(Enum): PLACED = auto() ACCEPTED = auto() PREPARING = auto() READY = auto() PICKED_UP = auto() DELIVERED = auto() CANCELLED = auto() class Order: order_id: str user_id: str restaurant_id: str items: list[OrderItem] # snapshot of (item, qty, price_at_order) status: OrderStatus payment: Payment delivery_agent_id: str | None created_at: datetime total_amount: float # State machine — declarative transition rules VALID_TRANSITIONS = { OrderStatus.PLACED: [OrderStatus.ACCEPTED, OrderStatus.CANCELLED], OrderStatus.ACCEPTED: [OrderStatus.PREPARING, OrderStatus.CANCELLED], OrderStatus.PREPARING: [OrderStatus.READY], OrderStatus.READY: [OrderStatus.PICKED_UP], OrderStatus.PICKED_UP: [OrderStatus.DELIVERED], } def update_status(self, new_status: OrderStatus) → bool: if new_status not in self.VALID_TRANSITIONS.get(self.status, []): raise InvalidTransitionError( f"Cannot go from {self.status} to {new_status}") self.status = new_status return True def can_cancel(self) → bool: return self.status in [OrderStatus.PLACED, OrderStatus.ACCEPTED] def calculate_total(self) → float: return sum(item.price_at_order * item.quantity for item in self.items)
Why a VALID_TRANSITIONS dictionary?

This is the State Machine pattern. Instead of spaghetti if/else inside every method, the rules live in one declarative data structure. Adding a new state is a one-line change. The Elevator problem uses a similar pattern for its direction-based movement logic.

MenuItem — with Inventory

class MenuItem: item_id: str name: str price: float is_available: bool inventory_count: int # in prod, use DB row locking for concurrency def is_in_stock(self, quantity: int) → bool: return self.is_available and self.inventory_count >= quantity def deduct_stock(self, quantity: int) → None: if not self.is_in_stock(quantity): raise InsufficientStockError(self.name) self.inventory_count -= quantity def restore_stock(self, quantity: int) → None: # Called on order cancellation self.inventory_count += quantity

DeliveryAgent — with Availability

class AgentStatus(Enum): AVAILABLE = auto() ON_DELIVERY = auto() OFFLINE = auto() class DeliveryAgent: agent_id: str name: str status: AgentStatus current_order_id: str | None location: Address def is_available(self) → bool: return self.status == AgentStatus.AVAILABLE def assign_order(self, order_id: str) → None: self.current_order_id = order_id self.status = AgentStatus.ON_DELIVERY def complete_delivery(self) → None: self.current_order_id = None self.status = AgentStatus.AVAILABLE

OrderItem — Price Snapshot

@dataclass class OrderItem: item_id: str name: str quantity: int price_at_order: float # frozen at time of order — never changes @property def subtotal(self) → float: return self.quantity * self.price_at_order
Step 4 · 10 min
Key Flows
Walk through the two most important operations end-to-end. This is where your entity design gets stress-tested.

Flow 1 — Placing an Order

Key insight: We deduct stock before creating the Order object. If stock deduction fails, no Order is created — no orphan orders in a bad state. On cancellation, restore_stock() reverses it.

Flow 2 — Order State Machine

PLACED → ACCEPTED → PREPARING → READY → PICKED_UP → DELIVERED

↳ PLACED or ACCEPTED can also transition to CANCELLED — the cancel window closes once PREPARING begins.

Why reject invalid transitions explicitly?

Silent ignoring of bad input creates subtle bugs. If update_status(DELIVERED) is called on a PLACED order and you just return False, the caller doesn't know why it failed and may retry. Raising InvalidTransitionError forces the caller to handle the failure correctly. This mirrors how Connect Four rejects moves on a full column — not silently, but with a clear error.

Flow 3 — Agent Assignment

def assign_delivery_agent(self, order_id: str) → DeliveryAgent: order = self.orders[order_id] restaurant = self.restaurants[order.restaurant_id] available = [a for a in self.agents.values() if a.is_available()] if not available: raise NoAgentAvailableError() # Simple strategy: nearest to restaurant # In prod: inject an AgentAssignmentStrategy (Strategy pattern) nearest = min(available, key=lambda a: distance(a.location, restaurant.location)) nearest.assign_order(order_id) order.delivery_agent_id = nearest.agent_id return nearest
Step 5 · Deep Dive
Design Patterns Applied
Don't sprinkle patterns randomly. Each pattern should solve a specific problem. Here's why each one was chosen — not just what it is.
State Machine (State Pattern) used in Order

Problem it solves: Without it, every method on Order would have a tangle of if status == X and new_status == Y checks that are easy to get wrong and hard to extend.

How it's used: The VALID_TRANSITIONS dict in Order is a declarative state machine. update_status() consults it before allowing any transition. Adding a new state (like RETURNED) is a one-line dict entry.

Interview cue: "I'm using the State Machine pattern here. The transition rules are centralized, so invalid operations are caught immediately and the logic doesn't leak into other classes."

Observer Pattern for notifications

Problem it solves: When an order moves to ACCEPTED, you might want to notify the user by push, SMS, and in-app badge simultaneously. Hardcoding all three into update_status() violates the Open/Closed Principle.

How it's used: Order maintains a list of OrderObserver objects. When status changes, it calls on_status_change(order, new_status) on each. You can add a PushNotificationObserver or SMSObserver without touching Order.

class OrderObserver: def on_status_change(self, order: Order, new_status: OrderStatus): ... class PushNotificationObserver(OrderObserver): def on_status_change(self, order, new_status): send_push(order.user_id, f"Order {new_status.name}")
Strategy Pattern for agent assignment

Problem it solves: "Pick the nearest agent" is one possible algorithm. "Pick the agent with fewest deliveries today" is another. You don't want to rewrite SwiggySystem every time this changes.

How it's used: Define an AgentAssignmentStrategy interface with find_agent(agents, order). Inject it into SwiggySystem at construction. Swapping algorithms is a one-line change at the call site.

Factory Pattern for Order creation

Problem it solves: Creating an Order involves multiple steps — generate UUID, snapshot prices, deduct inventory, set timestamp, create Payment. Scattered logic means steps get skipped.

How it's used: OrderFactory.create(user, restaurant, cart) encapsulates all creation steps. Nothing can create an Order in a half-initialized state.

Repository Pattern for data access

Problem it solves: Right now SwiggySystem uses plain dicts. In production, you'd use a database. Without Repository, switching storage touches business logic.

How it's used: Define OrderRepository with get(id), save(order), find_by_user(user_id). SwiggySystem talks to the interface — swap the implementation for a DB-backed version at any time.

Step 6 · Deep Dive
SOLID Principles
Show you're not applying SOLID as a checklist — explain how each principle shaped a specific decision in this design.
S
Single Responsibility
Order handles state transitions. Payment handles money state. MenuItem handles inventory. None bleeds into the others. SwiggySystem coordinates but doesn't implement rules.
O
Open / Closed
Adding PushNotificationObserver doesn't change Order. Adding a new AgentAssignmentStrategy doesn't change SwiggySystem. Extend by adding new classes, not editing existing ones.
L
Liskov Substitution
Any AgentAssignmentStrategy subclass must work wherever the base is expected. NearestAgentStrategy and HighestRatedStrategy fulfill the same contract — substitution is safe.
I
Interface Segregation
A DeliveryAgent doesn't need to implement cancel_order(). Interfaces are narrow: OrderObserver has one method, not ten.
D
Dependency Inversion
SwiggySystem depends on AgentAssignmentStrategy (an abstraction), not NearestAgentStrategy (a concrete class). Order depends on OrderObserver (abstract), not PushNotificationObserver. High-level modules don't depend on low-level details.
Step 7 · Deep Dive
Trade-offs & Design Decisions
Every design decision is a trade-off. Naming them explicitly makes you look senior. Here are the key ones in Swiggy's LLD.

1. OrderItem vs direct MenuItem reference

✓ Chosen: OrderItem snapshot
Prices are frozen at order time. If the restaurant changes the burger price next week, old orders still show ₹120. Supports refunds and invoice generation correctly.
✗ Alternative: direct MenuItem ref
Simpler. But if you run order.items[0].price 3 months later and the price changed, your order history is wrong. Never do this in a commerce system.

2. State Machine in Order vs separate class

✓ Chosen: state machine inside Order
All order logic in one place. Easy to understand. The state and the transitions that govern it live together — high cohesion.
✗ Alternative: separate OrderStateMachine
More extensible if states get very complex. But over-engineering for an LLD interview unless the interviewer explicitly asks for it.

3. In-memory dict vs Repository

✓ For LLD interview: plain dict
Fast to code, easy to reason about. Works perfectly for interview scope. Mention Repository as the upgrade path and you've covered both worlds.
Production: Repository pattern
Required for persistence, transactions, and concurrent access. Swap the dict for a PostgreSQL-backed repo without changing business logic.

4. Inventory — check-then-deduct vs atomic lock

✓ For LLD: simple deduct_stock()
Simple, clear, testable. Covers the single-threaded case perfectly for an interview scope.
Production: DB-level locking
Two threads can both pass is_in_stock() and both deduct, going negative. Real systems use SELECT FOR UPDATE or optimistic locking. Mention this — it shows you know the gap.

5. Agent assignment — eager vs lazy

✓ Chosen: assign when READY
Only assign once food is ready — reduces idle agent time. Agent doesn't wait at the restaurant during prep.
Alternative: assign when ACCEPTED
Agent can head to restaurant earlier. But agent might wait idle if prep takes long. Real Swiggy uses dynamic re-assignment based on ETA predictions.
Step 8 · Interview Prep
Interview Q&A
Common follow-up questions interviewers ask. Click each to reveal the answer.
How would you handle concurrent orders for the same item?
In the current design, MenuItem.deduct_stock() is not thread-safe — two threads could both pass the is_in_stock() check before either deducts. In production, use database-level row locking: SELECT FOR UPDATE in PostgreSQL, or optimistic locking with a version counter. The LLD interview answer: mention this gap explicitly and propose the DB-level fix. Don't pretend the dict solution is production-safe.
Why did you separate Payment from Order?
Single Responsibility. Order manages food lifecycle (state machine, items, agent). Payment manages money lifecycle (PENDING → PAID → REFUNDED). They change for different reasons — a new payment gateway integration shouldn't touch Order logic. Also, payment state can outlive the order: REFUNDED happens after DELIVERED or CANCELLED. Keeping them separate avoids conflicts.
How would you add coupon / discount support?
Add a Coupon entity with a code, discount_type (flat/percent), discount_value, and is_valid() method. In Order.calculate_total(), accept an optional coupon and apply it. SwiggySystem.place_order() accepts a coupon code, validates it, and passes it to OrderFactory. This doesn't require changing any existing class — just extending — which respects the Open/Closed Principle.
What if the restaurant rejects the order?
The order moves from PLACED → CANCELLED via update_status(OrderStatus.CANCELLED) — already a valid transition in the state machine. On cancellation: (1) restore inventory via item.restore_stock(qty), (2) trigger refund by setting payment to REFUNDED, (3) notify user via Observer. All three steps happen in SwiggySystem.cancel_order() which orchestrates them.
How would you support multiple addresses per user?
Change User.address from a single Address to a list[Address] with a default_address_id. Add add_address(), remove_address(), and set_default() methods. place_order() takes an optional delivery_address_id, defaulting to the user's default address. No other class needs to change.
How would you track order history per user?
Two options. Simple: User has a list[str] of order_ids. Fetch Order objects from SwiggySystem.orders when needed. Better with Repository: OrderRepository.find_by_user(user_id) queries the database directly. For an LLD interview the simple option is fine — mention the Repository approach as a production upgrade to impress the interviewer.
What design pattern did you use for notifications, and why not just call them directly?
Observer pattern. If you call notification methods directly inside Order.update_status(), Order becomes tightly coupled to notification logic. Adding email notifications means editing Order. Removing SMS means editing Order. With Observer, Order doesn't know who's listening — it just broadcasts. Each notification channel is an independent Observer. You can add, remove, or mock them without touching Order at all.
Walk me through what happens when an order is cancelled mid-preparation.
If status is PREPARING, order.can_cancel() returns False because PREPARING is not in the allowed-cancel states list. SwiggySystem.cancel_order() checks this first and raises a CancellationNotAllowedError. This is by design — once the restaurant is actively making the food, cancellation isn't fair to them. If asked "what if the user really wants to cancel?" — that would need a separate manual escalation flow, which is out of scope for this design.
