Step 1 · Requirements
Requirements & Clarifications
A notification system sounds simple until you realize it touches user preferences, templates, multiple channels, retries, failures, provider APIs, rate limits, idempotency, and async processing. The design only becomes clean after you separate what message should be sent from how it is delivered.
The prompt you'll receive
"Design a notification system. The system should send notifications to users through Email, SMS, Push, and WhatsApp."

This prompt hides almost all important decisions. Does one notification go through one channel or many? Are notifications sent immediately? Do users have channel preferences? Are templates static or dynamic? Do we retry failures? Does the caller need to know delivery status? Every answer changes the class diagram.

Clarifying Questions — with deep reasoning

You
"What types of notifications are we supporting? Are OTPs, marketing messages, order updates, and reminders all in scope?"
Why ask this first: Notification type drives priority, retry policy, rate limits, and user preference rules. OTPs are high priority and should ignore most marketing preferences. Promotional messages are low priority and must obey opt-outs. If you treat all notifications equally, you create a design where an OTP and a discount coupon compete in the same queue with the same retry behavior — operationally terrible.
Interviewer
"Support transactional notifications like OTPs, order updates, payment alerts, and low-priority promotional notifications."
You
"Can a notification be sent through multiple channels, or exactly one?"
Why this changes the data model: If exactly one channel is allowed, Notification.channel can be a single enum. If multiple channels are allowed, you need NotificationRequest.channels: list[ChannelType] and separate delivery attempts per channel. This is not a small change. Email might succeed while SMS fails. The notification-level status and channel-level status are now different concepts.
Interviewer
"Multiple channels are allowed. A single event can send Email + Push, or SMS + WhatsApp, etc."
You
"Do users have notification preferences per channel and per category?"
Why this is not optional: User preferences are a business rule, not a UI detail. A user may allow payment alerts on SMS but disable marketing SMS. If you only store sms_enabled: bool, you cannot represent this. The right model is a preference matrix: channel × notification category. This keeps your system from spamming users and also creates a clean place to encode category-specific opt-out rules.
Interviewer
"Yes. Preferences are per category and per channel. Transactional messages can be mandatory."
You
"Should messages be template-based with variables, or raw text passed by the caller?"
Why templates deserve their own object: Raw strings seem simpler, but they scatter message content across callers. Tomorrow product wants to change "Your OTP is {otp}" to "Use {otp} to sign in." If content is hardcoded in 14 services, you change 14 places. Templates centralize message structure and let callers pass only variables. This is the Template Method idea applied to content.
Interviewer
"Use templates. Callers pass template_id and variables."
You
"Should sending be synchronous inside the API call, or asynchronous using a queue and workers?"
Why this separates LLD from HLD but still matters: In pure LLD, you can model dispatch synchronously. In real systems, provider APIs are slow and flaky. If your API waits for SMS and Email providers, user-facing latency explodes. The clean LLD answer is: NotificationService creates and queues a notification; NotificationWorker sends it. That preserves object design while acknowledging production reality.
Interviewer
"Model asynchronous dispatch, but keep infra abstract. Use an in-memory queue or interface."
You
"Do we need retries for failed sends? If yes, should retry behavior differ by notification type or channel?"
Why retries must not live inside channel classes: Retry policy changes for business reasons. OTPs may retry aggressively for 2 minutes. Marketing messages may not retry at all. WhatsApp may back off differently from Email. If retry logic is inside EmailChannel.send(), you mix transport logic with policy logic. A separate RetryPolicy keeps the design flexible.
Interviewer
"Yes. Retry failed attempts up to a policy-defined limit."
You
"Do we track delivery status per channel?"
Why notification status alone is insufficient: Suppose a notification is sent over Email and SMS. Email succeeds, SMS fails. Is the notification SENT or FAILED? Both are partly true. You need DeliveryAttempt or ChannelDelivery per channel so each channel has its own status, provider response, attempt count, and timestamps.
Interviewer
"Yes. Track status per channel. Overall notification status can be derived."
You
"Do we need rate limiting and deduplication?"
Why this shows production awareness: Without rate limiting, a bug can send 1000 SMS messages to one user. Without deduplication, a caller retrying the same request can send duplicate OTPs or duplicate payment alerts. In LLD, we can keep both as interfaces — RateLimiter and IdempotencyStore — without implementing distributed Redis logic.
Interviewer
"Include basic rate limiting and idempotency, but keep implementation simple."

Final Requirements

In scope

  • Support Email, SMS, Push, and WhatsApp channels.
  • Support notification categories: OTP, ORDER_UPDATE, PAYMENT_ALERT, PROMOTION.
  • Support multiple channels for one notification.
  • Render messages from templates with variables.
  • Check user preferences before sending, except mandatory transactional messages.
  • Track per-channel delivery attempts and statuses.
  • Retry failed attempts using a configurable retry policy.
  • Dispatch asynchronously through a queue abstraction.
  • Rate-limit notifications to avoid spam.
  • Use idempotency keys to avoid duplicate notifications.

Out of scope

  • Real integrations with Twilio, SES, FCM, or WhatsApp Business API.
  • Distributed queue implementation like Kafka/SQS/RabbitMQ internals.
  • Provider webhook verification and signature validation.
  • Multi-language template localization.
  • A/B testing of message content.
  • Complex campaign scheduling.
Why this scope is interview-strong

It avoids overbuilding provider integrations while still modeling the hard parts: channel abstraction, preferences, retries, statuses, idempotency, and async dispatch. In interviews, this is the sweet spot: practical enough to show real-world thinking, small enough to implement in one whiteboard session.

Step 2 · Entities
Core Entities & Relationships
The trick in Notification LLD is not inventing many classes. The trick is putting each responsibility in the right class. Template rendering, sending, retrying, preference checking, and rate limiting are different reasons to change — so they should not all live inside one mega NotificationService.

Noun-by-noun evaluation

NounDecisionDeep reasoning — including what breaks if you get it wrong
KeepNotificationSystem Orchestrator The single entry point for external callers. It validates request, checks idempotency, creates Notification, and enqueues it. Without this facade, callers must know about templates, users, preferences, queues, and dispatchers directly. That leaks internal design to every caller.
KeepNotificationRequest Input DTO Represents what the caller wants: user_id, template_id, variables, category, channels, idempotency_key. It should not track lifecycle. Mixing request data with runtime delivery status creates a confused object that is half command and half state record.
KeepNotification Lifecycle entity Represents the created notification after validation. It owns delivery attempts, category, status, timestamps, and rendered message. If you skip this class, there is no durable object to retry or audit. Logs are not enough because retries need structured state.
KeepDeliveryAttempt Full class / dataclass One notification can go to multiple channels, and each channel can have multiple attempts. Email attempt 1 can fail while Push attempt 1 succeeds. Without DeliveryAttempt, you either lose history or incorrectly collapse all channels into one status.
KeepUser Entity Holds contact endpoints: email address, phone number, device token, WhatsApp phone. The sender needs these destinations. If these are passed loosely in every request, every caller becomes responsible for correctness and privacy.
KeepUserPreference Policy data + behavior Determines whether a user allows a category over a channel. It is not just flags. It enforces a rule: mandatory transactional categories may bypass opt-out. Keeping this rule here avoids duplicating preference checks across all channels.
KeepMessageTemplate Value object Stores template_id, body, required variables, category, and supported channels. If message content is hardcoded in service code, product copy changes require code deploys. Templates separate business content from delivery logic.
KeepTemplateRenderer Service Turns a template + variables into final text. If rendering is inside channels, Email and SMS duplicate the same variable replacement logic. Rendering is channel-independent, so it deserves one reusable component.
KeepNotificationChannel Interface Defines send(). Email, SMS, Push, and WhatsApp vary in transport details but share the same contract. This is exactly Strategy pattern territory.
KeepNotificationDispatcher Coordinator Consumes queued notifications and coordinates rendering, preference checks, channel sending, retries, and status updates. Without it, NotificationSystem becomes a god class.
KeepRetryPolicy Policy Retry behavior depends on category and channel. OTP, payment alert, and promotion should not share identical retry behavior. Policy belongs in a dedicated object because it changes for operational reasons.
KeepRateLimiter Guard service Prevents spam and provider abuse. It is separate from sending because rate limiting is a pre-send decision; channels should not know business limits like "max 3 OTP SMS per minute."
KeepIdempotencyStore Guard service Prevents duplicate notifications when callers retry API requests. Without it, network retries can send multiple OTPs or payment alerts.
SkipProvider Internal detail For LLD, provider can be hidden inside concrete channel classes. Unless the interviewer asks for multi-provider fallback like Twilio → Exotel, making Provider a separate abstraction too early adds noise.
The class-vs-field test for this problem

Ask: Does this thing own a rule that can change independently? RetryPolicy owns retry rules. RateLimiter owns spam-control rules. UserPreference owns opt-in rules. These should be classes. ChannelType is just a label like EMAIL or SMS, so it should be an enum.

Step 3 · Class Design
Class Design — Every Line Justified
Start with enums and data objects, then interfaces, then orchestrators. The key is dependency direction: high-level workflow classes should depend on abstractions like NotificationChannel, not concrete classes like EmailChannel.

Enums — make invalid states unrepresentable

Enums
class ChannelType(Enum): EMAIL = auto() SMS = auto() PUSH = auto() WHATSAPP = auto()
Why Enum? Channels are a fixed vocabulary. A string allows "whatsap" typo. An enum fails immediately. It also lets dictionaries use ChannelType keys safely, e.g. channels[ChannelType.EMAIL].
class NotificationCategory(Enum): OTP = auto() ORDER_UPDATE = auto() PAYMENT_ALERT = auto() PROMOTION = auto()
Why category? Category drives policy. OTPs may bypass preferences and use high priority. Promotions must obey opt-out. Payment alerts may have stronger retry guarantees. Without category, every notification becomes indistinguishable.
class DeliveryStatus(Enum): CREATED = auto() QUEUED = auto() SENDING = auto() SENT = auto() FAILED = auto() RETRY_SCHEDULED = auto() SKIPPED = auto()
Why these states? They model operational truth. SKIPPED is especially important: not every unsent notification is a failure. A promotional SMS skipped due to user preference is expected behavior, not an error.

NotificationRequest — input command, not lifecycle state

NotificationRequest
@dataclass(frozen=True) class NotificationRequest: user_id: str template_id: str variables: dict[str, str] category: NotificationCategory channels: list[ChannelType] idempotency_key: str
Why frozen dataclass? A request is a command from a caller. Once received, it should not mutate. If the variables or channels change midway, debugging becomes a nightmare. frozen=True makes the command stable.
Without freezing: A caller passes a request object, another component mutates channels from SMS to EMAIL before enqueue. The audit trail says SMS was requested but Email was sent. Ghost bug.
idempotency_key: str
Why not rely on generated notification_id? Generated IDs are created after the request arrives, so they cannot identify duplicate caller retries. Idempotency key comes from caller context, like order_123_payment_alert_v1. Same operation, same key, same result.

Notification — lifecycle aggregate

Notification
class Notification: notification_id: str user_id: str template_id: str variables: dict[str, str] category: NotificationCategory created_at: datetime attempts: list[DeliveryAttempt]
Why this class exists? This is the durable lifecycle record. Requests are input. Notifications are stateful records. A notification can be queued, attempted, retried, partially delivered, skipped, or failed. Those facts must live somewhere queryable.
def overall_status(self) -> DeliveryStatus: if any(a.status == SENT for a in self.attempts): return SENT if all(a.status in [FAILED, SKIPPED] for a in self.attempts): return FAILED return QUEUED
Why derive overall status? Overall status is a summary of per-channel attempts. Storing it separately creates two sources of truth. If Email succeeds and SMS fails, the attempts already contain the truth. Derive the summary instead of storing another field that can become stale.

DeliveryAttempt — status per channel per try

DeliveryAttempt
@dataclass class DeliveryAttempt: attempt_id: str notification_id: str channel: ChannelType attempt_number: int status: DeliveryStatus provider_message_id: str | None failure_reason: str | None attempted_at: datetime | None
Why not just store status on Notification? Because each channel has its own fate. Push can succeed instantly, SMS can fail due to provider outage, Email can be retried. One status field cannot represent this without lying.
If Notification has only one status: Email SENT, SMS FAILED. If you mark notification SENT, SMS failure disappears. If you mark FAILED, Email success disappears.
provider_message_id: str | None
Why store provider ID? Provider IDs are needed to reconcile delivery receipts and support debugging. If Twilio says message abc123 failed, you need to map that callback to your attempt. Without this field, provider callbacks are orphaned.

MessageTemplate and TemplateRenderer

Templates
@dataclass(frozen=True) class MessageTemplate: template_id: str body: str required_variables: set[str] supported_channels: set[ChannelType]
Why required_variables? It lets the system fail before sending. If template body needs {otp} but caller forgot it, you should raise a validation error before enqueueing. Otherwise the user receives "Your OTP is {otp}" — embarrassing and dangerous.
class TemplateRenderer: def render(self, template, variables): missing = template.required_variables - variables.keys() if missing: raise MissingVariableError(missing) return template.body.format(**variables)
Why renderer instead of template.render()? Both are acceptable. A separate renderer is useful when rendering becomes richer: escaping, channel-specific formatting, localization, markdown-to-HTML, truncation for SMS. It keeps templates as data and rendering as behavior.

NotificationChannel — Strategy interface

Channel Interface and Concrete Channels
class NotificationChannel(ABC): @abstractmethod def send(self, message: str, user: User) -> SendResult: pass
Why return SendResult? A send call needs structured output: success/failure, provider_message_id, and error message. Returning bool loses information. Raising exceptions for provider failures can work, but then normal provider failures become control-flow exceptions. A SendResult is explicit and testable.
class EmailChannel(NotificationChannel): def send(self, message, user): if not user.email: return SendResult.failure("missing email") return SendResult.success("email-provider-id")
Why validate destination here? Only EmailChannel knows what email needs. SMS needs phone number. Push needs device token. Putting all destination validation in Dispatcher creates a bloated switch statement that knows channel-specific details. Channel-specific validation belongs inside the channel.

UserPreference — policy around consent

UserPreference
class UserPreference: preferences: dict[ NotificationCategory, set[ChannelType] ] def allows(self, category, channel): if category in MANDATORY_CATEGORIES: return True return channel in self.preferences.get(category, set())
Why method, not raw dict access? The rule is not "look up a value." The rule is "mandatory categories bypass preferences, optional categories obey preferences." That logic deserves a method. Callers should ask preference.allows(), not inspect internal dictionaries.

NotificationSystem — public API facade

NotificationSystem
class NotificationSystem: users: dict[str, User] templates: dict[str, MessageTemplate] notifications: dict[str, Notification] queue: NotificationQueue idempotency_store: IdempotencyStore
Why own these registries? The public API needs to validate users and templates before creating notifications. It also needs to deduplicate requests before creating a new record. These are system-level responsibilities, not channel responsibilities.
def create_notification(self, request): if idempotency_store.exists(request.idempotency_key): return idempotency_store.get(request.idempotency_key) self.validate(request) notification = Notification.from_request(request) self.notifications[notification.id] = notification self.queue.enqueue(notification.id) idempotency_store.save(request.idempotency_key, notification.id) return notification
Why create then enqueue? Workers need a durable notification ID to process. If you enqueue raw request data without storing the notification first, a worker crash can lose state. Create record first, then enqueue its ID. The queue contains references to durable work, not the work itself.
If you enqueue before storing: Worker wakes up, tries to fetch notification_id, but record does not exist yet. Race condition.

NotificationDispatcher — worker-side coordinator

NotificationDispatcher
class NotificationDispatcher: channels: dict[ChannelType, NotificationChannel] renderer: TemplateRenderer retry_policy: RetryPolicy rate_limiter: RateLimiter
Why dispatcher has dependencies? Dispatcher coordinates the workflow but should not implement every detail. It delegates rendering to renderer, sending to channels, retry decisions to RetryPolicy, and spam prevention to RateLimiter. This keeps it a coordinator, not a god object.
def dispatch(self, notification_id): notification = store.get(notification_id) user = users[notification.user_id] template = templates[notification.template_id] message = renderer.render(template, notification.variables) for channel_type in notification.requested_channels: self.dispatch_channel(notification, user, message, channel_type)
Why loop per channel? Each channel is independently skippable, sendable, retryable, and auditable. A loop over requested channels creates one delivery path per channel. This naturally supports Email + SMS without special-case code.
def dispatch_channel(...): if not user.preference.allows(category, channel): mark_skipped() return if not rate_limiter.allow(user.id, category, channel): mark_retry_scheduled() return result = channels[channel].send(message, user) update_attempt(result)
Why this ordering? Preference check comes before rate limit because skipped messages should not consume quota. Rate limit comes before send because provider calls cost money. Send happens only after policy gates pass. This ordering prevents waste and respects user preferences.

Step 4 · Flows
Key Flows — Every Step Justified
Flows reveal whether the class design actually works. In notification systems, the two most important flows are: create notification and dispatch notification. Creation is caller-facing. Dispatch is worker-facing.

Flow 1 — Create Notification

create_notification()
1. Caller sends NotificationRequest
Why request object?A request object keeps the method signature stable. Without it, create_notification(user_id, template_id, variables, category, channels, idempotency_key) becomes long and brittle. Add priority tomorrow and every caller signature changes.
2. Check idempotency key
Why first?If the request is duplicate, you should return the original notification immediately. Doing validations or enqueueing before this can create duplicate work.
3. Validate user_id and template_id
Why before creating Notification?A Notification should represent valid work. Creating records for invalid users/templates pollutes the store and creates dead queue items workers can never process.
4. Validate requested channels supported by template
Why template-channel compatibility?Some templates may be too long for SMS or contain deep links only useful for Push. Sending a channel-incompatible template causes bad user experience and provider failures.
5. Create Notification record
Why durable record before queue?Queues should reference durable work. If a worker receives an ID, it must be able to fetch the notification. Create first, enqueue second.
6. Enqueue notification_id
Why enqueue ID not full object?Passing full objects through a queue creates serialization/versioning problems. Passing ID lets the worker fetch latest state from the store.

Flow 2 — Dispatch Notification

QUEUED→ SENDING→ SENT or FAILED or RETRY_SCHEDULED or SKIPPED
dispatch()
1. Worker dequeues notification_id
Why worker?Provider APIs are slow. Worker processing prevents user-facing API latency from depending on SMS/Email provider latency.
2. Fetch Notification, User, Template
Why fetch fresh?User preferences or contact info may have changed between enqueue and dispatch. Fetching fresh state improves correctness. For certain compliance-heavy systems, you may instead snapshot preferences at creation time; that is a tradeoff.
3. Render template
Why before channel loop?If all channels share the same plain message, render once. If channels need custom formatting, renderer can expose render_for_channel(). Do not duplicate rendering inside every channel.
4. For each channel: check preference
Why per-channel preference?A user may allow Email promotions but block SMS promotions. Preference is channel-specific, so the check must happen inside the channel loop.
5. Check rate limit
Why after preference?Skipped messages should not consume rate limit quota. Otherwise a disabled promotional SMS could block a later important transactional SMS.
6. Create DeliveryAttempt
Why before sending?If the process crashes during provider call, you still have a record that an attempt began. This makes stuck/in-flight attempts observable.
7. Call channel.send()
Why channel abstraction?Dispatcher does not care whether Email uses SES, SMS uses Twilio, or Push uses FCM. It only cares about SendResult. This is dependency inversion.
8. Update attempt result
Why structured result?You want to store provider message ID on success and failure reason on failure. A boolean cannot carry that operational detail.
9. Retry if policy allows
Why policy-driven?Retry behavior changes by category/channel. The dispatcher asks policy instead of hardcoding numbers. This keeps retry decisions configurable and testable.

Flow 3 — Idempotent caller retry

Scenario

Payment service calls NotificationSystem to send "Payment successful." Network times out before it receives response. Payment service retries the same request.

Idempotency flow
request.idempotency_key = "payment_123_success_v1"
Why caller-provided?The caller knows the business operation. The notification system only sees generic requests. Business uniqueness comes from the caller.
if key exists: return existing_notification
Why return existing?Idempotency does not mean "ignore duplicate." It means "same request produces same result." Returning the existing notification lets caller continue normally.
Step 5 · Patterns
Design Patterns Used
Do not name-drop patterns. In interviews, explain the problem each pattern solves. A pattern is useful only if it removes a concrete pain.
Facade Pattern NotificationSystem

Problem: External callers should not know about templates, queues, idempotency stores, and dispatchers.

Solution: Expose one public API: create_notification(request).

Why it matters: Caller code remains simple and stable even if internals change from in-memory queue to Kafka later.

Strategy Pattern NotificationChannel

Problem: Email, SMS, Push, and WhatsApp have different sending logic.

Solution: Common NotificationChannel.send() interface with concrete strategies.

Why it matters: Add Slack/Discord/In-app notifications without modifying dispatcher logic.

Policy Object RetryPolicy / RateLimiter

Problem: Retry and rate-limit rules change often and differ by category/channel.

Solution: Put these rules in dedicated policy objects.

Why it matters: Changing "OTP SMS max attempts from 3 to 2" should not require editing SMS sending code.

Command Object NotificationRequest

Problem: Many input parameters make method signatures unstable.

Solution: Bundle request data into a frozen command object.

Why it matters: Adding priority, schedule_at, or tenant_id later changes one class, not every method signature.

State Machine DeliveryStatus

Problem: Delivery has meaningful lifecycle states.

Solution: Explicit statuses like QUEUED, SENDING, SENT, FAILED, RETRY_SCHEDULED, SKIPPED.

Why it matters: Debugging and retries need state. "Boolean sent" is too primitive.

Step 6 · SOLID
SOLID Principles Applied
Notification systems are a perfect SOLID question because bad designs naturally become giant if-else monsters. Good designs separate policy, transport, orchestration, and data.
S
Single Responsibility
One class should have one reason to change.
Violation: NotificationService renders templates, checks preferences, sends SMS, retries, and rate-limits.
Fix: Renderer renders. Channel sends. RetryPolicy retries. Dispatcher coordinates.
O
Open/Closed
Open for extension, closed for modification.
Violation: Adding Slack requires editing if channel == ... in Dispatcher.
Fix: Add SlackChannel implementing NotificationChannel.
L
Liskov Substitution
Subtypes should be usable wherever the base type is expected.
Violation: PushChannel requires a completely different method signature.
Fix: All channels implement send(message, user) -> SendResult.
I
Interface Segregation
Clients should not depend on methods they do not use.
Violation: Channel interface has send_email, send_sms, send_push.
Fix: One minimal method: send().
D
Dependency Inversion
High-level modules should depend on abstractions, not concrete classes.
Violation: Dispatcher creates EmailChannel() internally and directly knows every provider.
Fix: Dispatcher receives dict[ChannelType, NotificationChannel]. It depends on the interface.
Step 7 · Trade-offs
Important Trade-offs
Trade-offs are where interviewers see maturity. Do not just say what you chose. Say what you rejected and why.
Synchronous send vs asynchronous queue
Scenario: User signs up and needs an OTP.
Async queueBetter throughput, isolates provider latency, enables retries. API returns quickly after enqueueing.
Synchronous sendSimpler but user-facing request waits for provider. Provider slowness becomes app slowness.

Chosen: Async queue for production-like design. In small demos, synchronous can be acceptable, but mention the limitation.

Store rendered message vs render at send time
Scenario: Template changes after notification is queued but before worker sends it.
Store rendered messageGuarantees user receives the exact content produced at creation time. Better auditability.
Render at send timeUses latest template. Useful for corrections, but can make audit confusing.

Interview answer: For transactional notifications, store rendered message or template version. For promotions, render-late can be acceptable.

Overall status as field vs derived from attempts
DerivedNo duplicate source of truth. Correct summary from attempts.
StoredFaster queries but risks inconsistency unless transactionally updated.

Chosen: Derive in LLD. In production, you may store a denormalized summary for query performance, but update it carefully.

One DeliveryAttempt per channel vs per retry
Per retryFull history: attempt 1 failed timeout, attempt 2 succeeded.
Per channel onlySimpler but overwrites failure history.

Chosen: Per retry attempt because debugging provider failures requires history.

Preferences checked at creation vs dispatch
Creation timeStable decision. User preference changes after enqueue do not affect already-created notifications.
Dispatch timeMore respectful of latest user preference but can change behavior after creation.

Chosen: Dispatch-time for freshness in this guide. For strict compliance/audit, snapshot preference decision at creation.

Step 8 · Interview Q&A
Common Interview Questions
These are the questions interviewers use to test whether you understand the design or merely memorized the class diagram.
Why not just have one NotificationService with sendEmail(), sendSms(), sendPush()?
Because every new channel modifies the same class. That violates Open/Closed Principle and creates a god service. The better design is a NotificationChannel interface with concrete channel implementations. The dispatcher works with the abstraction, not individual methods.
Why separate NotificationRequest and Notification?
NotificationRequest is the caller's command. Notification is the system's lifecycle record. Requests should be immutable input. Notifications need runtime state like attempts, timestamps, and statuses. Mixing them creates a confused object.
How do you add Slack notifications?
Add ChannelType.SLACK, implement SlackChannel(NotificationChannel), register it in the channel map, and add user destination data if needed. Dispatcher logic should not change.
How do you prevent duplicate OTPs?
Use caller-provided idempotency keys. For example, authentication service sends login_otp_user123_1700000000. If the same request is retried due to timeout, NotificationSystem returns the existing notification instead of creating a new one.
How do you prioritize OTP over promotions?
Add priority to NotificationRequest or derive priority from category. Queue can become a priority queue. OTP and payment alerts get high priority; promotions get low priority. In LLD, model this with a priority field or method on category policy.
Where should retry logic live?
In RetryPolicy, not channels. Channels know how to send. RetryPolicy knows whether to retry. Dispatcher coordinates the two. This separation lets retry rules change without touching provider logic.
How do you handle user preferences?
Use UserPreference.allows(category, channel). Preferences should be category-channel specific. Mandatory transactional categories like OTP can bypass opt-out; promotional categories must obey opt-out.
How do you handle provider outage?
Channel returns failure result. Dispatcher records attempt failure. RetryPolicy decides whether to retry. For advanced design, introduce provider fallback inside a channel, e.g. SMSChannel tries Twilio then Exotel. Keep fallback hidden behind the same NotificationChannel interface.
How would you scale this system?
Use durable queues, multiple workers, idempotency store, rate limiter, provider-specific worker pools, retry queues with delay, and partitioning by user or tenant. But for LLD, keep these as interfaces so the class design does not depend on Kafka/SQS specifics.

Final interview summary

What to say in 60 seconds

I would model NotificationSystem as the facade that accepts a NotificationRequest, validates it, enforces idempotency, creates a Notification record, and enqueues it. A worker-side NotificationDispatcher processes queued notifications. It renders a MessageTemplate, checks UserPreference, checks RateLimiter, then sends through channel strategies like EmailChannel, SmsChannel, PushChannel, and WhatsAppChannel. DeliveryAttempt tracks per-channel status and retries. RetryPolicy owns retry rules so channel logic stays clean. This design separates orchestration, transport, policy, rendering, and lifecycle state.
