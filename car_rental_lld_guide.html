System Overview

Understand the requirements before jumping into code — this is what separates a great interview answer.

📋
Functional Requirements
What the system must do
  • Customer can search for available cars by type, location, and date range
  • Customer can reserve and rent a car for a specific period
  • System must track car availability in real time
  • Customer can return a car and receive a bill
  • System supports multiple pricing strategies (daily, hourly, premium)
  • System can apply add-ons: insurance, GPS, child seat
  • Notifications sent on booking confirmation, return, and overdue alerts
⚡
Non-Functional Requirements
Quality attributes
  • Thread-safe reservation (prevent double booking)
  • Extensible pricing (add new strategies easily)
  • Separation of concerns across modules
  • Easily testable with pytest
🧩
Core Entities
Domain objects
  • Car, CarType, CarStatus
  • Customer, DrivingLicense
  • Reservation, ReservationStatus
  • Invoice, Payment
  • Branch, RentalSystem
  • PricingStrategy, AddOn

Key Design Decisions

💡 Think out loud in interviews. Explain WHY you're making a decision, not just what the decision is.
  • Strategy Pattern for Pricing: Different rates (daily, hourly, weekend) can be swapped without touching the Car or Reservation class.
  • State Pattern for Car: A car transitions through AVAILABLE → RESERVED → RENTED → MAINTENANCE. Each state controls what operations are valid.
  • Decorator for Add-Ons: Base rental cost + GPS + Insurance stacks cleanly using the Decorator pattern instead of hardcoded boolean flags.
  • Singleton for RentalSystem: One central registry of cars and reservations across the whole application.
  • Observer for Notifications: Reservation changes trigger customer notifications without coupling those concerns together.
  • Factory for Car creation: Centralizes creation logic and ensures all cars are properly initialized.

Singleton — RentalSystem

The Singleton ensures only one instance of RentalSystem exists throughout the entire application. Think of it like the "headquarters" — there's only one, and everyone talks to the same one.

🤔 Why here? We need a single source of truth for all cars and reservations. If two instances existed, one wouldn't know about bookings made via the other — race conditions and double bookings would follow.
singleton.py Thread-safe Singleton
import threading
from typing import Dict, Optional

class RentalSystem:
    """Singleton: one global registry of cars and reservations."""

    _instance: Optional['RentalSystem'] = None
    _lock = threading.Lock()

    def __new__(cls) -> 'RentalSystem':
        # Double-checked locking: fast path (no lock) + safe path (with lock)
        if cls._instance is None:
            with cls._lock:            # acquire lock only when needed
                if cls._instance is None:
                    cls._instance = super().__new__(cls)
                    cls._instance._cars: Dict[str, Car] = {}
                    cls._instance._reservations: Dict[str, Reservation] = {}
                    cls._instance._branches: Dict[str, Branch] = {}
        return cls._instance

    def add_car(self, car: Car) -> None:
        self._cars[car.car_id] = car

    def get_car(self, car_id: str) -> Optional[Car]:
        return self._cars.get(car_id)

    def available_cars(self, car_type: CarType = None) -> list[Car]:
        cars = [c for c in self._cars.values()
                if c.status == CarStatus.AVAILABLE]
        if car_type:
            cars = [c for c in cars if c.car_type == car_type]
        return cars
⚠️ Interview watch-out: A naive Singleton without the double-checked locking pattern can cause two threads to both see _instance is None simultaneously and create two objects. Always mention thread safety.

Factory Method — CarFactory

The Factory centralizes car creation. Instead of scattering Car(...) calls everywhere with different defaults, one factory method produces correctly configured cars.

🤔 Why here? Each car type has different default rates and attributes. If we change Economy's default rate, we change it in one place — not 20 places scattered across the codebase.
factory.py Car Factory
class CarFactory:
    """Creates cars with type-appropriate defaults."""

    # Default daily rates per type
    _RATES = {
        CarType.ECONOMY : 35.0,
        CarType.COMPACT : 50.0,
        CarType.SUV     : 80.0,
        CarType.LUXURY  : 150.0,
        CarType.VAN     : 90.0,
    }

    @staticmethod
    def create_car(
        make: str,
        model: str,
        year: int,
        car_type: CarType,
        branch_id: str,
        custom_rate: float = None
    ) -> Car:
        rate = custom_rate or CarFactory._RATES[car_type]
        return Car(
            make=make,
            model=model,
            year=year,
            car_type=car_type,
            status=CarStatus.AVAILABLE,
            daily_rate=rate,
            branch_id=branch_id,
        )

# Usage
economy_car = CarFactory.create_car(
    "Toyota", "Yaris", 2022, CarType.ECONOMY, branch_id="branch-1"
)
luxury_car = CarFactory.create_car(
    "BMW", "7 Series", 2023, CarType.LUXURY, branch_id="branch-2"
)

Strategy — PricingStrategy

The Strategy pattern lets us define a family of algorithms (pricing rules), put each in its own class, and swap them interchangeably. The Reservation just asks "what does this cost?" — it doesn't care how the answer is computed.

🤔 Why here? Pricing rules change often — weekends get discounts, long-term gets bulk rates, premium members get special deals. Without Strategy, every change means modifying a giant if-elif tree inside Reservation.
pricing.py Strategy Pattern
from abc import ABC, abstractmethod
from datetime import datetime

class PricingStrategy(ABC):
    """Abstract base — all pricing strategies must implement this."""
    @abstractmethod
    def calculate(self, pickup: datetime, return_dt: datetime, daily_rate: float) -> float:
        ...

class DailyPricingStrategy(PricingStrategy):
    """Standard per-day pricing."""
    def calculate(self, pickup, return_dt, daily_rate) -> float:
        days = (return_dt - pickup).days or 1   # minimum 1 day
        return days * daily_rate

class HourlyPricingStrategy(PricingStrategy):
    """Per-hour pricing, capped at daily rate."""
    HOURLY_FRACTION = 0.07   # 7% of daily rate per hour
    def calculate(self, pickup, return_dt, daily_rate) -> float:
        hours = (return_dt - pickup).total_seconds() / 3600
        hourly = daily_rate * self.HOURLY_FRACTION
        return min(hours * hourly, daily_rate)  # cap at daily rate

class WeeklyDiscountStrategy(PricingStrategy):
    """7+ days gets 15% off."""
    DISCOUNT = 0.15
    def calculate(self, pickup, return_dt, daily_rate) -> float:
        days = (return_dt - pickup).days or 1
        base = days * daily_rate
        return base * (1 - self.DISCOUNT) if days >= 7 else base

# In Reservation:
class Reservation:
    def __init__(self, ..., strategy: PricingStrategy = None):
        self.strategy = strategy or DailyPricingStrategy()

    def calculate_cost(self, car: Car) -> float:
        return self.strategy.calculate(
            self.pickup_date, self.return_date, car.daily_rate
        )

State — CarState

The State pattern represents a car's lifecycle as explicit state objects. Each state knows which transitions are valid and raises an error if an illegal transition is attempted.

🤔 Why here? A car under maintenance cannot be rented. A car that's already rented cannot be reserved. Without State, we'd have messy if-statements scattered everywhere. State centralizes transition logic.
state.py State Pattern for Car lifecycle
from abc import ABC, abstractmethod

class CarState(ABC):
    @abstractmethod
    def reserve(self, car: 'CarContext') -> None: ...
    @abstractmethod
    def rent(self, car: 'CarContext') -> None: ...
    @abstractmethod
    def return_car(self, car: 'CarContext') -> None: ...
    @abstractmethod
    def send_to_maintenance(self, car: 'CarContext') -> None: ...

class AvailableState(CarState):
    def reserve(self, car): car.set_state(ReservedState())
    def rent(self, car): car.set_state(RentedState())
    def return_car(self, car): raise ValueError("Car is already available")
    def send_to_maintenance(self, car): car.set_state(MaintenanceState())

class ReservedState(CarState):
    def reserve(self, car): raise ValueError("Car already reserved")
    def rent(self, car): car.set_state(RentedState())    # confirm pickup
    def return_car(self, car): car.set_state(AvailableState())  # cancelled
    def send_to_maintenance(self, car): raise ValueError("Cancel reservation first")

class RentedState(CarState):
    def reserve(self, car): raise ValueError("Car is currently rented")
    def rent(self, car): raise ValueError("Car is already rented")
    def return_car(self, car): car.set_state(AvailableState())
    def send_to_maintenance(self, car): raise ValueError("Return the car first")

class MaintenanceState(CarState):
    def reserve(self, car): raise ValueError("Car is in maintenance")
    def rent(self, car): raise ValueError("Car is in maintenance")
    def return_car(self, car): car.set_state(AvailableState())  # service done
    def send_to_maintenance(self, car): raise ValueError("Already in maintenance")

class CarContext:
    """Wraps a Car and delegates state transitions."""
    def __init__(self, car: Car):
        self._car = car
        self._state: CarState = AvailableState()

    def set_state(self, state: CarState): self._state = state
    def reserve(self): self._state.reserve(self)
    def rent(self): self._state.rent(self)
    def return_car(self): self._state.return_car(self)

Observer — ReservationObserver

The Observer pattern lets multiple notification handlers (email, SMS, push) react to reservation events without the Reservation class knowing anything about them.

🤔 Why here? When a reservation is confirmed, cancelled, or overdue, we need to notify the customer. But the notification method (email vs SMS vs push) should be pluggable, not hardcoded into Reservation logic.
observer.py Observer Pattern
from abc import ABC, abstractmethod

class ReservationObserver(ABC):
    """All notifiers must implement this contract."""
    @abstractmethod
    def on_status_change(self, reservation: Reservation, new_status: ReservationStatus) -> None:
        ...

class EmailNotifier(ReservationObserver):
    def on_status_change(self, reservation, new_status):
        print(f"[EMAIL] Reservation {reservation.reservation_id[:8]}... → {new_status.value}")
        # In production: call email service API here

class SMSNotifier(ReservationObserver):
    def on_status_change(self, reservation, new_status):
        print(f"[SMS] Booking update: {new_status.value}")

class ReservationSubject:
    """Manages observers and fires notifications."""
    def __init__(self):
        self._observers: list[ReservationObserver] = []

    def attach(self, observer: ReservationObserver): self._observers.append(observer)
    def detach(self, observer: ReservationObserver): self._observers.remove(observer)

    def notify(self, reservation: Reservation, status: ReservationStatus):
        for obs in self._observers:
            obs.on_status_change(reservation, status)

# Usage in RentalService:
subject = ReservationSubject()
subject.attach(EmailNotifier())
subject.attach(SMSNotifier())
subject.notify(reservation, ReservationStatus.CONFIRMED)

Decorator — Rental Add-Ons

The Decorator pattern wraps a base cost function with additional cost layers. Adding GPS, insurance, and a child seat stacks cleanly — each decorator adds its own cost and forwards the rest to the inner object.

🤔 Why here? Instead of if has_gps: total += 10; if has_insurance: total += 25 (which gets messy with combinations), each add-on wraps the previous one. Easy to add new add-ons without touching existing code. Open/Closed Principle!
decorator.py Add-On Decorator Pattern
from abc import ABC, abstractmethod

class RentalCost(ABC):
    """Base interface for any rental cost object."""
    @abstractmethod
    def calculate(self) -> float: ...
    @abstractmethod
    def description(self) -> str: ...

class BaseRentalCost(RentalCost):
    """The core rental cost — days × daily rate."""
    def __init__(self, days: int, daily_rate: float):
        self._days = days
        self._rate = daily_rate

    def calculate(self) -> float: return self._days * self._rate
    def description(self) -> str: return f"{self._days} days @ ₹{self._rate}/day"

class AddOnDecorator(RentalCost, ABC):
    """Wraps a RentalCost and adds extra cost on top."""
    def __init__(self, wrapped: RentalCost):
        self._wrapped = wrapped

    @abstractmethod
    def add_on_cost(self) -> float: ...
    @abstractmethod
    def add_on_name(self) -> str: ...

    def calculate(self) -> float: return self._wrapped.calculate() + self.add_on_cost()
    def description(self) -> str: return self._wrapped.description() + f" + {self.add_on_name()}"

class InsuranceAddon(AddOnDecorator):
    def add_on_cost(self) -> float: return 20.0
    def add_on_name(self) -> str: return "Insurance (₹20)"

class GPSAddon(AddOnDecorator):
    def add_on_cost(self) -> float: return 10.0
    def add_on_name(self) -> str: return "GPS (₹10)"

class ChildSeatAddon(AddOnDecorator):
    def add_on_cost(self) -> float: return 8.0
    def add_on_name(self) -> str: return "Child Seat (₹8)"

# Stacking add-ons: base → insurance → GPS → child seat
cost = BaseRentalCost(days=3, daily_rate=50.0)   # ₹150
cost = InsuranceAddon(cost)                         # ₹170
cost = GPSAddon(cost)                               # ₹180
cost = ChildSeatAddon(cost)                         # ₹188
print(cost.calculate())     # 188.0
print(cost.description())   # 3 days @ ₹50/day + Insurance + GPS + Child Seat

Complete Rental Flow

End-to-end flow with thread safety and the RentalService orchestrator.

The 6 steps of a rental

1
Customer searches for available cars
Queries RentalSystem for cars matching CarType, branch, and date range. Returns list of Car objects with status AVAILABLE.
2
Customer selects a car and creates a Reservation
RentalService.reserve() checks customer license validity, acquires a per-car lock, and creates a Reservation object. Car status moves to RESERVED.
3
Notification sent — Reservation CONFIRMED
The ReservationSubject fires notify(), which calls all attached observers (EmailNotifier, SMSNotifier). Decoupled from core booking logic.
4
Customer picks up car — status becomes RENTED
RentalService.pickup() transitions the car via the State pattern. Reservation status moves to ACTIVE.
5
Customer returns car — cost calculated
Actual return time is recorded. Late fee is added if past due date. PricingStrategy calculates base cost. Decorator stacks add-on costs. Invoice is generated.
6
Car becomes AVAILABLE again
Car state transitions back to AvailableState. Reservation is marked COMPLETED. Observer fires final notification.
rental_service.py Orchestrator with thread safety
import threading
from datetime import datetime
from typing import Dict, List

class RentalService:
    """Orchestrates the full rental lifecycle.
    Uses per-car locks to prevent double booking.
    """
    def __init__(self):
        self._system = RentalSystem()
        self._car_locks: Dict[str, threading.Lock] = {}
        self._locks_mutex = threading.Lock()
        self._subject = ReservationSubject()

    def _get_car_lock(self, car_id: str) -> threading.Lock:
        # Create a lock for a car the first time we see it
        with self._locks_mutex:
            if car_id not in self._car_locks:
                self._car_locks[car_id] = threading.Lock()
        return self._car_locks[car_id]

    def reserve(
        self,
        customer: Customer,
        car_id: str,
        pickup_date: datetime,
        return_date: datetime,
        add_ons: List[AddOnType] = None,
        strategy: PricingStrategy = None,
    ) -> Reservation:

        # 1. Validate customer license
        if not customer.license or not customer.license.is_valid():
            raise ValueError("Invalid or expired driving license")

        # 2. Per-car lock — prevents double booking
        car_lock = self._get_car_lock(car_id)
        with car_lock:
            car = self._system.get_car(car_id)
            if not car:
                raise ValueError(f"Car {car_id} not found")
            if car.status != CarStatus.AVAILABLE:
                raise ValueError(f"Car is not available (status: {car.status.value})")

            # 3. Create reservation
            reservation = Reservation(
                customer_id=customer.customer_id,
                car_id=car_id,
                pickup_date=pickup_date,
                return_date=return_date,
                add_ons=add_ons or [],
                status=ReservationStatus.CONFIRMED,
            )

            # 4. Change car status atomically (inside lock)
            car.status = CarStatus.RESERVED
            customer.reservations.append(reservation.reservation_id)

        # 5. Notify observers (outside lock — no need to hold it)
        self._subject.notify(reservation, ReservationStatus.CONFIRMED)
        return reservation

    def pickup(self, reservation: Reservation) -> None:
        car = self._system.get_car(reservation.car_id)
        car.status = CarStatus.RENTED
        reservation.status = ReservationStatus.ACTIVE
        self._subject.notify(reservation, ReservationStatus.ACTIVE)

    def return_car(self, reservation: Reservation, actual_return: datetime) -> Invoice:
        car = self._system.get_car(reservation.car_id)
        reservation.actual_return = actual_return

        # Calculate base cost via Strategy
        strategy = DailyPricingStrategy()
        base_cost = strategy.calculate(reservation.pickup_date, actual_return, car.daily_rate)

        # Stack add-on costs via Decorator
        cost_obj = BaseRentalCost(
            days=(actual_return - reservation.pickup_date).days or 1,
            daily_rate=car.daily_rate
        )
        addon_cost = 0.0
        for addon in reservation.add_ons:
            if addon == AddOnType.INSURANCE:
                cost_obj = InsuranceAddon(cost_obj)
                addon_cost += 20.0
            elif addon == AddOnType.GPS:
                cost_obj = GPSAddon(cost_obj)
                addon_cost += 10.0
            elif addon == AddOnType.CHILD_SEAT:
                cost_obj = ChildSeatAddon(cost_obj)
                addon_cost += 8.0

        # Late fee: 1.5x daily rate per extra day
        late_fee = 0.0
        if actual_return > reservation.return_date:
            extra_days = (actual_return - reservation.return_date).days
            late_fee = extra_days * car.daily_rate * 1.5

        total = cost_obj.calculate() + late_fee

        # Update statuses atomically
        car.status = CarStatus.AVAILABLE
        reservation.status = ReservationStatus.COMPLETED
        reservation.total_cost = total
        self._subject.notify(reservation, ReservationStatus.COMPLETED)

        return Invoice(
            reservation_id=reservation.reservation_id,
            base_cost=base_cost,
            add_on_cost=addon_cost,
            late_fee=late_fee,
            total=total,
        )

                datetime.now() - timedelta(days=1), "IN")
        )
        with pytest.raises(ValueError, match="Invalid or expired"):
            service.reserve(customer, sample_car.car_id,
                datetime.now(), datetime.now() + timedelta(days=1))

    def test_double_booking_raises(self, service, sample_car, valid_customer):
        pickup = datetime.now() + timedelta(days=1)
        ret = pickup + timedelta(days=3)
        service.reserve(valid_customer, sample_car.car_id, pickup, ret)
        with pytest.raises(ValueError, match="not available"):
            service.reserve(valid_customer, sample_car.car_id, pickup, ret)

    def test_return_generates_invoice(self, service, sample_car, valid_customer):
        pickup = datetime.now()
        ret = pickup + timedelta(days=3)
        res = service.reserve(valid_customer, sample_car.car_id, pickup, ret)
        service.pickup(res)
        invoice = service.return_car(res, ret)
        assert invoice.total > 0
        assert sample_car.status == CarStatus.AVAILABLE

    def test_late_return_adds_late_fee(self, service, sample_car, valid_customer):
        pickup = datetime.now()
        due = pickup + timedelta(days=3)
        actual = due + timedelta(days=2)  # 2 days late
        res = service.reserve(valid_customer, sample_car.car_id, pickup, due)
        service.pickup(res)
        invoice = service.return_car(res, actual)
        assert invoice.late_fee > 0

class TestThreadSafety:
    def test_concurrent_reservation_only_one_succeeds(
        self, service, sample_car, valid_customer
    ):
        """Two threads try to book same car simultaneously.
        Exactly one should succeed; the other should get ValueError."""
        results = []
        errors = []
        pickup = datetime.now() + timedelta(days=1)
        ret = pickup + timedelta(days=2)

        def attempt_booking():
            try:
                r = service.reserve(valid_customer, sample_car.car_id, pickup, ret)
                results.append(r)
            except ValueError as e:
                errors.append(str(e))

        t1 = threading.Thread(target=attempt_booking)
        t2 = threading.Thread(target=attempt_booking)
        t1.start(); t2.start()
        t1.join(); t2.join()

        assert len(results) == 1, "Exactly one booking should succeed"
        assert len(errors) == 1, "Exactly one booking should fail"

Interview Q&A

Click any question to reveal a strong answer. These are the questions you're most likely to get.

What design patterns did you use and why? ▾
I used six patterns, each motivated by a specific problem. Singleton for RentalSystem — there must be one source of truth for car inventory. Factory for car creation — different types have different defaults and we don't want that scattered everywhere. Strategy for pricing — pricing rules change often (weekends, bulk, premium members) and should be swappable without touching reservation logic. State for car lifecycle — prevents illegal transitions (you can't rent a car that's already rented) by encoding valid transitions in state objects. Observer for notifications — decouples reservation events from how they're communicated. Decorator for add-ons — GPS + Insurance + Child Seat stack cleanly instead of a mess of boolean flags.
How do you prevent double booking? ▾
I use per-car locks in RentalService. Each car gets its own threading.Lock. When a reservation is attempted, we acquire that car's lock, check its status, and update it atomically — all inside the lock. This means two threads trying to book the same car will be serialized: one waits while the other checks+updates. The first finds the car AVAILABLE and reserves it; the second finds it RESERVED and raises ValueError. I use a mutex to protect the lock-creation dictionary itself, avoiding a race condition in that step too. This is better than locking the entire system because it allows concurrent bookings of different cars.
How would you extend this to support dynamic pricing? ▾
The Strategy pattern makes this easy. I'd create a new class like DemandBasedPricingStrategy implementing PricingStrategy. Its calculate() method would check occupancy rate — if 90%+ of cars are rented, multiply the base rate by 1.3x. Since Reservation only depends on the PricingStrategy interface, we just pass in the new strategy at construction time. Zero changes to existing classes — this is the Open/Closed Principle: open for extension, closed for modification.
What's the difference between State and Strategy pattern in this design? ▾
Both use polymorphism to change behavior, but their intent differs. Strategy represents interchangeable algorithms that can be selected externally — the client picks which pricing algorithm to use. The algorithm doesn't change itself. State represents an object that changes its own behavior based on internal transitions — a car in RentedState transitions itself to AvailableState when returned. State transitions are driven from within the object. In code: RentalService passes a PricingStrategy to a Reservation (external control), while CarContext holds a CarState that swaps itself on valid transitions (internal control).
Why use Decorator for add-ons instead of just a list of add-on costs? ▾
A simple list works but couples the cost calculation logic to Reservation. Every new add-on type means modifying the return_car() method. With Decorator, adding a new add-on like SnowChainAddon means creating one new class and zero modifications elsewhere. The wrapping pattern also naturally gives us a description chain for the invoice — each decorator appends its own label. The trade-off: Decorator adds indirection and can be harder to debug. For a simple system with 3 fixed add-ons, a list might be fine. I chose Decorator to demonstrate the principle and make the system extensible.
How would you add support for multiple branches and inter-branch returns? ▾
Each Car has a branch_id and each Branch manages its own fleet. For search, I'd filter cars by branch. For inter-branch returns, I'd add an inter_branch_fee to Invoice, calculated based on distance between branches (or a flat fee). The return_car() method would check if actual_return_branch != reservation.pickup_branch, apply the fee, and update car.branch_id to the new branch. The Observer would then notify the original branch's manager. No changes to the core Reservation or Pricing logic needed.
How would you scale this for a production system? ▾
The in-memory Singleton works for a single-process system. For production: (1) Replace the Singleton with a database-backed repository. (2) Use distributed locks (Redis SETNX or a database row lock) instead of threading.Lock, since multiple server instances can't share in-process locks. (3) Add a database transaction wrapping the availability check + status update to ensure atomicity. (4) Move notifications to a message queue (Kafka/RabbitMQ) so Observer's notify() publishes an event, and email/SMS services consume it independently. (5) The core domain model — Car, Reservation, Pricing, Decorators — stays unchanged. This is the power of a clean LLD foundation.
What SOLID principles does this design follow? ▾
All five. S (Single Responsibility): Car holds car data, RentalService orchestrates flow, PricingStrategy handles billing — no class does two unrelated things. O (Open/Closed): Add a new pricing strategy or add-on without modifying existing classes. L (Liskov Substitution): Any PricingStrategy subclass can replace another without breaking RentalService. I (Interface Segregation): PricingStrategy has a single method calculate(); ReservationObserver has one method. No fat interfaces. D (Dependency Inversion): RentalService depends on PricingStrategy (abstraction), not DailyPricingStrategy (concrete class).
