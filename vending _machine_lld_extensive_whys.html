Step 1 · Requirements
Requirements & Clarifications
A vending machine sounds like a toy problem. It is not. The moment you add change-return, cancellation, stock, and invalid flows, it becomes a crisp state-machine + inventory + payment design. Your first job is to make the hidden rules explicit.
The prompt you'll receive
"Design a vending machine. Users should be able to select an item, pay, and receive the item."

Clarifying Questions — with deep reasoning

You
"What payment methods should we support — coins, notes, card, UPI, or all of them?"
Why ask this first: Cash and digital payments produce very different designs. Cash needs denomination tracking and change-making. Card/UPI needs a payment processor, pending state, timeout, and failure handling. If you design all of that when the interviewer wanted cash only, you burn time. If you design only a balance when the interviewer wanted exact change, you miss the real problem.
Interviewer
"Cash only. Coins/notes are enough."
You
"Should the machine return change? What if it cannot return exact change?"
Why this changes the class diagram: Returning change means the machine cannot just store current_balance. It needs CashInventory with denomination counts. A machine with ₹100 total cash may still be unable to return ₹30 if it only has one ₹100 note. This one requirement creates the most important supporting class.
Interviewer
"Yes, return exact change. If exact change is unavailable, refund the user."
You
"How are items represented? Does the user select a product name or a physical slot code like A1/B2?"
Why this prevents a beginner bug: Real vending machines are slot-based. Coke can be present in A1 and A2. If stock lives on Product, both slots accidentally share one global count. The correct model is Product = what the item is, Slot = where it physically sits and how many units remain.
Interviewer
"Users select a slot code. Each slot has one product and quantity."
You
"Can the user cancel after selecting or inserting money?"
Why cancellation is not a side feature: Cancellation forces you to track inserted denominations separately from the machine's permanent cash. Until the purchase succeeds, inserted money is in temporary escrow. If you immediately merge it into machine cash, refunding exactly what the user inserted becomes error-prone.
Interviewer
"Yes. User can cancel before dispense and receive inserted money back."
You
"Should we support admin restocking and collecting cash?"
Why this separates core user flow from maintenance flow: Restocking belongs to inventory management, not the normal purchase transaction. Cash collection belongs to machine maintenance. We can include simple methods, but avoid full admin auth/UI unless explicitly requested.
Interviewer
"Add simple restock support. Admin auth is out of scope."

Final Requirements

In scope

  • Users select an item by slot code, for example A1.
  • Each slot stores one product and a quantity.
  • Machine accepts supported cash denominations.
  • Machine tracks inserted money during an active transaction.
  • Machine dispenses only if selected item is in stock and enough money is inserted.
  • Machine returns exact change if possible.
  • If exact change is impossible, machine refunds inserted money and does not dispense.
  • User can cancel before dispense and get inserted money back.
  • Admin can restock slots and add cash/change to the machine.

Out of scope

  • Card/UPI/payment gateway integration.
  • Remote monitoring and IoT telemetry.
  • Expiry dates and product freshness checks.
  • Discounts, combos, coupons, or dynamic pricing.
  • Admin authentication and UI.
  • Multiple vending machines or central fleet management.
Why the vending machine is really a state-machine problem

The machine has strict legal sequences: select before paying, pay before dispensing, refund before reset, never dispense after cancel. If you do not model state explicitly, every method becomes a pile of defensive checks. With a state machine, invalid actions become easy to reject and easy to explain in an interview.

Step 2 · Entity discovery
Core Entities & Relationships
Circle the nouns, then apply the class test: does this thing have state that changes over time and behavior that enforces rules on that state? If yes, class. If it is just data, use a dataclass or enum. Do not make everything a class just because it has a name.

Noun-by-noun evaluation

NounDecisionDeep reasoning — including what breaks if you get it wrong
KeepVendingMachineFull classThe orchestrator. It owns the inventory, cash store, current transaction, and state. External code should talk to this one object, not directly mutate slots and cash. Without it, callers must know the whole internal choreography.
KeepSlotFull classHas mutable quantity and enforces the rule that stock cannot go below zero. This is the physical rack in the machine: A1, A2, B1. Same product can exist in multiple slots, so quantity belongs here, not on Product.
KeepInventoryFull classMaps slot code to Slot and validates slot lookup. If this logic is scattered inside VendingMachine, the orchestrator becomes a warehouse manager too. Inventory is the natural home for slot lookup and restocking.
KeepCashInventoryFull classTracks denomination counts and calculates exact change. A plain integer balance cannot answer "can I return ₹35?" Denominations matter. This class prevents fake correctness.
KeepTransactionFull classHolds temporary purchase state: selected slot, price snapshot, inserted denominations, inserted amount. This prevents reset bugs where one temporary field is cleared and another is accidentally left behind.
DataProductFrozen dataclassMostly immutable descriptive data: name and price. Product does not own stock because stock is physical-slot-specific. Make it frozen so name/price do not accidentally mutate during an active transaction.
EnumDenominationEnumOnly certain cash values are valid. With raw integers, someone can insert ₹37. Enum makes invalid denominations impossible through the public API.
EnumMachineStateEnumStates should not be raw strings. A typo like DISPENCING should fail immediately, not silently strand a transaction.
The key modeling insight

Product is not inventory. Product is the catalog definition. Slot is the physical container. Inventory is the collection of slots. This single separation avoids a surprising number of bugs.

Bad model
Product("Coke", price=40, quantity=8)
This says all Coke stock is one global number. But vending machines have physical slots.
Good model
Slot("A1", Coke, quantity=5)
Slot("A2", Coke, quantity=3)
Same product, two independent physical locations. Correct.
Step 3 · State machine
Machine Lifecycle
A vending machine has one active user flow at a time. State decides which actions are valid right now. This keeps the design honest.
IDLE→ HAS_SELECTION→ HAS_MONEY→ DISPENSING→ IDLE
OUT_OF_SERVICE

Why not use booleans?

Boolean soup
has_selection = False
has_money = True
is_dispensing = False
This impossible state says money exists without a selected item. Which method should be allowed now? Unclear.
Single state
state = MachineState.HAS_MONEY
One machine phase. No impossible combination. Easier guards, easier debugging.

Allowed transitions

VALID_TRANSITIONS = { MachineState.IDLE: [MachineState.HAS_SELECTION, MachineState.OUT_OF_SERVICE], MachineState.HAS_SELECTION: [MachineState.HAS_MONEY, MachineState.IDLE, MachineState.OUT_OF_SERVICE], MachineState.HAS_MONEY: [MachineState.DISPENSING, MachineState.IDLE, MachineState.OUT_OF_SERVICE], MachineState.DISPENSING: [MachineState.IDLE, MachineState.OUT_OF_SERVICE], }
What bug this prevents

Without transitions, someone may call dispense() from IDLE, or call insert_money() during DISPENSING. The transition table makes legal movement explicit and invalid movement loud.

Step 4 · Class design
Class Design — Every Line Justified
Start with the orchestrator, then the most rule-heavy classes. For every field ask: what bug appears if this field does not exist? For every method ask: who should own this rule?

VendingMachine — the Single Entry Point

VendingMachine — fields and methods
class VendingMachine:
Why a dedicated orchestrator?External code should not directly touch slots, cash, and transaction objects. The vending machine exposes a simple public API: select, insert money, buy, cancel. Internally it delegates to the right object.
inventory: Inventory
Why not dict[str, Slot] directly?A direct dict works for a tiny solution, but inventory has behavior: validate slot, restock, check availability. Putting that in Inventory prevents VendingMachine from becoming a god class.
cash_inventory: CashInventory
Why separate cash handling?Change-making is a real algorithm. It depends on denominations and counts. That logic deserves one home instead of being mixed into purchase orchestration.
If omitted: You can know the user inserted enough money, but not whether exact change can be returned.
current_transaction: Transaction | None
Why Optional?The machine spends most of its life with no active purchase. None makes that absence explicit. Methods must handle it instead of assuming a transaction exists.
def select_product(slot_code)
Why selection before payment?The machine needs to know the target price before it can evaluate inserted money. Also, selecting an out-of-stock slot should fail before the user inserts cash.
def buy()
Why explicit buy instead of auto-dispense?Some machines auto-dispense when enough cash is inserted. But explicit buy() is cleaner in interviews: it creates a clear commit point where stock, change, cash acceptance, and reset happen together.
def cancel()
Why does cancel live here?Cancellation coordinates transaction refund and state reset. It is a cross-entity operation, so the orchestrator owns the flow while Transaction owns the inserted denominations.

Slot — Stock Rules Encapsulated

Slot — physical rack with quantity
class Slot: code: str product: Product quantity: int
Why Slot as a class?The slot has mutable quantity and rules around that quantity. Product does not know where it sits physically. Slot does.
def is_available(self): return self.quantity > 0
Why not check quantity directly?Encapsulation. Callers ask the object whether it is available; they do not need to know which fields determine availability. Later you can add is_disabled without changing callers.
def dispense_one(self): if not self.is_available(): raise OutOfStockError() self.quantity -= 1 return self.product
Why re-check inside dispense?Because methods should protect their own invariants. Even if buy() checked stock earlier, another code path could call dispense_one() directly. This method guarantees quantity never goes below zero.
Without this: Quantity can become -1 after a race or forgotten check.
With this: The slot refuses invalid dispense at the source.

CashInventory — Denomination-Aware Change

CashInventory — machine's wallet
cash: dict[Denomination, int]
Why denomination counts?Change is not about total money. It is about whether the required amount can be composed from available coins/notes. ₹100 total cash does not imply ₹30 change is possible.
def can_make_change(amount): return get_change_combination(amount) is not None
Why check before dispense?If the machine dispenses first and then discovers it cannot return change, the transaction is already physically irreversible. Validate money side before releasing the item.
def get_change(amount): combo = calculate_combo(amount) deduct(combo) return combo
Why combine calculation and deduction?Returning change must update the machine's cash state. If callers calculate externally and forget to deduct, the machine believes it still has coins it already gave away.

Transaction — Temporary Purchase State

Transaction — active user session
slot_code: str price_at_selection: int inserted_denominations: list[Denomination] inserted_amount: int
Why this class exists?During a purchase, state is temporary. Selected slot, price, inserted cash, and inserted amount all belong together. One object makes reset/cancel safe.
price_at_selection: int
Why snapshot price?If the product price changes after selection, the transaction should still use the price the user saw. This is the same temporal-data idea used in order systems.
inserted_denominations: list[Denomination]
Why store exact inserted notes?On cancel, the machine should return what the user inserted. Storing only inserted_amount loses denomination information.
Step 5 · Flows
Key Flows — Every Step Justified

Successful purchase

1. User selects slot A1 2. Machine checks state is IDLE 3. Inventory validates A1 exists 4. Slot checks stock > 0 5. Machine creates Transaction with price snapshot 6. User inserts supported denominations 7. Transaction tracks inserted money 8. User presses buy 9. Machine checks sufficient amount 10. Machine checks exact change availability 11. Machine enters DISPENSING 12. Slot dispenses one product and decrements stock 13. Machine accepts inserted money into CashInventory 14. CashInventory returns change and deducts denominations 15. Machine clears transaction and returns to IDLE
Why this order matters

Stock and change are validated before dispense. Cash is permanently accepted only after the machine knows the purchase can succeed. This keeps temporary money separate from machine money until commit.

Insufficient money

1. User selects A1 priced ₹40 2. User inserts ₹20 3. User presses buy 4. Machine sees inserted_amount < price_at_selection 5. Machine refuses purchase 6. Transaction remains active so user can insert more or cancel

Do not auto-refund on insufficient money. The user may simply insert more cash.

Cannot return exact change

1. Product price = ₹40 2. User inserts ₹100 3. Change due = ₹60 4. CashInventory cannot compose ₹60 from available denominations 5. Machine refunds inserted ₹100 6. Machine does not dispense 7. Machine resets to IDLE

This is the flow that exposes weak designs. A simple balance cannot solve it. Denomination-aware cash inventory can.

User cancels

1. User selects product 2. User inserts money 3. User cancels before dispense 4. Machine returns Transaction.inserted_denominations 5. Machine clears current_transaction 6. Machine returns to IDLE

The inserted money was never merged into machine cash, so refund is simple and exact.

Step 7 · Patterns
Design Patterns Used
Facade Pattern VendingMachine

VendingMachine hides all internal complexity. A caller does not need to know about Inventory, Slot, CashInventory, or Transaction. They just call select_product(), insert_money(), buy(), and cancel().

State Machine Pattern MachineState

The legal lifecycle is explicit: IDLE → HAS_SELECTION → HAS_MONEY → DISPENSING → IDLE. This prevents weird flows like dispensing without payment.

Value Object Product

Product is frozen descriptive data. It is passed around safely and does not own mutable physical inventory.

Encapsulation Slot / CashInventory

Stock cannot be decremented except through dispense_one(). Cash denominations cannot be deducted except through get_change(). Rules stay next to the state they protect.

Step 8 · SOLID
SOLID Principles
S
Single Responsibility
One class, one reason to change.
Good: Slot handles stock. CashInventory handles change. Transaction handles temporary purchase state.
Bad: One giant VendingMachine class handles everything directly.
O
Open/Closed
Open for extension, closed for modification.
If card payment is added later, introduce PaymentProcessor and implement CashPaymentProcessor, CardPaymentProcessor, etc.
L
Liskov Substitution
Subclasses should be usable wherever parent type is expected.
A future PaymentMethod abstraction should let cash/card/UPI work without callers knowing which one they received.
I
Interface Segregation
Do not force classes to implement methods they do not need.
Bad: One interface with insert_coin(), swipe_card(), and scan_qr(). Cash does not scan QR. UPI does not insert coins.
D
Dependency Inversion
Depend on abstractions, not concrete implementations.
For an advanced version, inject a ChangeMakingStrategy or PaymentProcessor into VendingMachine. That makes testing and extension cleaner.
Step 9 · Trade-offs
Trade-offs & Edge Cases
Trade-off 1: Auto-dispense vs explicit buy
Scenario: User selects ₹40 Coke and inserts ₹50. Should the machine immediately dispense?
Auto-dispenseFeels like many real machines. Fewer button presses.
Explicit buyCleaner state model. User can cancel after inserting enough money. Better for interview clarity.

For LLD interviews, explicit buy() is usually easier to reason about.

Trade-off 2: Greedy change-making
Scenario: Need ₹30 change. Greedy tries ₹20 + ₹10 if available.

For Indian denominations, greedy usually works for normal coin/note systems, but in arbitrary denominations it can fail. Example denominations: 1, 3, 4. For amount 6, greedy picks 4 + 1 + 1, but optimal is 3 + 3. In production, use dynamic programming/backtracking for arbitrary denominations. For interview cash denominations, greedy is acceptable if you explicitly state the assumption.

Trade-off 3: Include inserted cash while making change?
Scenario: Machine has no ₹10 notes. User inserts ₹50 for ₹40 item. Can the inserted ₹50 help make ₹10 change? No. But if user inserts ₹20 + ₹20 + ₹10 for ₹40 item, the inserted ₹10 can be returned as change.

A realistic machine often holds inserted cash in escrow but may use it in the final change calculation. The implementation in this guide temporarily includes inserted denominations when checking change, then commits cash and deducts change at purchase time.

Trade-off 4: Concurrency
Scenario: Two users try to buy the last item in the same slot.

A physical vending machine usually serves one user at a time, so single-threaded transaction processing is a reasonable assumption. If this were an API for remote reservations, dispense_one() would need locking/transactions around stock decrement.

Step 10 · Interview defense
Interview Q&A
Why is Product not a full class with quantity?
Because quantity belongs to a physical slot. The same product can appear in multiple slots. If quantity lives on Product, A1 and A2 accidentally share stock. Product is catalog data; Slot is inventory state.
Why do we need Transaction?
Because selected slot, price snapshot, inserted amount, and inserted denominations are temporary state. Keeping them in one object prevents reset bugs and makes cancellation clean.
Why not add inserted cash to CashInventory immediately?
Because until purchase succeeds, that money belongs to the user. Treating it as machine cash early makes refund logic messy. The better model is escrow: hold inserted denominations in Transaction, then commit them after successful dispense.
What is the hardest part of this design?
Change-making. Most beginner solutions only track inserted amount and machine balance. But exact change depends on denomination counts. That is why CashInventory is a first-class class.
What happens if dispense fails mechanically after payment succeeds?
That is a production-grade extension. You would need a hardware result from the dispenser. If dispense fails, refund the user and keep/restore stock depending on whether the item actually moved. For interview scope, we assume dispense_one() succeeds once called.
How would you add UPI/card payment?
Introduce a PaymentProcessor interface. Cash payment would use denomination logic. UPI/card would have pending/success/failure states and probably no change-return. This is exactly why payment logic should not be hardcoded across the machine.
Final mental model

VendingMachine is the conductor. Inventory is the shelf system. Slot is one physical rack. Product is item information. Transaction is the current user's temporary session. CashInventory is the machine's wallet. MachineState is the guardrail that keeps the flow legal.
