Step 1 · ~5 minutes
Requirements & Clarifications
The interviewer gives you a vague sentence. Your first job is not to design — it is to interrogate. Every question you ask is a bug you're preventing before you write a single class. Run through the prompt below and notice how little it actually specifies.
The prompt you'll receive
"Design a food ordering app like Swiggy. Users should be able to browse restaurants, place orders, and track delivery."

This sentence leaves almost everything ambiguous. Who manages menus — engineers or restaurants in real time? What does "track delivery" mean exactly? Can an order be cancelled? By whom? When? Can one order span multiple restaurants? These aren't nitpicks — each answer changes a class you must write or eliminates one you shouldn't.

Clarifying Questions — with deep reasoning

You
"What are the core actors in this system? Is it just users ordering, or do restaurant employees and delivery agents also interact with the system?"
Why ask this first: The prompt only mentions "users." But a food ordering app has at least three actors: (1) a customer who places orders, (2) a restaurant employee who accepts/rejects and updates status, and (3) a delivery agent who picks up and delivers. Each actor implies a set of methods you must design. If you skip this question and assume only users exist, you will design a system with no way for a restaurant to say "order accepted" — which makes the state machine meaningless. This question takes 3 seconds to ask and defines the scope of your entire role model.
Interviewer
"Yes — users browse restaurants and menus, place orders, and get status updates. Restaurants accept/reject orders and update status up to READY. Delivery agents pick up and deliver."
You
"Does a single order span multiple restaurants, or is it always one restaurant per order?"
Why ask this — it changes the entire data model: Multi-restaurant orders are a fundamentally different design. A single Order would need multiple sub-orders, each with its own restaurant, prep time, restaurant acceptance, and potentially separate delivery agent. Your Order class would have an order_id → list of SubOrder hierarchy, each SubOrder with its own state machine. The cancellation rules become much more complex: can you cancel one sub-order while the other proceeds? If you start designing for single-restaurant and the interviewer meant multi-restaurant, you'll rebuild half your class diagram. This question saves 20 minutes.
Interviewer
"One restaurant per order. Keep it simple."
You
"What states does an order go through, from placement to delivery?"
Why this question defines your entire Order class: States are not cosmetic labels. Each state is a node in a directed graph. Each allowed transition is an edge. Each forbidden transition (like PLACED → DELIVERED directly) is a missing edge. If you don't know all seven states before writing Order, you will write a state machine with holes — transitions that are accidentally allowed or forbidden. For example: if you don't know the READY state exists, you won't realize agent assignment should happen at READY, not at ACCEPTED. The state machine is the backbone of the most complex class in the system. Get it right before writing any code.
Interviewer
"PLACED → ACCEPTED → PREPARING → READY → PICKED_UP → DELIVERED. And CANCELLED for rejections or early user cancellations."
You
"Who can cancel an order? And until what point in the lifecycle?"
Why the "who" and "until when" are two separate critical questions: The "who" tells you whether cancel_order() needs a cancelled_by parameter (user vs restaurant). If only the user can cancel, the parameter isn't needed. If both can, you may want different compensation rules: restaurant cancellation might give a coupon, user cancellation might not. The "until when" reveals a guard condition on the state machine: CANCELLED is only reachable from PLACED and ACCEPTED, not from PREPARING onward. Without this, a user could cancel while the cook is making their food — operationally absurd and unfair to the restaurant. This guard is encoded directly in your VALID_TRANSITIONS dict and your can_cancel() method.
Interviewer
"Both user and restaurant can cancel. Only before PREPARING starts."
You
"Should we integrate a real payment gateway, or just track payment state?"
Why this determines whether you design 1 class or 5: A real payment gateway (Razorpay, Stripe) involves: async webhooks from the gateway, idempotency keys to prevent double charges, retry logic with exponential backoff, PCI-DSS compliance requirements, and a PaymentGateway interface with concrete implementations per provider. This is an entire sub-system. If the interviewer wants state tracking only, your Payment class is a simple PENDING → PAID → REFUNDED state machine — about 10 lines. Designing the full gateway integration when they wanted just state tracking wastes 15 minutes on scope you don't need.
Interviewer
"Just track state — PENDING, PAID, REFUNDED. No real gateway."
You
"Do restaurants have operating hours? Should the system reject orders if a restaurant is closed?"
Why you need this before writing place_order(): If restaurants can be closed, place_order() must validate this early — before touching inventory. If you forget this requirement, you design a system that processes orders at 3am from a closed restaurant. When the requirement surfaces later, you have to find every place where "is the restaurant valid?" is checked and add the closed check — scattered instead of centralized. Asking now means you add a single field (is_open: bool on Restaurant) and a single guard clause in place_order(), and you're done.
Interviewer
"Yes — restaurants have an open/closed flag. Reject orders if closed."
You
"Is delivery agent assignment automatic or manual? And when in the lifecycle does it happen?"
Why the timing of assignment matters as much as whether it is automatic: "Automatic" means you need an algorithm — which is the Strategy pattern, a full design decision. "Manual" means a dispatcher assigns and you just record it. But the timing question is equally important. Assigning at ACCEPTED vs READY has major operational consequences (explored in trade-offs section). If you design assignment to happen at ACCEPTED without asking, the interviewer who imagined READY assignment will think you made a naive mistake. One sentence of clarification prevents this.
Interviewer
"Automatic — nearest available agent assigned when order reaches READY."
You
"Do menu items have a fixed inventory count? What should happen if two users simultaneously order the last available item?"
Why this question shows senior awareness: Without inventory, you would skip inventory_count entirely and never write deduct_stock(). But the concurrent ordering question is the deeper signal: it forces you to acknowledge that the naive "check-then-deduct" approach has a race condition. Two threads can both pass the is_in_stock() check before either deducts, resulting in negative inventory and two orders for one item. Naming this bug proactively — even if you don't solve it fully in LLD — is the kind of insight that distinguishes a junior from a senior candidate.
Interviewer
"Yes — items have inventory counts. Decrement on order. Raise an error if stock is zero."

Final Requirements

In scope

  • Users browse restaurants (filtered by city/cuisine). Menus show items with price and availability.
  • Users place a single-restaurant order with multiple items and quantities.
  • Full state machine: PLACED → ACCEPTED → PREPARING → READY → PICKED_UP → DELIVERED (+ CANCELLED before PREPARING).
  • Restaurants accept/reject and update status up through READY.
  • Delivery agents are auto-assigned at READY (nearest-to-restaurant), update to PICKED_UP and DELIVERED.
  • Menu items have inventory counts. Decrement on order, raise error if insufficient.
  • Restaurants have an is_open flag. Orders rejected if closed.
  • Payment state tracked: PENDING → PAID (on DELIVERED) → REFUNDED (on CANCELLED).

Out of scope

  • Real payment gateway (Razorpay, Stripe, UPI).
  • Multi-restaurant orders or cart splitting.
  • Coupons, surge pricing, or promotions.
  • GPS tracking and route optimization.
  • User authentication or sessions.
  • Ratings, reviews, and recommendations.
  • UI or rendering layer of any kind.
Why calling out "out of scope" matters

Writing out-of-scope items does three things simultaneously. First, it proves you thought about GPS routing, coupon logic, and auth — and made a deliberate decision to exclude them, not an accidental one. Second, it protects you from scope creep mid-design: if the interviewer drifts toward ratings, you can say "I scoped that out — want me to extend the design?" Third, it makes your design look complete rather than incomplete. A design with no stated boundaries looks like you forgot things. A design with explicit boundaries looks like engineering judgement.

Step 2 · ~5 minutes
Core Entities & Relationships
Read the requirements and circle every noun. That is your candidate list. Now apply a ruthless filter: does this noun have state that changes over time AND behavior (rules) that enforces rules on that state? If yes → class. If only data → dataclass or Enum. If neither → a field on something else. The most common beginner mistake is making everything a class. The second most common is collapsing things that should be separate.

Noun-by-noun evaluation

NounDecisionDeep reasoning — including what breaks if you get it wrong
KeepUser Full class Has mutable state (name, address, order history) and behavior (can place orders, can cancel before PREPARING). If User were just a user_id string, there is nowhere to put "what addresses does this user have saved?" When the interviewer later asks "support multiple delivery addresses," you would have no clean home for it.
KeepRestaurant Full class Has mutable state (is_open, its menu list) and behavior (can be opened/closed, can manage its menu). Critically, is_open is mutable — restaurants open and close throughout the day. If Restaurant were just a string name, place_order() would have no clean place to check whether the restaurant is open. You would end up with a dict like open_restaurants: set scattered somewhere, which is state that belongs on the object itself.
KeepMenuItem Full class This surprises beginners. "It's just a name and price — why not a dict?" Because inventory_count is mutable state, and the rules around it — you cannot go below zero, you must check both is_available AND count, restoring stock must be encapsulated — are behavior. If you store menu items as plain dicts, the inventory rules scatter into wherever they're needed. One caller forgets to check is_available. Another forgets to validate before decrementing. Encapsulating rules in MenuItem means the rules are always enforced, by everyone, automatically — because the only way to change inventory is through the method that validates it.
KeepOrder Full class The most obviously necessary entity. It is the central object that ties everything together: user, restaurant, items, payment, agent, and status. Has a rich state machine (7 states, ~10 transitions), complex behavior (calculate total, cancellation guard, transition validation), and it changes constantly throughout its lifecycle. Almost every system operation either creates, reads, or updates an Order.
KeepOrderItem Full class This is the insight that separates juniors from seniors. You cannot store a list[MenuItem] inside Order. Here is the concrete failure: Ravi orders a Chicken Biryani on Monday when the price is ₹220. The restaurant raises the price to ₹280 on Wednesday. If Order stores a reference to MenuItem, then on Thursday when Ravi opens order history, your code calls order.items[0].price and gets ₹280 — the current price, not the price Ravi was charged. His invoice is wrong. His refund amount is wrong. Your finance team's revenue reports are wrong. OrderItem is an immutable snapshot of exactly what was ordered at exactly what price at exactly the moment of ordering. This is called temporal data integrity — non-negotiable in any system that handles money.
KeepDeliveryAgent Full class Has mutable availability state (three meaningful states: AVAILABLE, ON_DELIVERY, OFFLINE — not a simple boolean) and behavioral rules: can only be assigned to one order at a time, can only complete a delivery if they currently have an assignment. These rules need to live somewhere consistent. If you scatter agent status into a dict or a primitive, there is nothing preventing code from assigning an OFFLINE agent or calling "complete delivery" when no order is active.
KeepPayment Full class Deliberately separate from Order even though it is 1:1 with it. This surprises beginners. "If every Order has exactly one Payment, why not put payment fields inside Order?" The answer is reason for change. Order changes when the food lifecycle changes. Payment changes when the money lifecycle changes. If Swiggy adds split payments (two users split a bill), the Payment model changes drastically — multiple payers, partial amounts, multiple statuses. That change should not open the Order class at all. Separating them now is designing for change — the change will happen, and when it does, it stays local.
SkipCity / Cuisine Enum / string No behavior. No mutable state. "Mumbai" does not do anything. "North Indian" does not enforce any rules. They are pure filter labels. Making them classes would give you an empty methods section — a code smell signaling over-engineering. Keep them as Enum (for type safety and autocomplete) or plain strings on Restaurant.
SkipAddress @dataclass Pure data: street, city, pincode, latitude, longitude. No behavior, no mutable state. Python's @dataclass or NamedTuple is ideal — structured (so fields have names, not just positions), lightweight (no boilerplate), but not a full class with methods. If you find yourself writing methods on Address, you have probably found a behavior that belongs somewhere else (like a distance calculation utility function, not Address itself).
The one-question test for "class vs field"

Ask yourself: "Would this thing ever need to enforce a rule on its own data?" MenuItem enforces "you cannot deduct stock below zero" — that is a rule, so it is a class. Address never enforces anything — it just holds data, so it is a dataclass. If you cannot think of a single meaningful method on your candidate entity, it is almost certainly not a class. When in doubt, start as a dataclass and promote to a full class when you need to add a method.

Why 1:1 Payment-to-Order is not over-engineering

Beginners often collapse Payment into Order because "they're always together." But ask: what changes them? Order changes because a restaurant accepted food, a cook started preparing, or a delivery agent picked up the order. Payment changes because money moved — authorization, capture, refund. These are different business events, often handled by different teams (logistics team vs. payments team). If Swiggy adds EMI installments, or split payments, or corporate billing — the Payment model becomes very complex. None of that complexity should leak into Order. Separate them now and each can evolve independently forever.

Step 3 · ~15 minutes
Class Design — Every Line Justified
Design top-down: start with the orchestrator (the single entry point), then the most complex domain object (Order), then supporting entities. For every field ask: "what happens if this field doesn't exist?" For every method: "who else would own this if I removed it?" If the answer is "nobody," the field/method belongs here.

SwiggySystem — the Single Entry Point

In every well-structured LLD there is one orchestrator class. In Amazon Locker it is Locker. In an Elevator problem it is ElevatorController. In Connect Four it is Game. For Swiggy it is SwiggySystem. External code talks to this class only — nothing else is public-facing. It owns all registries and coordinates all cross-entity operations. It does not implement business rules — it delegates to the right entity. Without this, callers must decide "should I call User.place_order() or Restaurant.receive_order()?" — ambiguity that causes coupling.

SwiggySystem — fields and methods
class SwiggySystem:
Why a dedicated orchestrator? Without a single entry point, callers must know which entity owns which operation. "Place order" — do I call User.place_order()? Restaurant.receive_order()? Both? SwiggySystem is the facade: one public interface for all operations. This hides the internal complexity from external code. The Facade pattern.
restaurants: dict[str, Restaurant]
Why dict, not list? Lookup by restaurant_id is O(1) with a dict vs O(n) with a list. Every operation — place order, search menus, update status — begins by fetching the relevant restaurant. A list means scanning all restaurants every time. Dicts give instant lookup with zero extra code.
If you used a list: place_order() scans all restaurants to find the right one. With 50,000 restaurants in the system, every order placement is a 50,000-item scan — a performance disaster.
users: dict[str, User]
Why validate users centrally? SwiggySystem must verify that a user_id belongs to a real registered user before processing their order. Without this registry, you trust whatever ID is passed in — a security hole. The dict is the system's source of truth for who exists.
Without this: Anyone can pass any user_id string and place an order. There is no check. The order succeeds for a non-existent user.
orders: dict[str, Order]
Why centralize orders instead of storing them on User? Orders are queried from multiple entry points: by users (order history), by restaurants (incoming orders), by agents (assigned order), and by system operations (status updates). Centralizing them in one dict means one authoritative source. If you scattered Order objects across User and Restaurant, you would have inconsistent state when the same Order is updated from two places.
agents: dict[str, DeliveryAgent]
Why does SwiggySystem own agents? Agent assignment is a system-level operation — scan all agents, find the nearest available one. No single entity (not Order, not Restaurant) has visibility into all agents. Only SwiggySystem does. Whoever needs to find agents must own the full list.
def place_order( user_id, restaurant_id, items) → Order
Why not Order.create() instead? Creating an Order requires: validate Restaurant (needs self.restaurants), check each MenuItem stock (needs restaurant's menu), deduct inventory, snapshot prices, create Payment. No single entity has visibility into all of these. If Order.create() existed, it would need a reference back to SwiggySystem — a circular dependency. SwiggySystem.place_order() is the right home because it already owns all necessary registries.
def cancel_order( order_id, cancelled_by) → bool
Why accept cancelled_by? Cancellation has cascading side effects: restore inventory, update payment to REFUNDED, notify user. The cancelled_by parameter enables: (1) audit logging, (2) different compensation logic — if the restaurant cancels, maybe the user gets a coupon; if the user cancels, no coupon. These business rules need to know who cancelled.
def update_order_status( order_id, new_status, actor_id) → bool
Why centralize all status updates here? Status changes have side effects: when READY → trigger agent assignment. When DELIVERED → mark Payment as PAID, free the agent. If entities updated their own status independently, these side effects would be missed. Routing all updates through SwiggySystem ensures the orchestrator can chain all necessary follow-up actions.
If updates were scattered: Restaurant calls order.status = READY directly. Agent assignment never happens. Order sits forever with food ready but no agent.

Order — the Most Complex Class

Order is the heart of the system. Every field exists because something else needs it. Every method enforces a rule that would otherwise be scattered and potentially forgotten. Spend the most time on this class.

Order — fields, state machine, methods
class OrderStatus(Enum): PLACED = auto() ACCEPTED = auto() PREPARING = auto() READY = auto() PICKED_UP = auto() DELIVERED = auto() CANCELLED = auto()
Why Enum instead of string constants? If status is a plain string, nothing prevents order.status = "prepearing" (typo). The bug is silent until something checks the status and the match fails. An Enum makes invalid states unrepresentable at the language level: OrderStatus.prepearing is a NameError caught the moment you type it. Enums also enable exhaustive checking — a type checker warns if you forget a case.
With strings: A typo in a status comparison silently fails. Orders get stuck in a state they can never exit because the comparison never matches.
With Enum: OrderStatus.PREPEARING is a Python error at write time, not a runtime bug at 2am in production.
order_id: str # UUID, not int
Why UUID, not an auto-increment integer? Integer IDs are sequential and guessable. A curious user sees their order ID is 10042 and tries 10043 to probe another user's order data. UUID strings (e.g. "a3f8b2c1-4d5e-...") are 128-bit random values — statistically impossible to guess. Even in an LLD interview, using UUID signals basic security hygiene awareness. It costs nothing and prevents an entire class of attack.
items: list[OrderItem]
Why OrderItem, not list[MenuItem]? This is the temporal data integrity point. MenuItem.price can change at any time. Storing MenuItem references means order.total() returns a different value next week if the restaurant changed their prices. Refund amounts would be wrong. Invoices would be wrong. OrderItem freezes the price at the moment of ordering — it is an immutable record of what was ordered at what price when.
delivery_agent_id: str | None
Why Optional/None, not empty string ""? An order starts with no agent — the agent is assigned only when READY. str | None makes this explicit: any code that uses delivery_agent_id must handle the None case or the type checker warns. An empty string "" as sentinel is easy to forget to check — it passes truthiness tests in some languages and is semantically confusing.
created_at: datetime
Why store creation time — it seems minor? It is never minor in production. Real needs: (1) SLA enforcement — alert if not ACCEPTED within 5 minutes; (2) auto-cancellation — cancel orders unaccepted for 10+ minutes; (3) analytics — peak order hours, average prep time per restaurant; (4) dispute resolution — "I placed this at 7:03, not 7:15." Without created_at, every one of these is either impossible or requires a painful migration to add later.
VALID_TRANSITIONS = { PLACED: [ACCEPTED,CANCELLED], ACCEPTED: [PREPARING,CANCELLED], PREPARING: [READY], READY: [PICKED_UP], PICKED_UP: [DELIVERED], }
Why a dict of allowed transitions? This is the State Machine pattern. The alternative is scattered if/else in every method — transition rules are duplicated and easy to get wrong. Here, all transition logic lives in one data structure. Adding a new state (e.g. RETURNED) is one dict entry. To audit what is allowed from any state, read one dict entry. Nothing is hidden in method bodies. The rules are data, not code.
Without this: You add a RETURNED status for door-step rejections. You add PICKED_UP → RETURNED to update_status(). But you forget to update can_cancel(), which now incorrectly allows cancelling RETURNED orders — triggering a double refund bug in production.
With VALID_TRANSITIONS: CANCELLED is simply absent from RETURNED's entry. The double-refund bug is structurally impossible — no special-case code needed.
def update_status(self, new): allowed = VALID_TRANSITIONS .get(self.status,[]) if new not in allowed: raise InvalidTransitionError() self.status = new
Why raise, not return False? A False return value is silent failure. Callers can and often do ignore it: order.update_status(CANCELLED) — return value discarded. The order stays in PREPARING. The bug is invisible until someone checks the status later and can't explain why a cancelled order is still being cooked. InvalidTransitionError is loud failure — the message says "tried to go from PREPARING to CANCELLED," which is debuggable. Always raise on invalid state transitions.
def can_cancel(self) → bool: return self.status in [ PLACED, ACCEPTED]
Why a dedicated can_cancel() on Order, not inline in SwiggySystem? This is the Tell, Don't Ask principle. SwiggySystem.cancel_order() could check if order.status in [PLACED, ACCEPTED] directly. But that means the rule "cancellable before PREPARING" lives in the orchestrator, not in Order. If you add a second cancellation path (admin portal, restaurant tablet), the developer must remember to copy the check. By putting it on Order, the rule is enforced no matter who calls cancel. The knowledge about what "cancellable" means belongs to Order, not its callers.
def calculate_total(self): return sum( i.price_at_order * i.quantity for i in self.items)
Why compute total dynamically instead of storing it? Storing total_amount as a field means two sources of truth that can drift apart. If an item quantity were somehow updated after creation (even theoretically), the stored total would be stale but the stored value would lie about the current state. Computing from OrderItems is always correct because OrderItems are the source of truth. This is the "derive, don't store" principle: if a value can be reliably computed from other fields, compute it — never store a redundant copy that can become stale.

MenuItem — Inventory Rules Encapsulated

MenuItem — all fields and inventory methods
inventory_count: int
Why int, not float? You cannot have 2.5 portions of a biryani. Inventory is a count of discrete physical units. Using float allows nonsensical states like inventory_count = 1.7 — which arises from floating-point arithmetic bugs like 2.0 - 0.3. int makes the implicit assumption explicit and prevents an entire class of floating-point bugs on something that should be a simple counter.
def is_in_stock(self, qty): return (self.is_available and self.inventory_count >= qty)
Why check BOTH is_available AND inventory_count? inventory_count might be 10, but the restaurant may have paused that item — they ran out of an ingredient. is_available is the restaurant's manual override flag. Both must be true simultaneously. By AND-ing them in one method, every caller gets both checks for free — they physically cannot forget to check one of them, because there's only one method to call.
If you only check inventory_count: A restaurant pauses a biryani because they ran out of rice. The item still shows stock=5. Orders come in. The restaurant can't fulfil them. Angry customers, forced cancellations.
def deduct_stock(self, qty): if not self.is_in_stock(qty): raise InsufficientStockError() self.inventory_count -= qty
Why re-check inside deduct_stock() when place_order() already checked? This is design by contract. Methods must validate their own preconditions regardless of what the caller did. If someone adds a new code path that calls deduct_stock() directly (bypassing the external check), the guard here catches it. Methods should never trust that callers did the right thing — callers are under-pressure developers who sometimes skip guards. The check is one method call — cheap. The bug it prevents is a negative inventory count — expensive to detect and fix in production.
def restore_stock(self, qty): self.inventory_count += qty
Why a restore method instead of direct field access? cancel_order() could do item.inventory_count += qty directly. But that exposes MenuItem's internals to the outside world. If the restore logic ever becomes conditional (e.g. "only restore if the item is still on the menu, not if it was permanently removed"), you would need to find and update every caller. restore_stock() is the single point of change — the logic is encapsulated. This is the principle of encapsulation: expose behavior, not data.

DeliveryAgent — Availability as First-Class State

DeliveryAgent — status, fields, methods
class AgentStatus(Enum): AVAILABLE = auto() ON_DELIVERY = auto() OFFLINE = auto()
Why three states instead of a boolean is_available? A boolean can only distinguish two states: available or not. But "not available" has two meaningfully different sub-cases: the agent is currently on a delivery (they will become available again soon), or they logged out of the app for the day (they will not). This matters for: dispatch scheduling (don't wait for an OFFLINE agent to free up), agent earnings tracking (OFFLINE agents are not working), and re-assignment logic (ON_DELIVERY agents will be available soon). Collapsing these two states into a single False loses information you will inevitably need.
current_order_id: str | None
Why store the order ID on the agent? Two reasons: (1) Consistency check — if status is ON_DELIVERY but current_order_id is None, something went wrong. This inconsistency is immediately detectable. (2) O(1) re-assignment lookup — if an agent cancels mid-delivery, you need to know which order to re-assign. With this field it's instant. Without it, you scan all orders looking for one with this agent's ID — O(n) in the number of orders.
def assign_order(self, order_id): self.current_order_id = order_id self.status = ON_DELIVERY
Why update both fields in one atomic method? These two fields must always be consistent: ON_DELIVERY requires a non-None current_order_id. If you update them separately, there is a window between the two assignments where the object is in an inconsistent state: status is ON_DELIVERY but current_order_id is still None. Even in a single-threaded system this is fragile. One method that does both updates eliminates the inconsistent window — the same reasoning behind database transactions.
def complete_delivery(self): self.current_order_id = None self.status = AVAILABLE
Why not let SwiggySystem update these fields directly? If SwiggySystem did agent.status = AVAILABLE; agent.current_order_id = None directly, it would be reaching into DeliveryAgent's internals. If the completion logic needs to change (e.g. "log delivery duration before going available"), you must find every place that does this sequence. complete_delivery() is the single place — encapsulation means one place to change, not many.

OrderItem — the Price Snapshot (Immutable Value Object)

OrderItem — frozen dataclass with computed subtotal
@dataclass(frozen=True) class OrderItem: item_id: str name: str quantity: int price_at_order: float
Why @dataclass(frozen=True) — what does "frozen" do? frozen=True makes all fields read-only after construction. No code can do order_item.price_at_order = new_price — Python raises a FrozenInstanceError. This enforces the price snapshot guarantee at the language level, not just by convention. Without frozen=True, someone could accidentally mutate the price field — slowly corrupting order history in a way that's hard to trace.
Without frozen: A developer writes a "price correction" script that iterates all OrderItems and updates prices. Suddenly all order history shows current prices. Customers dispute their bills. Finance reconciliation is impossible.
With frozen=True: That script fails immediately on the first FrozenInstanceError. The mistake is caught before any data is corrupted.
@property def subtotal(self): return (self.quantity * self.price_at_order)
Why @property for subtotal instead of storing it? Subtotal = quantity × price_at_order. Both inputs are already stored, so the calculation is always correct and trivially cheap. Storing subtotal as a separate field would create a third source of truth that could drift (if you update quantity but forget to recalculate subtotal). The @property decorator makes it look like a field to callers (item.subtotal not item.subtotal()) while computing it fresh every time. Derive, never store, values that are purely a function of other fields.
Step 4 · ~10 minutes
Key Flows — Every Step Justified
Flows stress-test your entity design. For each step, the question is: which entity is responsible for this, and why that entity rather than any other? Tracing flows end-to-end reveals missing methods, wrong ownership, and sequencing bugs before you write code.

Flow 1 — Placing an Order

SwiggySystem.place_order(user_id, restaurant_id, items_requested)
Step 1: Validate user user = self.users.get(user_id) if not user: raise UserNotFoundError()
Why validate user first — before anything else? Fail fast on the cheapest check. A missing user_id means no order is possible, and validating it costs one dict lookup. If you check it last, you have already done restaurant validation, menu parsing, and inventory checking — all wasted work on a request that will fail anyway. Always guard cheapest preconditions first. This is the "guard clauses first" pattern.
Step 2: Validate restaurant is open if not restaurant.is_open: raise RestaurantClosedError()
Why check is_open before inventory? If the restaurant is closed, no order can proceed regardless of stock levels. Checking this before touching any MenuItem is more efficient (skip all item lookups) and gives the user a clearer, more relevant error: "restaurant is closed" rather than "item out of stock" (which would be misleading — the item isn't the problem). Validate structural preconditions before operational ones.
Step 3: Check ALL items first for item, qty in items_requested: if not item.is_in_stock(qty): raise InsufficientStockError( item.name)
Why check ALL items before deducting ANY? This is the most critical sequencing decision. Imagine you deduct item A, then find item B is out of stock. You must now roll back A's deduction — restore its inventory. That rollback is extra code and an extra risk (what if the rollback fails?). By checking all items first in a read-only pass, and only then deducting, you ensure you will never need a rollback. If any item fails the check, the whole function exits without touching inventory. This is the two-phase commit pattern applied to a simple list.
If you check-and-deduct per item: Items A and B are deducted. Item C check fails. You must now restore A and B's stock. You have introduced a rollback code path. That path rarely executes, so it is rarely tested, so it often has its own bugs.
Step 4: Deduct stock for item, qty in items_requested: item.deduct_stock(qty)
Why deduct before creating the Order object? If deduction fails (race condition: another user grabbed the last item between steps 3 and 4), no Order object is ever created. There is nothing to clean up — no orphan Order records in a bad state. Ordering it this way means a failed order leaves zero trace in the system. The invariant is: if an Order object exists, its inventory was successfully deducted.
Step 5: Snapshot prices order_items = [ OrderItem( item_id=item.item_id, name=item.name, quantity=qty, price_at_order=item.price) for item, qty in items_requested]
Why snapshot here, and what exactly are we snapshotting? We snapshot after confirming all items are valid and in stock — not before, because there's no point creating OrderItems for items that will be rejected. At this exact moment, item.price is captured into price_at_order. Any price change the restaurant makes one second later is irrelevant to this order. We also snapshot item.name — so even if the item is renamed, the order history shows the name that was visible at ordering time. This is the commit point: these items, at these prices, in these quantities, are locked.
Step 6: Create Order order = Order( order_id=str(uuid4()), user_id=user_id, restaurant_id=restaurant_id, items=order_items, status=OrderStatus.PLACED, created_at=datetime.now())
Why PLACED and not ACCEPTED as the initial status? PLACED means "the user has committed to this order but the restaurant has not yet responded." This separation exists because restaurant acceptance is asynchronous — it happens seconds to minutes later when a restaurant employee taps "Accept" on their tablet. If you start at ACCEPTED, you're claiming the restaurant agreed before they were asked. PLACED captures the correct initial state: user commitment, no restaurant response yet.
Step 7: Create Payment payment = Payment( order_id=order.order_id, amount=order.calculate_total(), status=PaymentStatus.PENDING)
Why create Payment last, after Order? Payment is the financial record of commitment. It should only exist if the Order successfully exists. If Order creation somehow fails (UUID collision — rare, but possible), there should be no Payment record orphaned without an Order. PENDING means "money has been authorized but not captured." It moves to PAID only at DELIVERED, because that's when the service is fully rendered. REFUNDED on cancellation returns the money.
Step 8: Register and return self.orders[order.order_id] = order return order
Why register in the dict before returning? The registry is the system's source of truth. If you return the Order but forget to store it, any subsequent call — update_order_status(order_id), cancel_order(order_id) — will raise "order not found." Storing must happen before returning so the Order is immediately findable by anything that comes after. In production this would be a database write inside a transaction along with the inventory deductions — atomicity ensures either everything commits or nothing does.

Flow 2 — The State Machine (each transition explained)

PLACED→ ACCEPTED→ PREPARING→ READY→ PICKED_UP→ DELIVERED

↳ CANCELLED is reachable from PLACED and ACCEPTED only — the cancellation window closes at PREPARING.

Why each transition is what it is
PLACED → ACCEPTED PLACED → CANCELLED
Why does PLACED have exactly two exits? The restaurant either says yes (ACCEPTED) or no (CANCELLED). There is no other reasonable outcome of a freshly placed order. The user can also cancel immediately (changed their mind). These are the only two actors who can act on a PLACED order, and both of their actions are covered. Any other state from PLACED would be operationally nonsensical — you cannot jump to PREPARING without the restaurant accepting first.
ACCEPTED → PREPARING ACCEPTED → CANCELLED
Why can ACCEPTED still go to CANCELLED? ACCEPTED means the restaurant tapped "Accept" — but the cook hasn't started yet. There is a brief window (seconds to a minute) where cancellation is still fair to everyone. Once actual cooking begins, cancellation wastes food, ingredients, and labor. So the cancel window closes not at ACCEPTED (when the intent to cook exists) but at PREPARING (when actual cooking begins). This is a deliberate business policy encoded as a state machine constraint.
PREPARING → READY only CANCELLED not allowed
Why block cancellation after PREPARING? Ingredients have been used. Labor has been spent. A biryani half-cooked cannot become uncooked. Cancellation at this point means wasted food, wasted effort, and lost revenue. The constraint is enforced by simply not including CANCELLED in PREPARING's VALID_TRANSITIONS list. No special case code needed — the absence of the entry makes it structurally impossible to cancel. This is the power of declarative state machines over procedural if/else.
READY → PICKED_UP (agent assigned automatically)
Why assign the agent at READY, not ACCEPTED? Assigning at ACCEPTED means the agent might arrive at the restaurant 15–20 minutes before the food is done. The agent is idle, the food isn't ready. At scale — 50,000 concurrent orders on a Friday night — a large fraction of your agent fleet is sitting outside restaurants waiting. That idle time is lost earnings for agents and wasted capacity for the platform. Assigning at READY means the agent is dispatched when the food is literally ready to go. Food may wait 1–2 minutes for the agent to arrive — far better than an agent waiting 15 minutes for food.
DELIVERED → (terminal) No further transitions allowed
Why make DELIVERED a terminal state with no exits? DELIVERED means the customer physically received the food. There is no reasonable next state. Making it terminal has three important effects: (1) complete_delivery() frees the agent to take new assignments; (2) Payment moves to PAID — the service is rendered; (3) Any call to update_status() on a DELIVERED order raises InvalidTransitionError — preventing accidental double-delivery records or status corruption from a bug that re-processes old events.

Why call is_available() instead of checking status directly? Encapsulation. is_available() owns the definition of "available." Today it checks status == AVAILABLE. Tomorrow it might also check "has this agent done fewer than 10 deliveries today?" (fatigue limit) or "is this agent within the service zone?" Every caller benefits from the updated definition automatically. If you check agent.status == AVAILABLE directly in 6 places, you must update all 6 when the definition changes.

Why nearest-to-restaurant, not nearest-to-user? The agent's first task is to pick up the food from the restaurant. Nearest-to-restaurant minimizes the time between READY and PICKED_UP — the food doesn't wait cooling. After pickup, the agent goes to the user regardless. "Nearest to user" would only make sense with predictive routing (agent pre-positions near user before food is ready), which requires ETA prediction models — out of scope for LLD. This is also exactly where the Strategy pattern becomes powerful: replacing this lambda with an injected AgentAssignmentStrategy.find_agent() lets you swap algorithms without touching this method.

Why update both Order and Agent? The relationship must be recorded bidirectionally. Order knows which agent is handling it (delivery_agent_id). Agent knows which order they're on (current_order_id). This enables O(1) lookup in both directions: "which agent is on my order?" from Order. "Which order is this agent handling?" from Agent. Without one direction, you'd scan all orders or all agents — O(n) — to answer a very common question.
Step 4b · Deep Dive
place_order() — Bad to Great
The most common operation in the system is also the one with the most ways to get wrong. place_order() touches five entities, has three distinct validation layers, and contains a subtle sequencing decision that, if reversed, introduces a class of bugs that is nearly impossible to fix cleanly without a rewrite. Let's build it up from scratch.
Bad Check-and-deduct per item, validation at the end

Approach

The "obvious" implementation: loop through items, check stock and deduct in the same iteration, then validate the restaurant at the end — or worse, don't validate it at all.

def place_order(self, user_id, restaurant_id, items): order_items = [] for item, qty in items: # Check and deduct in the same loop — fatal mistake if not item.is_in_stock(qty): raise InsufficientStockError(item.name) item.deduct_stock(qty) # ← deducted before all items checked order_items.append(OrderItem(...)) # Validate restaurant AFTER touching inventory — wrong order restaurant = self.restaurants.get(restaurant_id) if not restaurant or not restaurant.is_open: raise RestaurantClosedError() # inventory already deducted! 🔥 order = Order(...) return order

Why this fails — two separate bugs

Bug 1 — Partial inventory deduction: You have 3 items. Item A and B deduct successfully. Item C is out of stock — exception raised. But A and B were already deducted. The exception propagates up, the order is never created, but A and B's inventory is now permanently reduced. Two items of stock vanish with no corresponding order. You have "ghost deductions" — inventory that's gone but no order to explain why. In a restaurant with 5 portions of a dish, this means real orders get rejected for "out of stock" that doesn't actually exist.

Bug 2 — Restaurant closed after inventory touched: If the restaurant is closed, this code deducts inventory before discovering the restaurant can't fulfil the order. Again: deductions with no order. The rollback is now required, and it's not there.

The rollback trap: The "fix" that seems obvious is adding rollback logic — if something fails, restore all the deductions made so far. This is more code, more complexity, and a path that executes rarely (in tests, never). Rollback code is the most undertested code in any codebase. Its bugs are discovered in production at the worst possible moment.
Good Validate everything first, then deduct

The insight

Separate the two phases completely. Phase 1: read-only validation — check everything that can fail, touch nothing that changes state. Phase 2: mutation — deduct inventory, create objects. If Phase 1 passes completely, Phase 2 cannot fail (within our in-memory model). No rollback needed because no mutation happened during validation.

def place_order(self, user_id, restaurant_id, items): # ── Phase 1: Read-only validation. Nothing mutated. ────────────── user = self.users.get(user_id) if not user: raise UserNotFoundError() restaurant = self.restaurants.get(restaurant_id) if not restaurant: raise RestaurantNotFoundError() if not restaurant.is_open: raise RestaurantClosedError() # Check ALL items before deducting ANY for item, qty in items: if not item.is_in_stock(qty): raise InsufficientStockError(item.name) # ── Phase 2: Mutation. Only reached if Phase 1 passed fully. ───── for item, qty in items: item.deduct_stock(qty) order_items = [OrderItem( item_id=item.item_id, name=item.name, quantity=qty, price_at_order=item.price ) for item, qty in items] order = Order( order_id=str(uuid4()), user_id=user_id, restaurant_id=restaurant_id, items=order_items, status=OrderStatus.PLACED, created_at=datetime.now()) payment = Payment( order_id=order.order_id, amount=order.calculate_total(), status=PaymentStatus.PENDING) order.payment = payment self.orders[order.order_id] = order return order

Why this is the right structure

The invariant is clean: if an Order object exists in the system, its inventory was fully and correctly deducted. If any validation fails, no inventory is touched, no Order is created, and there is nothing to roll back. The two-phase structure makes this invariant trivially easy to reason about.

The remaining question

The code is correct, but within Phase 1, does the order of validation checks matter? User check vs. restaurant check vs. stock check — does it matter which comes first?

Great Two-phase with guard ordering: cheapest checks first

The insight

Within Phase 1, order your checks from cheapest to most expensive and from most-likely-to-fail to least. A missing user_id is one dict lookup. Checking whether the restaurant is open is one field access. Scanning all items for stock availability requires N dict lookups. If the user doesn't exist, skip all of it. This is the "guard clause first" pattern — fail fast on the cheapest precondition.

# Guard ordering — most-likely-to-fail first, cheapest first # # 1. user validation — O(1) dict lookup. If no user, nothing else matters. # 2. restaurant found — O(1) dict lookup. Cheap structural check. # 3. restaurant open — O(1) field access. Cheap. Fails for 3am orders. # 4. stock checks — O(N) item loop. Expensive. Only run if above pass. # # Why does this order matter beyond performance? # Error clarity: if the restaurant is closed, the user gets # "restaurant is closed" — not "item out of stock." # The latter is misleading: the item IS in stock, but the # restaurant can't serve it right now. Wrong error → wrong # user action → frustrated customer → support ticket. def place_order(self, user_id, restaurant_id, items): # Guard 1: entity existence — O(1) each user = self.users.get(user_id) if not user: raise UserNotFoundError(user_id) restaurant = self.restaurants.get(restaurant_id) if not restaurant: raise RestaurantNotFoundError(restaurant_id) # Guard 2: structural state — O(1) if not restaurant.is_open: raise RestaurantClosedError(restaurant.name) # Guard 3: per-item stock — O(N), only if above passed for item, qty in items: if not item.is_in_stock(qty): raise InsufficientStockError( f"{item.name}: requested {qty}, " f"available {item.inventory_count}") # ── Phase 2: mutation — only reachable if all guards passed ────── for item, qty in items: item.deduct_stock(qty) order_items = [OrderItem( item_id=item.item_id, name=item.name, quantity=qty, price_at_order=item.price ) for item, qty in items] order = Order( order_id=str(uuid4()), user_id=user_id, restaurant_id=restaurant_id, items=order_items, status=OrderStatus.PLACED, created_at=datetime.now()) order.payment = Payment( order_id=order.order_id, amount=order.calculate_total(), status=PaymentStatus.PENDING) self.orders[order.order_id] = order return order

The three properties this design guarantees

  • No partial mutations: Phase 2 only starts if Phase 1 passes completely. The invariant "Order exists ↔ inventory was fully deducted" holds unconditionally.
  • No rollback code: Rollback is only needed when you mutate before you're sure you should. Two-phase design eliminates the need entirely.
  • Correct error messages: Guard ordering ensures the user gets the right error for the right reason. A closed restaurant surfaces that error, not a downstream stock error that would be misleading.
What to say in the interview: "I separate place_order() into two phases: a read-only validation phase and a mutation phase. Phase 2 only starts if Phase 1 passes completely. This eliminates any need for rollback — no mutation happens until I'm certain everything is valid. Within Phase 1, I order checks from cheapest to most expensive, which also ensures the user gets the most meaningful error message."

Design Patterns — Justified, Not Just Named
Don't sprinkle patterns to impress. Name a pattern only when you can say: "Without this pattern, here is the specific problem I would face." Each pattern below includes the concrete problem it solves, how it's applied, and code showing the before-vs-after.
1. State Machine Pattern applied in Order

The specific problem without it: Every method on Order that cares about valid transitions would have scattered if/else blocks. update_status(), cancel_order(), and update_status(DELIVERED) each need to know what transitions are legal. Each time you add a state, you must find and update every method. Missing one causes bugs like: a DELIVERED order that can still be CANCELLED, triggering a double-refund.

There are three levels to how well you can model this. Each solves a specific failure the previous level has.

Bad Scattered if/else in every method

Approach

Each method that cares about transitions encodes the rules itself. update_status() has one set of checks. can_cancel() has another. Each developer who adds a feature adds their own if block.

def update_status(self, new): if self.status == PLACED: if new not in [ACCEPTED, CANCELLED]: raise InvalidTransitionError() elif self.status == ACCEPTED: if new not in [PREPARING, CANCELLED]: raise InvalidTransitionError() elif self.status == PREPARING: if new != READY: raise InvalidTransitionError() # ... 4 more elif blocks for remaining states def can_cancel(self): # Developer separately hard-codes the cancellable states return self.status in [PLACED, ACCEPTED]

Why this fails

The rules are now in two places: update_status() and can_cancel(). They are independent. When a developer adds a RETURNED state (food rejected at the door), they add PICKED_UP → RETURNED to update_status() but forget to check can_cancel(). can_cancel() checks status in [PLACED, ACCEPTED] — RETURNED is not in that list, so it returns False correctly by accident. But if CANCELLED is never added to RETURNED's allowed transitions, you now have an inconsistency — one method allows the transition, another blocks it. Future developers can't tell which one is right.

The real production bug: Add RETURNED → CANCELLED to update_status (seems reasonable) but forget to add RETURNED to can_cancel's list. Now cancel_order() calls can_cancel(), gets False, and rejects the cancellation — even though update_status would have allowed it. Two methods, two conflicting answers to the same question. Customer can't cancel their rejected order. Support tickets flood in.
Good Centralise into VALID_TRANSITIONS dict

The insight

Rules as data beats rules as code. Move all transition logic into one dict. update_status() consults the dict. The if/else chain collapses to two lines. Adding a new state is one dict entry, zero method changes.

VALID_TRANSITIONS = { PLACED: [ACCEPTED, CANCELLED], ACCEPTED: [PREPARING, CANCELLED], PREPARING: [READY], READY: [PICKED_UP], PICKED_UP: [DELIVERED], # DELIVERED and CANCELLED intentionally absent — terminal states } def update_status(self, new): allowed = VALID_TRANSITIONS.get(self.status, []) if new not in allowed: raise InvalidTransitionError( f"Cannot go from {self.status} to {new}") self.status = new def can_cancel(self): # Still separate: checks status directly return self.status in [PLACED, ACCEPTED]

Why this is much better

Adding a RETURNED state is now one dict entry: PICKED_UP: [DELIVERED, RETURNED]. update_status()'s body never changes. You can audit the entire state machine in seconds by reading the dict. Terminal states (DELIVERED, CANCELLED) are represented by their absence from the dict — no special-case code needed.

The remaining problem

can_cancel() still hard-codes [PLACED, ACCEPTED]. It is not derived from VALID_TRANSITIONS. They can still drift apart. If a future developer adds RETURNED → CANCELLED to the dict but forgets to add RETURNED to can_cancel()'s list, the inconsistency is back. The centralisation is incomplete.

Great can_cancel() derives from VALID_TRANSITIONS — single source of truth

The insight

"This order is cancellable" means exactly one thing: the CANCELLED state is a valid transition from the current status. That fact is already encoded in VALID_TRANSITIONS. can_cancel() should not maintain its own separate list — it should ask the dict.

VALID_TRANSITIONS = { PLACED: [ACCEPTED, CANCELLED], ACCEPTED: [PREPARING, CANCELLED], PREPARING: [READY], READY: [PICKED_UP], PICKED_UP: [DELIVERED], } def update_status(self, new): allowed = VALID_TRANSITIONS.get(self.status, []) if new not in allowed: raise InvalidTransitionError() self.status = new def can_cancel(self): # Derived directly from the dict — no separate hard-coded list return CANCELLED in VALID_TRANSITIONS.get(self.status, []) # Now add RETURNED state — only one change needed: # PICKED_UP: [DELIVERED, RETURNED] # RETURNED: [CANCELLED] # can_cancel() automatically returns True for RETURNED. # No other code changes. Structural impossibility of drift.

Why this is the right design

"Cancellable" is defined entirely by whether CANCELLED appears in a state's allowed transitions. By deriving can_cancel() from VALID_TRANSITIONS rather than a separate list, there is literally only one place to update. The drift bug is structurally eliminated — not by discipline or code review, but by design.

Adding RETURNED state with this design: Add two entries to the dict. can_cancel(), update_status(), and every other method that uses the state machine automatically handle it correctly. Zero additional changes. Zero risk of the inconsistency that plagued the Bad solution.
What to say in the interview: "I derive can_cancel() from VALID_TRANSITIONS rather than maintaining a separate list. This way there is exactly one source of truth for transition rules. Adding any new state is one dict entry — no method needs to be audited for missed cases."

The key insight: Rules as data (the dict) beats rules as code (if/else). Data is easier to inspect, test, and extend. You can print the entire state machine in one line: print(VALID_TRANSITIONS). You cannot "print" scattered if/else logic. And deriving all guards from the single dict closes the last gap.

Hardcoding all four actions inside Order.update_status() means Order now has four reasons to change: adding WhatsApp requires editing Order, removing SMS requires editing Order, changing payment pre-auth logic requires editing Order. 3. Strategy Pattern for agent assignment algorithm

The specific problem without it: "Nearest available agent" is the current algorithm. Swiggy might later want "highest rated agent," "agent with fewest deliveries today," or "agent matched to restaurant's average speed." Without Strategy, each algorithm change requires editing SwiggySystem.assign_delivery_agent() — touching the most critical part of the system for what is essentially a business policy decision.

Before we look at the pattern, let's see how the algorithm itself evolves. Three quality levels — each solves a real failure the previous one has.

Bad Random / first-available agent

Approach

The simplest thing: iterate the agents list, return the first one whose status is AVAILABLE. No algorithm at all.

def assign_agent(self, order): for agent in self.agents.values(): if agent.status == AVAILABLE: return agent raise NoAgentAvailableError()

Why this fails

The order of iteration over a dict is insertion order — the first agent registered is always chosen until they go offline. Agent #1 gets 40 deliveries, Agent #7 gets 3. Agent #1 burns out. Agent #7 files a complaint about unfair work distribution. More critically: Agent #1 might be 12 km away from the restaurant while Agent #3 is 500 m away. The customer waits 25 minutes instead of 8. This is not a hypothetical — in a real system at peak hour, bad assignment directly translates to cold food and poor reviews.

Good Nearest available agent (distance only)

Approach

Filter to available agents, then pick the one with minimum Euclidean distance to the restaurant. The agent closest to where the food is ready arrives fastest.

def find_agent(self, agents, order): available = [a for a in agents if a.status == AVAILABLE] if not available: raise NoAgentAvailableError() return min(available, key=lambda a: distance( a.location, order.restaurant.location))

Why this is better

This is correct for the common case and easy to explain. Nearest agent minimises pickup time, which directly reduces food wait time for the customer. This is what you should implement and defend in most interviews.

The remaining problem

Distance doesn't account for agent load. Imagine all 5 available agents are bunched 400 m from the restaurant. The nearest one has already done 11 deliveries today and is exhausted. Meanwhile a fresh agent 600 m away could handle it better. Pure distance ignores operational health of the agent pool — a real concern at scale where agent fatigue leads to more cancellations and slower deliveries over time.

There's a subtler issue too: if the system has 2 orders arriving simultaneously from nearby restaurants, the nearest-agent algorithm might dispatch the same agent to both (one gets missed), rather than spreading the two orders across two nearby agents intelligently.

Great Weighted score: distance + load + rating

The insight

A single metric (distance) is rarely enough for a production system. Model agent suitability as a score composed of multiple weighted factors. Each factor contributes differently, and the weights can be tuned operationally without touching the algorithm code.

class WeightedScoringStrategy(AgentStrategy): # Weights tunable via config — no code change needed W_DISTANCE = 0.6 W_LOAD = 0.25 W_RATING = 0.15 def find_agent(self, agents, order): available = [a for a in agents if a.status == AVAILABLE] if not available: raise NoAgentAvailableError() rest_loc = order.restaurant.location def score(agent): # Lower distance → higher score (invert) dist_score = 1 / (1 + distance(agent.location, rest_loc)) # Fewer deliveries today → higher score (invert) load_score = 1 / (1 + agent.deliveries_today) # Higher rating → higher score (direct) rating_score = agent.rating / 5.0 return (self.W_DISTANCE * dist_score + self.W_LOAD * load_score + self.W_RATING * rating_score) return max(available, key=score)

Why this is better

Distance still dominates (0.6 weight) — so nearby agents are preferred. But a fresh agent 200 m farther scores better than an exhausted agent right next door. A highly rated agent breaks ties meaningfully. The weights are configuration, not code — an ops team can tune them based on real delivery data without a deployment.

The key point for the interviewer: In an interview you implement the "Good" tier — nearest available — and then proactively say: "In production, I'd use a weighted scoring strategy that factors in agent load and rating. The Strategy pattern means this is a new class with zero changes to SwiggySystem — I swap the injection." That sentence earns you senior-level credit without spending time implementing the full scorer.

Now the pattern itself: All three algorithms above implement the same interface. That's Strategy. SwiggySystem doesn't know or care which one is wired in — it just calls self.strategy.find_agent(available, order).

4. Factory Pattern for Order creation

The specific problem without it: Order creation involves 7+ steps: generate UUID, validate inputs, check inventory, deduct stock, snapshot prices, create OrderItems, create Payment. If you spread these steps across a constructor and a few helper methods, it's possible to create a half-initialized Order — one where inventory was deducted but Payment was never created, or one where items were snapshotted but stock was never deducted. A half-initialized object is a silent bug that corrupts data silently.

The Factory solution: OrderFactory.create(user, restaurant, cart) owns all creation steps in sequence. It either returns a fully initialized Order or raises an exception. No caller can construct an Order any other way. The constructor can be made private or simply documented as "do not call directly." The invariant is enforced structurally.

5. Repository Pattern for storage abstraction

The specific problem without it: SwiggySystem currently uses Python dicts. Step 6 · Deep Dive
SOLID Principles — Each With a Real Violation Scenario
SOLID is not a checklist to recite in interviews. Each principle solves a real category of bug. For each principle: the formal definition, what a violation looks like concretely in this Swiggy design, and the specific decision that upholds it.
S
Single Responsibility Principle
"A class should have one, and only one, reason to change."
Violation in Swiggy: Imagine Order has: update_status(), calculate_total(), send_push_notification(), charge_payment(), and assign_nearest_agent(). Now Order changes when: the state machine rules change (logistics team), pricing rules change (product team), notification service changes (mobile team), payment gateway changes (payments team), and agent algorithm changes (ops team). Five teams edit the same class. Constant merge conflicts. One team's change accidentally breaks another team's logic.
How this design upholds it: Order has one reason to change — order lifecycle rules. Payment has one reason — money state. MenuItem has one reason — inventory rules. Each class is the exclusive owner of one concept, edited by one team, for one reason.
O
Open / Closed Principle
"Software entities should be open for extension, but closed for modification."
Violation in Swiggy: Swiggy wants to add WhatsApp order notifications. Without OCP: open Order.update_status(), add whatsapp_service.send(...) after the push notification call. Now Order — already tested and stable — is edited. Any mistake in the WhatsApp logic can break push notifications. Every new channel requires editing this method forever.
How this design upholds it: Add a WhatsAppObserver class implementing OrderObserver. Register it with Order at startup. Order.update_status() doesn't change — it just calls obs.on_status_change() for each observer, and the new observer is now in the list. Extension by addition, not modification.
L
Liskov Substitution Principle
"Subclasses must be substitutable for their base class without breaking the program."
Violation in Swiggy: HighPriorityStrategy extends AgentAssignmentStrategy but overrides find_agent() to return None when no premium agent is available — breaking the contract that find_agent() always returns an agent or raises an exception. SwiggySystem calls agent.assign_order(order_id) on the returned value. It crashes with AttributeError: 'NoneType' has no attribute 'assign_order'. The base class promised an Agent. The subclass returned None. Substitution is broken.
How this design upholds it: Every concrete strategy must fulfill the same contract: find_agent() returns a DeliveryAgent or raises NoAgentAvailableError — never None, never a different exception type. SwiggySystem handles both cases. Any strategy can be substituted safely.
I
Interface Segregation Principle
"Clients should not be forced to depend on interfaces they don't use."
Violation in Swiggy: A single ISystemActor interface with methods: place_order(), cancel_order(), accept_order(), assign_agent(), mark_delivered(). DeliveryAgent implements this interface — but must provide empty or raising implementations for place_order(), cancel_order(), and accept_order() since agents do none of those. Empty method bodies are a code smell: they signal a poorly designed interface forcing actors to implement things irrelevant to them.
How this design upholds it: Separate narrow interfaces: IOrderPlacer (for Users: place, cancel), IOrderAcceptor (for Restaurants: accept, reject, update status), IDeliverer (for Agents: pick up, deliver). Each actor implements only the interface relevant to their role. OrderObserver has exactly one method. No one implements what they don't need.
D
Dependency Inversion Principle
"High-level modules should not depend on low-level modules. Both should depend on abstractions."
Violation in Swiggy: SwiggySystem.__init__() creates self.strategy = NearestAgentStrategy() directly. Order.__init__() creates self.notifiers = [TwilioSMSNotifier(), FirebasePushNotifier()] directly. Now: (1) You cannot test SwiggySystem without running the real nearest-agent algorithm. (2) You cannot test Order without the real Twilio and Firebase SDKs. (3) Switching from Twilio to AWS SNS means opening Order's constructor — a high-level business class — to change a low-level infrastructure detail.
How this design upholds it: SwiggySystem.__init__(strategy: AgentStrategy) — the abstraction is injected, not created internally. Order.__init__(observers: list[OrderObserver]) — same. In production code: SwiggySystem(strategy=NearestAgentStrategy()). In tests: SwiggySystem(strategy=FakeStrategy()). Tests run in milliseconds, with zero network calls, zero real notifications, zero real agent selection. This is only achievable if dependencies are injected from outside — high-level logic never knows which low-level implementation is wired in.

The practical impact: A full test suite for SwiggySystem that runs in under 1 second with zero real services is only possible because of DIP. Without it, tests require real infrastructure, run slowly, and fail randomly due to network issues. DIP is not about cleanliness — it is about making the system testable, which is about making it maintainable at scale.

Step 7 · Deep Dive
Trade-offs — Concrete Failures, Not Abstractions
Every design decision is a bet. The wrong choice doesn't fail aesthetically — it fails with a specific, real scenario. For each trade-off: what breaks, why it breaks, and what the correct choice costs.
1. OrderItem price snapshot vs direct MenuItem reference
Concrete failure if you store MenuItem references: Ravi orders Chicken Biryani at ₹220 on Monday. Restaurant changes the price to ₹280 on Wednesday. On Thursday, Ravi opens order history. Your code calls order.items[0].item.price and gets ₹280 — not the ₹220 Ravi was charged. (1) His receipt is wrong. (2) If he cancels for a refund, your system calculates the refund as ₹280 — ₹60 more than he paid. (3) Your finance team's daily revenue report adds ₹280 to Monday's revenue, not ₹220. (4) Your tax reporting is wrong. This is a data integrity failure with direct financial and legal consequences. No amount of "we'll be careful" prevents this — it's a structural issue.
✓ Chosen: OrderItem snapshotprice_at_order is captured at the moment of ordering and frozen forever. Order history is historically correct regardless of any future price changes. Refund amounts are always exactly what the customer paid. Financial reconciliation is auditable.
Cost: slight storage duplicationYou duplicate item name and price in every OrderItem. For millions of orders, this is real storage — but negligible compared to the cost of financial data corruption. Data correctness always outweighs storage savings in financial systems.
2. State machine dict vs scattered if/else
Concrete failure without the state machine: A developer adds a RETURNED status for orders where the customer refused delivery at the door. They add PICKED_UP → RETURNED to the update_status() if/else block. But they forget to update can_cancel(), which still only checks status in [PLACED, ACCEPTED] — RETURNED is not in that list, so the check passes. Now a user can call cancel_order() on a RETURNED order, triggering a refund. The order was already RETURNED (food came back to restaurant) — the restaurant isn't compensated, and the customer gets a refund for food they actually received. With VALID_TRANSITIONS, CANCELLED is simply absent from RETURNED's entry. can_cancel() never needed to be updated — it delegates to the dict, which already prevents it.
✓ Chosen: declarative VALID_TRANSITIONS dictRules are data, visible in one place. Adding a state = one dict entry. Method body never changes. Testing all valid transitions = one assertion. Structurally impossible to have an unintentional transition.
Alternative: separate OrderStateMachine classBetter for very complex state machines with entry/exit actions per state. For 7 states with simple transition rules, the dict approach is more readable and sufficient. Over-engineering for interview scope.
3. Inventory check-then-deduct vs atomic database lock
Concrete failure with check-then-deduct in a concurrent system: Restaurant has 1 Gulab Jamun left (inventory_count = 1). Priya and Arnav both open Swiggy simultaneously. Both apps call is_in_stock(1) — both see True (inventory is 1, neither has deducted yet). Priya's thread calls deduct_stock(1) first — inventory = 0. Then Arnav's thread calls deduct_stock(1) — inventory = -1. Both orders succeed. The restaurant receives two orders they can only fulfil one of. They must cancel one, refund a customer, and issue an apology coupon. This is a TOCTOU race condition (Time-Of-Check-Time-Of-Use) — the most common inventory bug in e-commerce systems. It is structurally unavoidable with check-then-deduct in a multi-threaded or multi-server environment.
✓ For LLD interview: simple deduct_stock()The single-threaded in-memory model works correctly for interview scope. The code is clean, readable, and demonstrates the intent. Mention the race condition explicitly — earning credit without needing to implement SQL locking in a 35-minute interview.
Production fix: SELECT FOR UPDATEIn PostgreSQL: SELECT * FROM menu_items WHERE id = ? FOR UPDATE acquires a row-level lock. Check and deduct become one atomic operation. Alternatively: optimistic locking with a version field — if version mismatches on update, retry the operation. Redis: DECR is atomic by design — use it for high-throughput inventory.
4. Agent assignment: at READY vs at ACCEPTED
Concrete operational failure if you assign at ACCEPTED: Restaurant accepts order at 7:00pm. An agent is immediately dispatched and arrives at the restaurant at 7:08pm. The restaurant takes until 7:25pm to prepare the food (complex order on a busy evening). The agent has been waiting outside for 17 minutes — idle time with no earnings, wasted fuel, blocking their next assignment. At peak demand on a Friday evening with 50,000 concurrent orders in Mumbai, assume 30% of orders have agents waiting 10+ minutes for food. That is 15,000 agents spending 10 minutes idle per order. Multiply by agent opportunity cost and fuel: millions of rupees of waste per week. Real food platforms assign at READY (or even later, using predictive ETA models) precisely to minimize this.
✓ Chosen: assign at READYAgent is dispatched exactly when food is ready for pickup. Minimizes idle wait time at restaurant. Maximizes deliveries per agent per shift. Food may wait 1–2 minutes for agent arrival — far better than 10–15 minutes of agent idle time.
Cost: brief post-READY waitFood sits warm for a minute or two. Real Swiggy mitigates this with predictive pre-positioning — routing agents toward restaurants whose orders will likely hit READY soon. This requires ETA prediction ML models, out of LLD scope. Worth mentioning to show awareness of real-world solutions.
5. Payment as separate class vs fields inside Order
Concrete merge conflict scenario if payment is inside Order: Swiggy adds split payments (two users split the bill). Now Order needs two payer_user_id fields, two payment_status fields, two amounts. Swiggy adds EMI — a payment now has a list of installments, each with due date and status. Order is bloated. Swiggy adds corporate billing — companies get monthly invoices. Order needs an invoice_id, a billing_company_id, and payment_status becomes "invoiced" not "paid." Now the payments engineer and the logistics engineer both modify Order constantly. Every time either team deploys, the other team must rebase and resolve merge conflicts on the same file. This is the operational cost of insufficient separation.
✓ Chosen: Payment as separate classOrder tracks food lifecycle. Payment tracks money lifecycle. Each class can evolve independently and in parallel. Adding new payment types never requires opening Order. Different teams own different classes with no merge conflicts.
Cost: one extra indirectionFor the simple single-payer case, a separate Payment class feels like indirection for no benefit. Acceptable: the cost is one extra method call. The benefit is a clean boundary that pays off the moment payment requirements change — and payment requirements always change.

Interview Q&A — Deep Answers
These are the questions that come after you present your design. They test whether you understand your own design or just memorized it. Every answer below goes deeper than the surface — not just what, but exactly why and what breaks if you choose differently.
How would you handle concurrent orders for the same last item?

This is a TOCTOU (Time-Of-Check-Time-Of-Use) race condition — the most common inventory bug in e-commerce. Here's the exact failure sequence:

Both Priya and Arnav call is_in_stock(1) — both see inventory_count = 1, both get True. Neither has deducted yet. Priya's thread deducts: inventory = 0. Arnav's thread then deducts: inventory = -1. Both orders succeed. Negative inventory means one order can't be fulfilled.

For the LLD interview (what you say)

"My current in-memory implementation has this race condition. In production I'd fix it at the database layer using SELECT FOR UPDATE — this acquires a row-level lock on the menu_items row, making the check and deduct one atomic operation. Alternatively, optimistic locking: store a version number on each row, read it with the check, and include WHERE version = $read_version in the UPDATE. If version mismatched (someone else deducted first), retry. For very high throughput, use Redis DECR which is atomically guaranteed."

Simply naming the race condition and its production fix earns significant credit. Most candidates don't acknowledge it at all.

Why did you separate Payment from Order when they're always 1:1?

The 1:1 ratio today does not predict the 1:1 ratio tomorrow. Here's the argument from first principles:

Reason-for-change test: Order changes when the food lifecycle changes — restaurant accepts, cook starts, agent picks up. Payment changes when money moves — authorization, capture, refund. These are different events, triggered by different systems, modified by different teams.

What happens if you collapse them: Swiggy adds split payments (two friends share a meal). Now Order needs two payer_user_ids, two amounts, two payment_status fields. The payments engineer opens Order to add these. Meanwhile the logistics engineer is adding estimated delivery time fields. Both engineers are in the same file. Merge conflicts weekly.

The correct answer when pushed: "The classes are 1:1 today, but they have different reasons to change and different rates of change. Order evolves when the logistics flow changes. Payment evolves when the money flow changes. Keeping them separate means each team owns their class with no cross-team interference. The cost is one extra object dereference — easily worth the boundary it creates."

How would you add coupon/discount support without breaking existing classes?

This tests your understanding of the Open/Closed Principle — extend by adding, not by modifying.

Step 1: New Coupon entity. Create a Coupon class with: code: str, discount_type: Enum (FLAT or PERCENT), discount_value: float, min_order_value: float, is_valid(order_total) → bool.

Step 2: Minimal change to calculate_total. Add an optional parameter: def calculate_total(self, coupon: Coupon | None = None) → float. If coupon is provided and valid, apply the discount. Existing callers pass no coupon — they are unaffected.

Step 3: Coupon validation in SwiggySystem. place_order() accepts an optional coupon_code: str | None. If provided, fetch the coupon from a coupon_registry dict, validate it, and pass it to OrderFactory or calculate_total. Coupon validation — is it expired? minimum order met? user hasn't used it already? — lives on Coupon, not scattered in SwiggySystem.

Net result: one new class, two small additions to existing method signatures. No existing class is fundamentally changed. This is the OCP in action.

What happens end-to-end when a restaurant rejects an order?

The full sequence when a restaurant taps "Reject":

1. Status transition: order.update_status(OrderStatus.CANCELLED) — PLACED → CANCELLED is a valid transition in VALID_TRANSITIONS. No exception raised.

2. Inventory restoration: For each item in order.items, call menu_item.restore_stock(order_item.quantity). The items are back in the restaurant's available stock. Another customer can order them.

3. Payment refund: order.payment.status = PaymentStatus.REFUNDED. In production this triggers a refund to the customer's payment method. In our LLD, it's a status update.

4. Observer notifications: Order calls obs.on_status_change(order, CANCELLED) for all registered observers. PushNotificationObserver sends "Your order was cancelled by the restaurant." SMSObserver sends an SMS. Each observer acts independently — adding a new notification channel doesn't change this flow.

5. Compensation (optional extension): A CancellationCompensationObserver can detect that cancelled_by == restaurant and issue a discount coupon. None of the existing code changes for this — just a new observer registered at startup.

The key point for the interviewer: all five steps are coordinated by SwiggySystem.cancel_order() but executed by the right entity each time. SwiggySystem orchestrates; it doesn't do the work itself.

How would you support multiple saved addresses per user?

This tests whether your design is extensible without major surgery. Here's the minimal set of changes:

On User: Change address: Address to addresses: list[Address] and add default_address_id: str. Add methods: add_address(address) → str (returns new ID), remove_address(address_id), set_default_address(address_id), get_address(address_id) → Address.

On OrderItem / Order: Add a delivery_address: Address field to Order. This is not a reference to the User's address — it is a snapshot of the address at order time. (Same reasoning as price snapshots: if the user changes their address after placing the order, the in-progress order should still go to the original address.)

On place_order(): Add an optional address_id: str | None parameter. If provided, fetch that specific address. If None, use the user's default_address_id. Snapshot the address into order.delivery_address.

The key insight for the interviewer: no other class changes. Restaurant doesn't care. DeliveryAgent doesn't care. Order just gains one extra field that was already conceptually missing. This is what "designed for change" looks like in practice.

Why does can_cancel() live on Order and not in SwiggySystem?

This is the Tell, Don't Ask principle in action, which is a corollary to encapsulation.

The alternative that seems simpler: SwiggySystem.cancel_order() just checks if order.status in [PLACED, ACCEPTED]: proceed. This works. But now the rule "cancellable before PREPARING" lives in SwiggySystem, not in Order.

Why that's a problem: Later you add an admin portal with its own cancellation endpoint. The developer writes AdminSystem.force_cancel(order_id) and forgets to copy the status check — because it was buried in SwiggySystem, not obviously associated with Order itself. Now admins can cancel DELIVERED orders, triggering refunds for completed deliveries.

With can_cancel() on Order: Every code path that touches cancellation — SwiggySystem, AdminSystem, a future API endpoint — calls order.can_cancel(). The rule is in one place, always enforced, impossible to forget. You are "telling" the Order what to do (cancel itself) rather than "asking" it for its status and making decisions externally.

The general principle: if a rule is about an entity's own state, put the enforcement of that rule on the entity itself.

Walk me through the complete sequence when an order is successfully delivered.

The agent marks the order delivered. Here's every step in order, with the entity responsible for each:

1. Actor validation: SwiggySystem.update_order_status(order_id, DELIVERED, actor_id=agent_id) verifies the actor_id matches order.delivery_agent_id. An agent can only update the order they're assigned to.

2. State transition: order.update_status(DELIVERED) — PICKED_UP → DELIVERED is valid. Status is updated. No exception.

3. Observer chain fires: Order broadcasts to all observers:

→ PushNotificationObserver: sends "Your order has been delivered!" to the user.

→ PaymentObserver: updates payment.status = PAID. Money is captured.

→ AnalyticsObserver: logs delivery time, restaurant performance, agent performance.

→ RatingRequestObserver: queues a rating prompt to appear in the app 5 minutes later.

4. Agent is freed: agent.complete_delivery() atomically sets current_order_id = None and status = AVAILABLE. The agent immediately becomes eligible for the next assignment. Without this step, the agent would stay ON_DELIVERY forever, unavailable for new orders.

5. Order is terminal: DELIVERED has no exits in VALID_TRANSITIONS. Any future call to update_status() on this order raises InvalidTransitionError. The order is immutable from this point.

Key takeaway for the interviewer: SwiggySystem orchestrates the sequence but each entity handles its own piece. No single class does everything — each class does exactly its one thing.
