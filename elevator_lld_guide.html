Requirements & Scope
💡 In an interview, always spend 3–4 minutes clarifying requirements before drawing any diagram. This shows structured thinking.
Functional Requirements
FR-01Building has N floors and M elevators
FR-02Users can press hall buttons (up/down) on any floor
FR-03Users inside car press destination buttons
FR-04System dispatches the best elevator to a request
FR-05Elevator opens/closes doors, shows direction & floor
FR-06Support emergency stop button
FR-07Maintain passenger count and respect weight capacity
Non-Functional Requirements
NF-01Efficiency: minimise total wait time across all riders
NF-02Scalability: easily add more floors or elevators
NF-03Extensibility: swap scheduling algorithm without changing elevators
NF-04Thread safety: concurrent requests handled correctly
NF-05Observability: events logged/notified to displays and monitors
Out of Scope (for this interview)
VIP / express floors Power failure handling Fire emergency mode Predictive ML dispatch Hardware motor control

Core Classes & UML

These six entities represent the real world. Each has a single, clear responsibility.

ElevatorDirection

Enum: UP, DOWN, IDLE. Encodes the direction a car is moving or if it is stationary.

ElevatorState

Enum: MOVING, DOORS_OPEN, IDLE, EMERGENCY. Drives State pattern.

Request

A dataclass holding floor + direction. Represents one hall button press.

Elevator

Core entity. Holds current floor, direction, state, pending stops, capacity. Moves step-by-step.

ElevatorController

Manages all elevators. Receives external requests, delegates to a Scheduler.

Scheduler (Abstract)

Strategy interface. Concrete: NearestCarScheduler, SCANScheduler.

⚡ Interview tip: The key insight is that Elevator does NOT know about scheduling logic. It just has a set of stops. The Scheduler decides which elevator gets assigned. This separation is why we can swap algorithms easily.

State Pattern Behavioural
An elevator behaves very differently depending on whether it is moving, idle, or has its doors open. Instead of a giant if/elif chain, each behaviour lives in its own state class.
Python — ElevatorState (State Pattern)
from abc import ABC, abstractmethod
from enum import Enum, auto

class Direction(Enum):
    UP   = auto()
    DOWN = auto()
    IDLE = auto()

# ── Abstract state interface ─────────────────────
class ElevatorStateHandler(ABC):
    @abstractmethod
    def on_request(self, elevator: "Elevator", floor: int) -> None: ...
    @abstractmethod
    def step(self, elevator: "Elevator") -> None: ...

# ── Concrete state: Idle ─────────────────────────
class IdleState(ElevatorStateHandler):
    def on_request(self, elevator, floor):
        elevator.stops.add(floor)
        # Transition: decide direction then move
        if floor > elevator.current_floor:
            elevator.direction = Direction.UP
        else:
            elevator.direction = Direction.DOWN
        elevator.state = MovingState()

    def step(self, elevator):
        pass  # Nothing to do — already idle

# ── Concrete state: Moving ───────────────────────
class MovingState(ElevatorStateHandler):
    def on_request(self, elevator, floor):
        elevator.stops.add(floor)  # Accept on the way

    def step(self, elevator):
        if elevator.direction == Direction.UP:
            elevator.current_floor += 1
        else:
            elevator.current_floor -= 1

        if elevator.current_floor in elevator.stops:
            elevator.stops.remove(elevator.current_floor)
            elevator.state = DoorsOpenState()  # → open doors
            elevator.notify("door_open")
            return

        # No more stops in current direction → go idle
        if not elevator.stops:
            elevator.direction = Direction.IDLE
            elevator.state = IdleState()

# ── Concrete state: DoorsOpen ────────────────────
class DoorsOpenState(ElevatorStateHandler):
    def on_request(self, elevator, floor):
        elevator.stops.add(floor)

    def step(self, elevator):
        # Doors close after one "tick"
        elevator.state = MovingState() if elevator.stops else IdleState()
        elevator.notify("door_close")
🎯 Why not an enum? If you used if state == "MOVING": ... everywhere, all state logic is scattered. With the State pattern, adding an EmergencyState is one new class — no existing code changes.

Strategy Pattern Behavioural
Different scheduling algorithms (Nearest Car, SCAN/elevator algorithm, load-balanced) are interchangeable. The controller doesn't know or care which one is active.
Python — Scheduler Strategy
from abc import ABC, abstractmethod

class Scheduler(ABC):
    @abstractmethod
    def select(self, elevators: list, request: "Request") -> "Elevator": ...

# ── Strategy 1: Nearest Car ──────────────────────
class NearestCarScheduler(Scheduler):
    def select(self, elevators, request):
        def cost(e):
            distance = abs(e.current_floor - request.floor)
            # Prefer idle elevators heavily
            idle_bonus = 0 if isinstance(e.state, IdleState) else 3
            # Prefer elevators already going the same direction
            wrong_dir   = 0 if e.direction == request.direction else 2
            return distance + idle_bonus + wrong_dir
        return min(elevators, key=cost)

# ── Strategy 2: SCAN (elevator algorithm) ────────
class SCANScheduler(Scheduler):
    def select(self, elevators, request):
        def scan_cost(e):
            cur, req = e.current_floor, request.floor
            if e.direction == Direction.UP:
                if req >= cur: return req - cur          # On the way up
                return (2 * e.max_floor - cur - req)       # Past top, coming back
            elif e.direction == Direction.DOWN:
                if req <= cur: return cur - req          # On the way down
                return (cur + req)                         # Past bottom, going up
            return abs(req - cur)                         # Idle: pure distance
        return min(elevators, key=scan_cost)

# ── Controller uses any scheduler ────────────────
class ElevatorController:
    def __init__(self, elevators, scheduler: Scheduler):
        self.elevators  = elevators
        self.scheduler  = scheduler  # Swappable at runtime!

    def set_scheduler(self, scheduler: Scheduler):
        self.scheduler = scheduler  # Hot-swap algorithm

    def request_elevator(self, request):
        best = self.scheduler.select(self.elevators, request)
        best.add_stop(request.floor)
        return best

Observer Pattern Behavioural
Floor displays, loggers, and alarms all need to react to elevator events. Instead of the elevator calling each one directly (tight coupling), it publishes events to subscribers.
Python — Observer
from abc import ABC, abstractmethod

class ElevatorObserver(ABC):
    @abstractmethod
    def on_floor_change(self, elevator, floor: int): ...
    @abstractmethod
    def on_door_open(self, elevator): ...
    @abstractmethod
    def on_state_change(self, elevator, state): ...

# ── Concrete Observer: Floor Display ─────────────
class FloorDisplay(ElevatorObserver):
    def on_floor_change(self, elevator, floor):
        arrow = "▲" if elevator.direction == Direction.UP else "▼"
        print(f"[Display] Elevator {elevator.id} → Floor {floor} {arrow}")
    def on_door_open(self, elevator):
        print(f"[Display] 🚪 DOORS OPEN @ Floor {elevator.current_floor}")
    def on_state_change(self, elevator, state): pass

# ── Concrete Observer: Event Logger ──────────────
class EventLogger(ElevatorObserver):
    def __init__(self):
        self.log = []
    def on_floor_change(self, elevator, floor):
        self.log.append(("floor", elevator.id, floor))
    def on_door_open(self, elevator):
        self.log.append(("open", elevator.id, elevator.current_floor))
    def on_state_change(self, elevator, state):
        self.log.append(("state", elevator.id, type(state).__name__))

# ── Elevator: Observable subject ─────────────────
class Elevator:
    def __init__(self, id, max_floor):
        self.id           = id
        self.max_floor    = max_floor
        self.current_floor = 1
        self.direction    = Direction.IDLE
        self.state        = IdleState()
        self.stops        = set()
        self._observers   = []

    def subscribe(self, observer: ElevatorObserver):
        self._observers.append(observer)

    def notify(self, event: str):
        for obs in self._observers:
            if   event == "floor": obs.on_floor_change(self, self.current_floor)
            elif event == "door_open": obs.on_door_open(self)
            elif event == "state": obs.on_state_change(self, self.state)

Singleton Pattern Creational
There is exactly one ElevatorController for a building. Using Singleton prevents accidental creation of a second controller which would create conflicting dispatch decisions.
Python — Thread-safe Singleton
import threading

class ElevatorController:
    _instance = None
    _lock     = threading.Lock()

    def __new__(cls, *args, **kwargs):
        if cls._instance is None:
            with cls._lock:               # double-checked locking
                if cls._instance is None:
                    cls._instance = super().__new__(cls)
        return cls._instance

    def __init__(self, elevators=None, scheduler=None):
        if hasattr(self, '_ready'): return
        self.elevators = elevators or []
        self.scheduler = scheduler or NearestCarScheduler()
        self._ready    = True

# Usage — always returns the same object
ctrl1 = ElevatorController()
ctrl2 = ElevatorController()
assert ctrl1 is ctrl2  # ✓ same instance
⚠️ Interview warning: Interviewers sometimes ask "why not just a module-level variable?" The answer is that Singleton gives you lazy initialisation, thread-safe creation, and the ability to subclass or mock for testing.

Factory Method Pattern Creational
Creating an Elevator involves wiring up observers, setting capacity, floor range, etc. A factory encapsulates this so client code doesn't duplicate setup logic.
Python — ElevatorFactory
from dataclasses import dataclass

@dataclass
class ElevatorConfig:
    id:           int
    min_floor:    int
    max_floor:    int
    capacity:     int  # max passengers
    observers:    list = None

class ElevatorFactory:
    @staticmethod
    def create(config: ElevatorConfig) -> "Elevator":
        elev = Elevator(
            id        = config.id,
            max_floor = config.max_floor,
            min_floor = config.min_floor,
            capacity  = config.capacity,
        )
        for obs in (config.observers or []):
            elev.subscribe(obs)
        return elev

    @staticmethod
    def create_fleet(n: int, max_floor: int) -> list:
        """Create N identical elevators with shared logger."""
        logger = EventLogger()
        display = FloorDisplay()
        return [
            ElevatorFactory.create(ElevatorConfig(
                id=i, min_floor=1, max_floor=max_floor,
                capacity=10, observers=[logger, display]
            ))
            for i in range(n)
        ]

Command Pattern Behavioural
Button presses are encapsulated as Command objects. This lets us queue, log, undo, or replay requests — essential for features like "cancel my request" or audit trails.
Python — Command Pattern for button presses
from abc import ABC, abstractmethod
from dataclasses import dataclass, field

class Command(ABC):
    @abstractmethod
    def execute(self): ...

@dataclass
class HallCallCommand(Command):
    """Pressed UP/DOWN button on a floor."""
    controller: ElevatorController
    floor:      int
    direction:  Direction

    def execute(self):
        req = Request(floor=self.floor, direction=self.direction)
        return self.controller.request_elevator(req)

@dataclass
class CarCallCommand(Command):
    """Pressed floor button inside the car."""
    elevator: Elevator
    floor:    int

    def execute(self):
        self.elevator.add_stop(self.floor)

@dataclass
class EmergencyStopCommand(Command):
    elevator: Elevator

    def execute(self):
        self.elevator.emergency_stop()

# ── Command Queue (Button panel) ──────────────────
class ButtonPanel:
    def __init__(self):
        self._history: list[Command] = []

    def press(self, cmd: Command):
        result = cmd.execute()
        self._history.append(cmd)
        return result
Pattern Summary
Pattern Type Applied To Benefit
State Behavioural Elevator No giant if/else for moving/idle/open. Easy to add new states.
Strategy Behavioural Scheduler Swap dispatch algorithm (Nearest, SCAN) without touching Elevator or Controller.
Observer Behavioural Elevator → displays/loggers Elevator doesn't know about UI or logging. Zero coupling.
Singleton Creational ElevatorController Guarantees exactly one dispatch brain. Thread-safe creation.
Factory Creational ElevatorFactory Centralises wiring (observers, capacity, floors). Eliminates duplicate setup.
Command Behavioural ButtonPanel Button presses are objects → queueable, undoable, auditable.

"""
Elevator Control System — complete LLD implementation
Patterns: State, Strategy, Observer, Singleton, Factory, Command
"""
from __future__ import annotations
import threading
from abc import ABC, abstractmethod
from dataclasses import dataclass, field
from enum import Enum, auto
from typing import Optional

# ═══════════════════════════════════════════════════════
#  ENUMS
# ═══════════════════════════════════════════════════════
class Direction(Enum):
    UP   = auto()
    DOWN = auto()
    IDLE = auto()

# ═══════════════════════════════════════════════════════
#  REQUEST (Data class)
# ═══════════════════════════════════════════════════════
@dataclass
class Request:
    floor:     int
    direction: Direction = Direction.UP

# ═══════════════════════════════════════════════════════
#  OBSERVER INTERFACE + IMPLEMENTATIONS
# ═══════════════════════════════════════════════════════
class ElevatorObserver(ABC):
    @abstractmethod
    def on_floor_change(self, elevator: Elevator, floor: int): ...
    @abstractmethod
    def on_door_open(self, elevator: Elevator): ...
    @abstractmethod
    def on_state_change(self, elevator: Elevator, state): ...

class FloorDisplay(ElevatorObserver):
    def on_floor_change(self, elevator, floor):
        arrow = "▲" if elevator.direction == Direction.UP else "▼"
        print(f"  [Display E{elevator.id}] Floor {floor:2d} {arrow}")
    def on_door_open(self, elevator):
        print(f"  [Display E{elevator.id}] 🚪 DOORS OPEN at floor {elevator.current_floor}")
    def on_state_change(self, elevator, state): pass

class EventLogger(ElevatorObserver):
    def __init__(self): self.log = []
    def on_floor_change(self, e, floor): self.log.append(("floor", e.id, floor))
    def on_door_open(self, e): self.log.append(("open", e.id, e.current_floor))
    def on_state_change(self, e, s): self.log.append(("state", e.id, type(s).__name__))

# ═══════════════════════════════════════════════════════
#  STATE PATTERN — Elevator states
# ═══════════════════════════════════════════════════════
class ElevatorStateHandler(ABC):
    @abstractmethod
    def on_request(self, elev: Elevator, floor: int): ...
    @abstractmethod
    def step(self, elev: Elevator): ...
    def __repr__(self): return self.__class__.__name__

class IdleState(ElevatorStateHandler):
    def on_request(self, elev, floor):
        elev.stops.add(floor)
        elev.direction = Direction.UP if floor >= elev.current_floor else Direction.DOWN
        elev.state = MovingState()
        elev.notify("state")
    def step(self, elev): pass

class MovingState(ElevatorStateHandler):
    def on_request(self, elev, floor): elev.stops.add(floor)
    def step(self, elev):
        elev.current_floor += 1 if elev.direction == Direction.UP else -1
        elev.notify("floor")
        if elev.current_floor in elev.stops:
            elev.stops.remove(elev.current_floor)
            elev.state = DoorsOpenState()
            elev.notify("state")
            elev.notify("door_open")
            return
        if not elev.stops:
            elev.direction = Direction.IDLE
            elev.state = IdleState()
            elev.notify("state")

class DoorsOpenState(ElevatorStateHandler):
    def on_request(self, elev, floor): elev.stops.add(floor)
    def step(self, elev):
        elev.notify("door_close")
        elev.state = MovingState() if elev.stops else IdleState()
        elev.notify("state")

class EmergencyState(ElevatorStateHandler):
    def on_request(self, elev, floor):
        print(f"[E{elev.id}] ⚠️  EMERGENCY — ignoring request")
    def step(self, elev): pass

# ═══════════════════════════════════════════════════════
#  ELEVATOR (Core entity)
# ═══════════════════════════════════════════════════════
class Elevator:
    def __init__(self, id: int, max_floor: int,
                 min_floor: int = 1, capacity: int = 10):
        self.id            = id
        self.max_floor     = max_floor
        self.min_floor     = min_floor
        self.capacity      = capacity
        self.current_floor = min_floor
        self.direction     = Direction.IDLE
        self.state: ElevatorStateHandler = IdleState()
        self.stops: set[int]              = set()
        self._observers: list[ElevatorObserver] = []
        self._lock          = threading.Lock()

    def subscribe(self, obs: ElevatorObserver): self._observers.append(obs)

    def notify(self, event: str):
        for obs in self._observers:
            if   event == "floor":      obs.on_floor_change(self, self.current_floor)
            elif event == "door_open": obs.on_door_open(self)
            elif event == "state":     obs.on_state_change(self, self.state)

    def add_stop(self, floor: int):
        with self._lock:
            self.state.on_request(self, floor)

    def step(self):
        with self._lock:
            self.state.step(self)

    def emergency_stop(self):
        self.stops.clear()
        self.state = EmergencyState()
        self.direction = Direction.IDLE
        print(f"[E{self.id}] ⚠️  EMERGENCY STOP at floor {self.current_floor}")

    def __repr__(self):
        return (f"Elevator(id={self.id}, floor={self.current_floor}, "
                f"dir={self.direction.name}, state={self.state}, stops={sorted(self.stops)})")

# ═══════════════════════════════════════════════════════
#  STRATEGY PATTERN — Schedulers
# ═══════════════════════════════════════════════════════
class Scheduler(ABC):
    @abstractmethod
    def select(self, elevators: list[Elevator], request: Request) -> Elevator: ...

class NearestCarScheduler(Scheduler):
    def select(self, elevators, request):
        def cost(e):
            dist      = abs(e.current_floor - request.floor)
            idle_pen  = 0 if isinstance(e.state, IdleState) else 3
            wrong_dir = 0 if (e.direction == request.direction
                               or e.direction == Direction.IDLE) else 2
            busy_pen  = len(e.stops)
            return dist + idle_pen + wrong_dir + busy_pen
        return min(elevators, key=cost)

class SCANScheduler(Scheduler):
    def select(self, elevators, request):
        def scan_cost(e):
            cur, req = e.current_floor, request.floor
            if e.direction == Direction.UP:
                return req - cur if req >= cur else (2*e.max_floor - cur - req)
            elif e.direction == Direction.DOWN:
                return cur - req if req <= cur else cur + req
            return abs(req - cur)
        return min(elevators, key=scan_cost)

# ═══════════════════════════════════════════════════════
#  SINGLETON — ElevatorController
# ═══════════════════════════════════════════════════════
class ElevatorController:
    _instance: Optional[ElevatorController] = None
    _lock = threading.Lock()

    def __new__(cls, *args, **kwargs):
        if not cls._instance:
            with cls._lock:
                if not cls._instance:
                    cls._instance = super().__new__(cls)
        return cls._instance

    def __init__(self, elevators=None, scheduler: Scheduler = None):
        if hasattr(self, '_ready'): return
        self.elevators = elevators or []
        self.scheduler = scheduler or NearestCarScheduler()
        self._ready = True

    def set_scheduler(self, s: Scheduler): self.scheduler = s

    def request_elevator(self, request: Request) -> Elevator:
        best = self.scheduler.select(self.elevators, request)
        print(f"[Ctrl] Assigning E{best.id} to floor {request.floor} ({request.direction.name})")
        best.add_stop(request.floor)
        return best

    def step_all(self):
        for e in self.elevators: e.step()

    @classmethod
    def reset(cls):
        """For testing only — reset singleton"""
        cls._instance = None

# ═══════════════════════════════════════════════════════
#  FACTORY
# ═══════════════════════════════════════════════════════
class ElevatorFactory:
    @staticmethod
    def create_fleet(n: int, max_floor: int,
                     observers: list = None) -> list[Elevator]:
        return [
            Elevator(id=i, max_floor=max_floor,
                     min_floor=1, capacity=10)
            for i in range(1, n+1)
        ]

@dataclass

    floor: int

@dataclass

@dataclass

class TestElevatorState:
    def test_idle_transitions_to_moving_on_request(self):
        e = Elevator(id=1, max_floor=10)
        assert isinstance(e.state, IdleState)
        e.add_stop(5)
        assert isinstance(e.state, MovingState)

        assert isinstance(e.state, DoorsOpenState)

        e.step()            # close doors, no more stops → idle

        e = Elevator(id=1, max_floor=10)
        e.add_stop(5)
        e.emergency_stop()
        assert isinstance(e.state, EmergencyState)
        assert len(e.stops) == 0

Interview Q&A
What design pattern is most important here and why?
Core concept▶
The State pattern is the most critical because it solves the core complexity of the system — an elevator has very different behaviour depending on whether it is idle, moving, or has its doors open. Without State, every method would have a chain of if/elif checks like if self.state == "MOVING": .... With State, you add a new behaviour (e.g., EmergencyState or MaintenanceState) by creating one new class without touching any existing code. This is the Open/Closed Principle in action.
How would you handle multiple elevators efficiently?
System design▶
The Strategy pattern for the Scheduler is the key. The controller asks the scheduler "which elevator should handle this request?" and the scheduler applies an algorithm. Nearest Car: minimise distance + penalise wrong direction. SCAN: like a disk algorithm, serve floors in one direction before reversing — great for consistent throughput. The controller never changes code; you just hot-swap the scheduler. For the interview, describe the cost function: distance + idle_bonus + direction_penalty + load_penalty.
How would you make the system thread-safe?
Concurrency▶
Three places need locks: (1) Elevator.add_stop() and Elevator.step() — use threading.Lock() per elevator so concurrent button presses and the movement thread don't race. (2) ElevatorController.__new__() — double-checked locking for the Singleton ensures only one instance is created even under concurrent initialisation. (3) The Scheduler.select() reads elevator state — if elevators can move while selection is happening, take a snapshot of the state first, then select from the snapshot.
How would you extend this for VIP/express floors?
Extensibility▶
Because of the Strategy pattern, you add a VIPScheduler that: only considers elevators designated as express, filters out non-VIP requests from those elevators, and routes VIP card-holders to those cars. The Elevator class doesn't change at all. You'd also add an access_level field to Request and have the controller pass that to the scheduler. This is a clean extension with zero modifications to core classes.
Explain the Observer pattern usage here.
Design pattern▶
The Elevator is the subject. Floor displays, logging systems, alarm systems, and building management software are all observers. When an elevator's state changes (moves to a new floor, opens doors), it calls notify() which triggers all registered observers. The key benefit: the Elevator has zero knowledge of who is listening — you can add a new monitoring system by just implementing ElevatorObserver and calling subscribe(). No elevator code changes.
Why use the Command pattern for button presses?
Design pattern▶
A button press is a request — encapsulating it as a Command object gives you: (1) Queueing — you can queue commands when elevators are busy and process them in order. (2) Undo/cancel — a cancel button can call undo() on the last command. (3) Audit trail — since every command is an object, you can log them with timestamps. (4) Decoupling — the button panel doesn't need to know about the elevator or controller directly; it just executes Commands.
What is the SCAN algorithm? When is it better than Nearest Car?
Algorithm▶
SCAN (also called the elevator algorithm) moves in one direction, picking up all requests on the way, then reverses at the top/bottom. Like a disk read head. Nearest Car advantage: lower average wait time for random, sparse requests. SCAN advantage: prevents starvation (no request waits too long), better throughput under heavy load (rush hour), and more predictable worst-case latency. In a real building with rush hours, you'd blend: SCAN during busy periods, Nearest Car otherwise.
How would you add capacity tracking (weight limit)?
Feature addition▶
Add current_passengers: int and capacity: int to Elevator. When doors open, increment/decrement passengers. In the scheduler's select() cost function, add a large penalty for elevators at capacity — effectively excluding them: capacity_pen = float('inf') if e.current_passengers >= e.capacity else 0. The DoorsOpenState.step() triggers observer events so a weight sensor observer can update the count. No core logic changes needed.
Why is ElevatorController a Singleton? What are the risks?
Trade-offs▶
It's Singleton because there is one brain per building — a second controller would create conflicting dispatch decisions. Risks: (1) Global state makes unit testing hard — you must reset the instance between tests. (2) It violates dependency injection principles, making mocking harder. Mitigations: Add a reset() class method for testing, or use dependency injection where the controller is passed in rather than retrieved from the class. For multi-building systems, the Singleton would be per-building, not global.
How would you design this for a skyscraper with 80+ floors?
Scale▶
Three key changes: (1) Zone-based dispatch — divide floors into zones (1–20, 21–40, etc.) and assign elevators to zones. Add a ZoneScheduler that first selects the right zone, then the best elevator in that zone. (2) Sky lobbies — express elevators to mid-building lobbies, then local elevators from there. Model each zone as its own Controller. (3) Destination Dispatch — instead of up/down buttons, users enter their floor at a lobby kiosk. The system can group passengers going to the same floor into one elevator, dramatically reducing stops.
What would you change if this were a real production system?
Production▶
Several additions: (1) Persistence — log every event to a database for maintenance analytics. (2) Hardware abstraction — the Elevator class would publish commands to motor controllers via a MotorDriver interface; the simulation just has a mock driver. (3) Health monitoring — each elevator has a heartbeat; if it misses a beat, remove it from the fleet and alert maintenance. (4) Fire safety mode — a special state that sends all elevators to ground floor and locks them. (5) Load prediction — pre-position elevators to floors with historically high demand at certain times.
